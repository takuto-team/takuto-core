# Maestro — Phase Plan for 2026-05-21 Clean-Code Audit

Companion to `lore/audits/2026-05-21-clean-code.md` (the audit) and `CODING_STANDARDS.md` (the enforceable rules). This document locks scope and acceptance criteria for the five phases pre-named in tasks #3–#7 of the `audit-followup` team.

All criteria are written as **mechanical, yes/no checks**. A reviewer or CI job must be able to run each one and get a binary answer. Anything fuzzier ("looks clean", "is more readable") is not an acceptance criterion.

---

## Phase ordering rationale

The order tracks the audit's §8 prioritised fix list — blast radius first — and respects the hard data dependencies between phases. **Phase 1 (module splits)** lands first because `config.rs` (2,693 LOC) and `routes/workflows.rs` (2,313 LOC) are merge-conflict magnets that block every other concurrent change; splitting them is a precondition for clean per-handler/per-section diffs in later phases. **Phase 2 (MaestroError restructure)** depends on Phase 1: typing `Result<_, MaestroError>` touches the same handler bodies in `routes/workflows.rs`, so doing it after the file split lets each per-handler change live in a tiny commit instead of a 2k-line conflict. **Phase 3 (Docker hardening)** is independent of the Rust phases and can run in parallel — it only touches `Dockerfile` and the compose files. **Phase 4 (async hygiene)** depends on Phase 1+2: the `clippy::await_holding_lock` sweep and the `JoinSet` plumbing edit the same files Phase 1 reshapes and Phase 2 retypes; running it last avoids re-touching code three times. **Phase 5 (UI splits)** is independent of the Rust+Docker work, since it lives entirely in `ui/src/`; it is scheduled last only because Phase 2 (Designer) writes the cut plan it consumes.

## Phase 1 — Module splits (Rust)

**Audit drivers:** §3 worst offenders #1 (`config.rs` 2,693 LOC), #2 (`routes/workflows.rs` 2,313 LOC), §4 "God modules vs the `engine/` rule".

### Scope

- `crates/maestro-core/src/config.rs` (2,693 LOC, 26 structs+enums) → 8-file `config/` module per audit §6:
  - `config/agent.rs` — agent provider selection + per-step config
  - `config/general.rs` — top-level toggles (concurrency caps, log level, polling)
  - `config/git.rs` — git remote + GitHub App credentials
  - `config/jira.rs` — Jira site, polling, prompt-mode policy
  - `config/web.rs` — HTTP/WebSocket + runtime patches
  - `config/runtime.rs` — Docker/Network/Editor/Terminal/Provisioning/Dev
  - `config/template.rs` — `interpolate_*` and `shell_escape_value`
  - `config/mod.rs` — `Config` aggregate, `Config::load`, **re-exports preserving every public path**
- `crates/maestro-web/src/routes/workflows.rs` (2,313 LOC, 26 async handlers) → 8-file `routes/workflows/` module per audit §6:
  - `dto.rs`, `list.rs`, `lifecycle.rs`, `manual.rs`, `editor.rs`, `run_commands.rs`, `definitions.rs`, `port_tracking.rs`
  - `mod.rs` — router builder + `pub(crate) use` re-exports only; **no handler bodies**

### Out of scope

- **No** signature changes. Handlers keep their existing return type (still `Result<_, String>` until Phase 2).
- **No** behaviour changes. Same router path → handler mapping, same struct shapes, same `Config::load` semantics.
- **No** new public API. No newly-`pub` items beyond what is needed to re-export across the split.
- **No** touching `routes/workflows.rs` handlers' internals (string-typed errors, `tokio::spawn` patterns, `.clone()` calls) — those are Phase 2 and Phase 4.
- **No** changes to `step_runner.rs`, `container/runner.rs`, `engine/mod.rs`, `engine/types.rs`, `auth_resolver.rs`, `docker_hooks.rs`. The audit flagged them but they are deferred; this phase is **only** the two top-of-list god files.

### Acceptance criteria

- `wc -l crates/maestro-core/src/config.rs` returns "no such file or directory" (file removed in favour of the module).
- **Production (non-test) files** in `crates/maestro-core/src/config/` are each ≤ 600 LOC. Verify with `wc -l crates/maestro-core/src/config/*.rs` excluding `tests.rs` / `*_tests.rs`.
- `crates/maestro-core/src/config/mod.rs` is ≤ 200 LOC — **thin facade only**: `pub mod` declarations, `pub use` re-exports, the `Config` aggregate struct definition. `Config::load`, `Config::apply_runtime_dashboard_patch`, and any helpers move to dedicated files (`config/load.rs`, `config/patches.rs`).
- **Test files** in `crates/maestro-core/src/config/` (`tests.rs`, `*_tests.rs`) are each ≤ 1000 LOC. **Recommended structure** (CODING_STANDARDS §2): no standalone `tests.rs` — each `#[cfg(test)] mod tests` block lives at the bottom of the submodule it exercises. A single shared `tests.rs` is acceptable as a transitional state if the test code spans submodules; track the split as a follow-up backlog item.
- `wc -l crates/maestro-web/src/routes/workflows.rs` returns "no such file or directory".
- **Production files** in `crates/maestro-web/src/routes/workflows/` are each ≤ 500 LOC; `mod.rs` ≤ 150 LOC (router wiring + `pub(crate) use` only — no handler bodies, no `Config::load`-style heavy methods). Test files ≤ 1000 LOC and ideally inlined per CODING_STANDARDS §2.
- `grep -nE "^(async fn|fn|pub async fn|pub fn) [a-z_]+.*Handler" crates/maestro-web/src/routes/workflows/mod.rs` returns 0 — only the router wiring lives there.
- `git grep -nE "use maestro_core::(config::)?Config\b" crates/` returns the **same set** of hits as before the split (no caller updated their import).
- `git grep -nE "use crate::routes::workflows::" crates/` returns the **same set** of hits as before the split.
- `cargo build --workspace` exits 0 with **zero warnings**.
- `cargo test --workspace` exits 0; every test that existed before the split still exists and still passes (verify by test-name diff: `cargo test --workspace -- --list | sort > /tmp/tests-after; diff /tmp/tests-before /tmp/tests-after` is empty).
- `git diff <base>..HEAD -- crates/maestro-web/src/routes/workflows/mod.rs` shows the `.route(…)` list contains the **same path strings** in the same order as the original file's router builder.

### Commit shape

- **Two PRs, ~16 commits total.** PR 1: config split. PR 2: workflows split. Within each PR, one commit per extracted file, in this order:
  1. Create empty new file with the eventual `mod`/`pub use` stub.
  2. Move the code block from the god file into the new file (mechanical cut/paste, no edits).
  3. Verify `cargo build --workspace` zero-warnings + `cargo test --workspace` green before the next commit.
- The final commit in each PR deletes the god file and updates `crates/maestro-core/src/lib.rs` / `crates/maestro-web/src/lib.rs` to declare the new module.

### Risks and mitigations

- **Risk:** moving structs to a new module breaks `pub use` in `lib.rs`. **Mitigation:** add `pub use config::*` (or explicit per-type re-exports) in `config/mod.rs` and `pub use config::*;` in `lib.rs`; verify the `git grep` AC above before merging.
- **Risk:** `routes/workflows.rs` uses module-private helpers shared across handlers (e.g. `run_command_port_tracker`). **Mitigation:** put shared helpers in `routes/workflows/mod.rs` as `pub(super) fn`, not in any per-handler file. If a helper is used by 2+ handler files, it lives in `mod.rs`; if used by 1, it lives there.
- **Risk:** test modules at the bottom of the god file. **Mitigation:** per CODING_STANDARDS §2, each `#[cfg(test)] mod tests` block moves to the bottom of the submodule it exercises. Tests that span multiple submodules go in `mod.rs`'s test block.

### Verifier (Phase 1)

The tester runs:
1. `cargo test --workspace`
2. `cargo build --workspace 2>&1 | grep warning` → empty
3. The two `git grep` ACs above to confirm no public-path change
4. Spot-check: `curl -s localhost:8787/api/workflows` returns the same JSON shape before/after on a smoke run (only meaningful if a local server is reachable; otherwise rely on the test-name diff AC)

## Phase 2 — MaestroError restructure (Rust)

**Audit drivers:** §4 "Stringly-typed `MaestroError`" (11/16 variants wrap `String`; 33 `Result<_, String>` signatures); §3 worst offender #8 (`error.rs:13`).

### Scope

- `crates/maestro-core/src/error.rs` — replace `String`-wrapped variants with typed payloads:
  - `Jira { ticket: String, action: String, source: Box<dyn std::error::Error + Send + Sync> }`
  - `Git { op: String, path: PathBuf, stderr: String }`
  - `GitHubApp { source: Box<dyn std::error::Error + Send + Sync> }`
  - `Claude { source: Box<dyn std::error::Error + Send + Sync> }`
  - `AiAgent { provider: String, source: Box<dyn std::error::Error + Send + Sync> }`
  - `Database(#[from] rusqlite::Error)` — replace the manual `impl From` block with `#[from]` on the variant.
  - `Auth { kind: AuthKind, source: Box<dyn std::error::Error + Send + Sync> }` where `AuthKind` is a new enum (`Pin`, `Token`, `MasterKey`, …).
  - `Config { section: String, reason: String }`
  - Add `#[from] reqwest::Error`, `#[from] serde_json::Error`, `#[from] chrono::ParseError` where they currently lose their cause through `.to_string()`.
- Sweep every `Result<_, String>` signature (33 per audit) and replace with `Result<_, MaestroError>`. Touched crates:
  - `crates/maestro-core/src/**`
  - `crates/maestro-web/src/routes/**` (post-Phase-1 split)
  - `crates/maestro-cli/src/main.rs` — including `run_server`'s `Box<dyn Error>` return type → `Result<(), MaestroError>`.
- Sweep every `MaestroError::*(e.to_string())` call site (audit lists 41 such); replace with `?` where `#[from]` covers it, or `MaestroError::Variant { source: Box::new(e), … }` otherwise.

### Out of scope

- **No** new file moves. Phase 1 already reshaped the layout.
- **No** error-handling logic changes beyond payload typing — same fallback behaviour, same retry policy, same user-visible message text (the `#[error("…")]` strings are preserved verbatim where they exist).
- **No** changes to `tokio::spawn` patterns, lock-across-await sites, or `.clone()` cleanup — those are Phase 4.
- **No** touching error sites in `test_helpers.rs` or `#[cfg(test)]` blocks — they already use `unwrap()` and that is acceptable per CODING_STANDARDS §2.

### Acceptance criteria

- `grep -rnE "Result<[^,]+, ?String>" crates/*/src --include="*.rs" | wc -l` returns **0**.
- `grep -rnE "MaestroError::\w+\([^)]*\.to_string\(\)" crates/ --include="*.rs" | wc -l` returns **0** (down from ~41).
- `grep -nE "Box<dyn .*Error>" crates/maestro-cli/src/main.rs` returns **0**.
- `grep -cE "^\s+(Jira|Git|GitHubApp|Claude|AiAgent|Database|Auth|Config)\s*\(String\)" crates/maestro-core/src/error.rs` returns **0**.
- `grep -cE "#\[from\]" crates/maestro-core/src/error.rs` returns **≥ 5** (Io + TomlParse already present, plus rusqlite, reqwest, serde_json, chrono).
- The manual `impl From<rusqlite::Error> for MaestroError` block at the bottom of `error.rs` is gone (replaced by `#[from]` on the variant).
- A new unit test in `error.rs` asserts `MaestroError::from(rusqlite_err).source().is_some()` — i.e. the wrapped cause is recoverable, not stringified.
- `cargo build --workspace` exits 0 with **zero warnings**.
- `cargo test --workspace` exits 0.
- The public signature of `run_server` is `pub async fn run_server(...) -> Result<(), MaestroError>`; verify with `grep -nE "fn run_server" crates/maestro-cli/src/main.rs`.

### Commit shape

- **One PR, ~6–8 commits.**
  1. Restructure `error.rs` variants (compiler errors expected across the workspace from now on; do not fix yet).
  2. Add `#[from]` impls and the new test.
  3–N. Sweep call sites, one crate at a time (`maestro-core`, then `maestro-web`, then `maestro-cli`). Each commit leaves `cargo build` green.
  - Final commit: delete any remaining `Result<_, String>` aliases / re-exports.

### Risks and mitigations

- **Risk:** Promoting `Database(String)` to `Database(#[from] rusqlite::Error)` changes the variant's tuple arity; any `match MaestroError::Database(s)` site breaks. **Mitigation:** the compiler flags every site; fix them mechanically in the same commit. The audit listed the hot sites — `crates/maestro-core/src/db/credentials.rs`, `db/auth.rs`, etc.
- **Risk:** `Box<dyn Error>` in the variant payload looks like it violates CODING_STANDARDS §2 ("no `Box<dyn Error>` in public API"). **Mitigation:** the §2 rule applies to **return types**, not internal `#[source]` fields. `MaestroError` itself remains the public return type; the boxed source is an implementation detail of one variant. Add an inline comment in `error.rs` documenting this distinction.
- **Risk:** sweeping `Result<_, String>` may surface latent bugs where a handler relied on `.map_err(|e| e.to_string())` to flatten an `anyhow`-style chain. **Mitigation:** keep the test suite green between every commit; if a test starts failing on the typed error, the failure mode was already wrong and the test is the source of truth.

### Verifier (Phase 2)

The tester runs:
1. `cargo test --workspace`
2. The three `grep` ACs above (`Result<_, String>` count, `to_string()` call sites, `Box<dyn .*Error>` count).
3. Adds a new test (per task #8) that pattern-matches on a typed `MaestroError::Database { .. }` or `MaestroError::Jira { ticket, .. }` variant from a real failure path and confirms the payload fields carry structured data.

## Phase 3 — Docker image hardening

**Audit drivers:** §3 worst offender #10 (`Dockerfile:3,14,40` — 3 `FROM`, 0 `@sha256`); §4 "Unpinned base images", "Runtime bundles build toolchains", "`@latest` in image build", "29 `RUN` layers".

### Scope

- `Dockerfile` (362 LOC, 3 `FROM`, 32 total `RUN` lines, 26 in the runtime stage).
- `compose.yml` and `compose.override.yml` (only if a build target changes name).
- Specific changes per audit §6:
  - **Pin all `FROM` lines by `@sha256:` digest.** Three bases: `rust:1.88-bookworm`, `node:23-bookworm-slim`, `debian:bookworm-slim`.
  - **Split runtime stage** into `runtime-base` (no toolchain) and `runtime-build-tools` (current shape). Publish two image targets: `maestro:slim` (`= runtime-base`) and `maestro:full` (`= runtime-build-tools`).
  - **Pin every `@latest`** — `@anthropic-ai/claude-code`, `opencode-ai`, any other npm global; pin Cursor agent, `ttyd`, `openvscode-server`, and the Node tarball to explicit versions.
  - **Verify checksums** on every direct `curl … | bash` or tarball download (replace `curl … | bash` for Cursor with a pinned tarball + sha256 verify).
  - **Combine related `RUN` apt installs** to drop runtime-stage `RUN` count from 26 to ≤ 15.
  - **Add explicit `USER maestro`** directive at the end (drop root before `ENTRYPOINT`).
- Add a preamble comment block to the runtime stage stating the kitchen-sink trade-off (cf. `lore/code-quality-principles.md`).

### Out of scope

- **No** changes to the builder stages (`ui-builder`, `builder`) beyond `@sha256:` pinning.
- **No** changes to the application code, `compose.yml` env, or `entrypoint.sh` behaviour.
- **No** removal of any bundled CLI from the **default** `maestro:full` build. `maestro:slim` is opt-in via build target, not a replacement.
- **No** introduction of `WITH_*` build args (that is P2-5 territory and out of audit scope).

### Acceptance criteria

- `grep -cE "^FROM .*@sha256:" Dockerfile` returns **3** (one per stage).
- `grep -nE "@latest" Dockerfile` returns **0**.
- `grep -nE "curl .*\| ?bash" Dockerfile` returns **0** (replaced with pinned tarball + sha256 verify).
- `grep -nE "^RUN " Dockerfile` filtered to the runtime stage (`awk '/^FROM .* AS runtime/,/^FROM /'`) shows **≤ 15** `RUN` lines.
- `grep -nE "^USER maestro" Dockerfile` returns **≥ 1** and the matched line precedes the final `ENTRYPOINT` / `CMD` line.
- `docker build --target runtime-base -t maestro:slim .` exits 0.
- `docker build --target runtime-build-tools -t maestro:full .` exits 0.
- `docker run --rm maestro:full sh -c 'which cargo && which rustc && which claude && which codex && which opencode && which cursor-agent'` exits 0 (all six binaries present).
- `docker run --rm maestro:slim sh -c 'command -v cargo'` exits **non-zero** (toolchain absent from slim).
- `docker run --rm maestro:slim sh -c 'which claude'` exits 0 (CLIs are runtime-side, not toolchain).
- `docker compose up -d` followed by `curl -sf http://localhost:8787/healthz` returns 200 with the new `maestro:full` image.
- The runtime stage opens with a comment block whose first non-blank line matches `^# Maestro runtime image — `.

### Commit shape

- **One PR, ~5 commits:**
  1. Pin all `FROM` lines by digest (no behaviour change).
  2. Pin all `@latest` / shell-pipe installs (lockstep version + checksum).
  3. Combine `RUN` layers.
  4. Split runtime stage into `runtime-base` + `runtime-build-tools` and define `maestro:slim` / `maestro:full` targets.
  5. Add `USER maestro` drop + preamble comment.

### Risks and mitigations

- **Risk:** digest-pinning breaks when bases are rotated upstream. **Mitigation:** add a one-line note in `lore/audits/2026-05-21-plan.md` (this file) and the Dockerfile preamble pointing to a future Renovate / weekly-bump TODO. The pin is correct; the bump cadence is a separate item.
- **Risk:** dropping to a non-root `USER maestro` breaks files in `/workspace`, `/etc/maestro`, or the editor session that assumed root. **Mitigation:** run `docker compose up -d` after the `USER` switch and verify (a) editor session opens, (b) a workflow runs end-to-end on a smoke ticket, (c) `egress-rules.sh` still applies (it currently runs at root in `worker-entrypoint.sh` — keep that path's privilege escalation explicit if it's still needed).
- **Risk:** combining `RUN` layers may exceed BuildKit's per-layer scratch limits or break cache hit rates. **Mitigation:** group only logically-related installs (apt deps together, npm globals together). Do not pack the Rust toolchain installer with the apt block.

### Verifier (Phase 3)

The tester runs:
1. `docker build --target runtime-base -t maestro:slim .` and `docker build --target runtime-build-tools -t maestro:full .` both succeed.
2. `docker images maestro` shows both tags; `docker image inspect` confirms `maestro:slim` is materially smaller (≥ 30% smaller than `maestro:full`; this is a soft check — log the ratio, do not fail on it).
3. `docker compose up -d && curl -sf localhost:8787/healthz` returns 200.
4. The `grep` ACs above for `@latest`, `@sha256:`, `RUN` count, `USER` directive.

## Phase 4 — Async hygiene (Rust)

**Audit drivers:** §4 "Fire-and-forget `tokio::spawn`" (21 of 34 spawns drop their handle); §4 "Lock-across-`.await`" (heuristic 328 sites); §4 "`eprintln!` in production paths" (42, of which 19 in `docker_hooks.rs`); §3 worst offender #9.

### Scope

- **Enable `clippy::await_holding_lock` workspace-wide.** Add to `crates/maestro-core/src/lib.rs` and `crates/maestro-web/src/lib.rs`:
  ```rust
  #![deny(clippy::await_holding_lock)]
  ```
  Or, preferred, add to root `Cargo.toml`:
  ```toml
  [workspace.lints.clippy]
  await_holding_lock = "deny"
  ```
  Fix every flagged site.
- **Route every fire-and-forget `tokio::spawn` through a `JoinSet`** owned by the relevant lifetime holder:
  - `WorkflowEngine` gains a `tasks: tokio::task::JoinSet<()>` field (or `Arc<Mutex<JoinSet<()>>>` if cross-task abort is needed).
  - `AppState` gains a `background_tasks: tokio::task::JoinSet<()>` field.
  - Every spawn site uses `engine.tasks.spawn(...)` / `state.background_tasks.spawn(...)`.
  - Graceful shutdown drains the `JoinSet` on engine/app teardown (this is already partially wired via `CancellationToken`; integrate the `JoinSet` with the existing token).
- **Replace 42 `eprintln!`/`println!` in `crates/*/src/**`** with `tracing::info!`/`tracing::warn!`/`tracing::error!` per the call's existing severity. `docker_hooks.rs` (19 sites) is the hot spot.
- **Replace the 5 prod-path `unwrap()`/`expect()`** flagged in the audit:
  - `crates/maestro-cli/src/main.rs:1204` (`SIGTERM` handler install) → `.expect("static invariant: signal handler installation cannot fail on supported platforms")` with an inline `// SAFETY:` comment, or propagate via `?` if practical.
  - `crates/maestro-cli/src/main.rs:1216` (Ctrl+C handler) → same.
  - Any remaining hits in `crates/maestro-core/src/{server,routes}/**` outside test files.
- Production-code count: `cargo test --workspace` files like `github_app.rs`, `skill_resolve.rs`, `config_watcher.rs` use `.unwrap()` inside `#[cfg(test)]` blocks — those are **out of scope** (test code is allowed per CODING_STANDARDS §2).

### Out of scope

- **No** file moves (Phase 1's job).
- **No** error-type changes (Phase 2's job).
- **No** `.clone()` reduction — the audit flagged 793 clones; that is a separate item beyond this phase. Phase 4 only touches lines that have a `tokio::spawn`, a lock-across-await, or a `println!`/`eprintln!`/`unwrap()` on them.
- **No** UI changes.
- **No** new `tracing` subscribers — assume the existing `tracing_subscriber` setup; only the call sites change.

### Acceptance criteria

- `cargo clippy --workspace --all-targets -- -D warnings` exits 0. (The `await_holding_lock` lint is enabled and every flagged site is fixed.)
- `grep -rnE "tokio::spawn\(" crates/*/src --include="*.rs" | grep -vE "(joinset|tasks|background_tasks)\.spawn\(" | wc -l` returns **0**. Every spawn goes through a `JoinSet`.
- `grep -rnE "^\s+(eprintln|println)!" crates/*/src --include="*.rs" | grep -v "^.*tests\.rs:" | grep -v "#\[cfg(test)\]" | wc -l` returns **0**.
- `grep -rnE "\.(unwrap|expect)\(" crates/*/src --include="*.rs" | grep -vE "(tests?\.rs|test_helpers|#\[cfg\(test\)\])" | wc -l` returns **≤ 0 sites without an adjacent `// SAFETY:` comment**. (Any remaining `.expect("static invariant: …")` must be preceded by a `// SAFETY:` line — verify with `awk` script: every match in production code has a `// SAFETY:` on the previous non-blank line.)
- `WorkflowEngine` struct definition contains a `JoinSet` field; `grep -nE "JoinSet" crates/maestro-core/src/workflow/engine/mod.rs` returns ≥ 1.
- `AppState` contains a `JoinSet` field; `grep -nE "JoinSet" crates/maestro-web/src/state.rs` returns ≥ 1.
- `cargo build --workspace` exits 0 with **zero warnings**.
- `cargo test --workspace` exits 0.
- A new test in `engine/mod.rs` (or a new `engine/shutdown.rs`) asserts: spawn a background task that loops on a `tokio::time::sleep`; call `engine.shutdown()`; assert the task's `JoinHandle` aborts within 100 ms. (Per task #8 expectation.)

### Commit shape

- **One PR, ~6 commits:**
  1. Enable `clippy::await_holding_lock` and fix every flagged site (one commit per crate if it's noisy).
  2. Add `JoinSet` field to `WorkflowEngine`; migrate `WorkflowEngine`-owned spawns.
  3. Add `JoinSet` field to `AppState`; migrate `AppState`-owned spawns.
  4. Wire `JoinSet` drain into the existing `CancellationToken` shutdown path; add the shutdown test.
  5. Replace `println!`/`eprintln!` → `tracing::*` across `docker_hooks.rs` and the other 23 sites.
  6. Replace the 5 prod-path `unwrap()`/`expect()` per the list above.

### Risks and mitigations

- **Risk:** enabling `await_holding_lock` produces a large lint wall (heuristic 260+ matches). **Mitigation:** prioritise by lock type — `std::sync::Mutex` held across `.await` is a real bug; `parking_lot::RwLock::read()` released before `.await` is benign and clippy will not flag it. The lint only catches the former. Expect ~10–30 real flags, not 260.
- **Risk:** retrofitting `JoinSet` ownership on `WorkflowEngine` may require an `Arc<Mutex<JoinSet>>` if spawns happen from cloned handles. **Mitigation:** keep the `JoinSet` behind a `tokio::sync::Mutex` only if cross-task spawn is genuinely needed; otherwise own it directly and route all spawns through engine-owned methods.
- **Risk:** swapping `eprintln!` → `tracing::warn!` in `docker_hooks.rs` may change output destination during test runs that capture stderr. **Mitigation:** the test suite uses `tracing_test` or equivalent; verify a sample test's expected output still matches.
- **Risk:** the SIGTERM/Ctrl+C `expect` calls actually can't fail on Linux/macOS; rewriting them with `?` adds noise. **Mitigation:** keep `.expect(...)` with a precise message and an inline `// SAFETY: signal handler installation is infallible on tokio-supported platforms` comment — this is the rare case CODING_STANDARDS §2 permits.

### Verifier (Phase 4)

The tester runs:
1. `cargo clippy --workspace --all-targets -- -D warnings`
2. `cargo test --workspace` including the new shutdown test
3. The four `grep` ACs above (`tokio::spawn`, `println!`, `unwrap()`, `Box<dyn Error>`)
4. A local smoke run: start a workflow, send `SIGTERM` to the server, confirm the workflow's background tasks are aborted (no orphaned `tokio` threads in `ps`).

## Phase 5 — UI component splits (React)

**Audit drivers:** §3 worst offenders #11 (`TicketDetailModal.tsx` 501 LOC, 12 `useState`), #12 (`Dashboard.tsx` 440 LOC, 21 hook calls); §4 "State-rich page components"; §4 "`useEffect` for data loading".

### Scope

- `ui/src/components/modals/TicketDetailModal.tsx` (501 LOC) → 6 files per audit §6:
