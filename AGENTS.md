# Takuto — context for AI coding agents

**New agent session:** this file must be the **first** project file you read (before `README.md`, `ARCHITECTURE.md`, or the crate tree). Cursor and Claude project rules enforce that.

This file is the **canonical high-level reference** for what the repository is, how it is structured, and how the main runtime paths work. Human-oriented detail lives in `README.md`, `ARCHITECTURE.md`, and `docs/workflow.md`; prefer those for setup, troubleshooting, and deep diagrams. **Keep this file aligned with the code** when you change behavior or layout (see `.cursor/rules/` and `CLAUDE.md`).

## What this project does

**Takuto** is a Rust application that **polls Jira** for work in **To Do**, then for each ticket runs an **automated pipeline**: assign / transition status, **clone work via a git worktree**, optional **worktree init commands** (stored per `(user, workspace)` in the DB and editable from the admin Worktree Settings tab — there is **no** `config.toml` source for these; see below), then configurable **workflow definition steps** (Claude Code or Cursor Agent in **headless** mode — implement, review, lint/tests, **open a PR with `gh`**, or any custom sequence defined in TOML workflow files). The engine **does not** run a separate PR step: when the last agent session exits successfully, the workflow **finalizes** to **`Done`**. An optional PR URL for the dashboard comes from **`.takuto/outcome.toml`** (`pr_url = "…"`) or a stdout line **`TAKUTO_PR_URL: …`** (see **`agent_prompt`** headless suffix). A small **web dashboard** lists workflows, streams terminal output over **WebSocket**, and exposes REST endpoints for control and config.

**Dry mode** (`[general] dry_mode = true` or CLI `--dry-run`) skips real Jira/GitHub side effects while still running local work (worktrees, commands, Claude) through the `ExternalActions` trait (`RealActions` vs `DryRunActions`).

**Ticketing modes** (`[general] ticketing_system` in `config.toml`, default `none`):

- **`jira`** — current behavior: polls Jira for To Do tickets via `acli`, assigns/transitions tickets, full Jira integration. Requires Atlassian CLI auth (`make setup` Step 2). If acli auth fails at startup, falls back to no-ticketing mode with a warning.
- **`github`** — polls GitHub Issues from the repo's git remote via `gh api`; no Atlassian auth required. Issues appear in the dashboard `+` picker. No Jira transitions — `jira_available` is `false` for all workflows. `when: ticketing` steps run; `when: no_ticketing` steps do not.
- **`none`** (default) — no ticketing system. The Jira poller and GitHub poller are both idle. Workflows started manually only (dashboard `+` opens a paste-description modal). `when: no_ticketing` steps run; `when: ticketing` steps do not. Dashboard shows an **amber warning banner** and a one-time **alert dialog** on page load.

The dashboard hides polling controls when `ticketing_system = none`. In Jira and GitHub modes polling controls are visible. When `ticketing_system = none`: the **+** button opens a **1000px-wide paste-description modal** (workflow name + description textarea); the workflow name is slugified into the **`ticket_key`** (which drives the branch and worktree names, e.g. `feat/add-user-login`) while the original name becomes **`ticket_summary`**. When the name field is left empty, the server falls back to **`MANUAL-{timestamp_millis}`**. The user-provided description is stored as **`ticket_description`** and available in prompts via **`{ticket_description}`** or the **`{description}`** alias. When `ticketing_system = github`, the `+` button opens a GitHub Issues picker (calls `GET /api/github/issues`) instead. `GET /api/config` returns **`jira_available: bool`** and **`ticketing_system: string`** (`"jira"`, `"github"`, or `"none"`); `WorkflowSummary` includes **`jira_available`** and **`ticket_description`** per workflow. The `Workflow` struct carries **`jira_available: bool`** (persisted in snapshots with `#[serde(default)]` = `true` for backward compatibility).

## How the binary starts

`crates/takuto-cli/src/main.rs`:

1. Parses CLI (`--config` / `TAKUTO_CONFIG`, default `config.toml`; `--dry-run`).
2. Loads `Config` from file or defaults.
3. Initializes **JSON** logging via `tracing_subscriber` and `EnvFilter` (including `general.log_level`).
4. Builds `Arc<RwLock<Config>>` and `Arc<dyn ExternalActions>` from dry mode.
5. Probes acli authentication (`docker_hooks::check_acli_auth()`) → `jira_available: Arc<AtomicBool>`. When `false`: logs a warning, workflows skip Jira steps, poller is idle.
6. Constructs `WorkflowEngine` with **`[general] max_concurrent_workflows`** (drives an internal semaphore for concurrent **mise** / **install** / **agent** sessions) **and** `jira_available`. Calls **`restore_persisted_workflows`**, which rebuilds the in-memory map from the **`work_items`** table (DB-first); the per-workspace snapshot is read only as a fallback for `(ticket_key)` rows the DB lacks (DB-wins on conflict), before the poller’s first poll. **All workspaces** are restored at once — workflows from every workspace coexist in the single in-memory `HashMap<String, Workflow>`, tagged by `Workflow.workspace_name`. Snapshots are **per-workspace** at **`{data_dir}/workspaces/{workspace_name}/workflow_snapshot.json`**, where the **persistent data directory** (`data_dir`) is resolved as: **`$TAKUTO_DATA_DIR`** → **`$TAKUTO_HOME/.takuto`** → **`$HOME/.takuto`** → **`{git.repo_path}/.takuto`** (legacy fallback). In Docker, the **`takuto-data`** named volume is mounted at **`/home/takuto/.takuto`** and **`TAKUTO_DATA_DIR`** points there. On startup, `read_all_workspace_snapshots(data_dir)` scans `{data_dir}/workspaces/*/workflow_snapshot.json` and merges all records. A 3-tier migration handles legacy layouts: workspace-specific → in-repo legacy (`{git.repo_path}/.takuto/`) → old global (`{data_dir}/workflow_snapshot.json`). The Jira poller separately enforces **`[general] max_active_workflows`** (see **Jira poller**).
7. Constructs `JiraPoller` with a shared `CancellationToken` and **`Arc<AtomicBool>`** polling pause flag (shared with `AppState`). The flag is **`false`** by default (polling enabled); set **`[general] auto_polling = false`** in **`config.toml`** to start in the same paused state as the dashboard **Pause polling** action, then `build_router`. When `!jira_available`, the poller future is replaced with `std::future::pending()` (never runs).
8. **Initialises the multi-user database** via `Database::connect(data_dir, &[database], allow_auto_generate_secret_key)` (plan-11). When `[database].connection` is empty (default), Takuto builds a SQLite pool against `{data_dir}/takuto.db` — WAL journal mode and foreign keys are applied per-connection by sqlx. When `connection` is `postgres://…` / `postgresql://…` / `mysql://…`, the equivalent sqlx pool is built; a 5 s `SELECT 1` probe gates `fail_fast` (abort vs. fall back to local SQLite). All sqlx migrations run on the resolved backend. If a legacy `{data_dir}/takuto.db` exists and the target lacks `system_metadata.import_complete`, the one-shot SQLite → remote importer runs inside a single transaction (plan-11 §8). The resulting `Database` is passed to `AppState.db` as `Some(db)`. When the database has **zero users**, `GET /api/auth/status` returns `setup_required: true` and the dashboard shows a **first-user setup page** where the admin creates their account via `POST /api/auth/register`. If `data_dir` cannot be resolved or the backend is unreachable with `fail_fast = true`, `db` is set to `None` and multi-user auth is unavailable.
9. Creates a `ConfigWriter` from the config file path and a `ConfigWatcher` that polls for external edits (5 s interval). Both share a `last_write_epoch_ms` atomic to prevent reload loops from API writes.
10. Runs poller, Axum server, config watcher, snapshot syncer, PR merge poller with graceful shutdown on cancel, and signal handler (`SIGINT` / `SIGTERM` on Unix). On **graceful** shutdown the process cancels the token, **`persist_interrupt_for_restart`** (calls **`write_all_workspace_snapshots`** — groups all in-memory workflows by `workspace_name`, writes each group to `{data_dir}/workspaces/{name}/workflow_snapshot.json` — and cancels their driver tokens **without** Jira unassign / **`Stopped`**), then waits briefly for cleanup. On restore, **terminal** workflows (**Done**, **Stopped**, **Error**) are loaded with **no driver** (idle on the dashboard); **Paused** and in-progress workflows get a driver spawned. The last Claude/Cursor **session ID** (`Workflow.last_session_id`) is persisted in the snapshot; on resume the first re-executed agent step uses **`--resume`** with the prompt **"Resume what you were doing before the session was closed."** so the agent continues its prior conversation. **`POST …/stop`** and **`stop_all_workflows`** still unassign and move tickets back to **To Do** (explicit stop, not container resume) — **only when `jira_available` is true**.

## Workflow model

### State machine

Defined in `crates/takuto-core/src/workflow/state.rs` as `WorkflowState`: `Pending` → `Assigning` → `RetrievingDetails` → `CreatingWorktree` → **`AddressingTicket { pass }`** (`pass` is the outer cycle index, always `1` — the built-in step list runs once; use per-step **`repeat`** for multiple runs) → **`Done`**, `Error { .. }`, `Paused { .. }`, `Stopped`. The **`Reviewing`**, **`CreatingPR`**, **`AddressingPrComments`**, and **`MergingBaseBranch`** variants remain for **deserialization of older persisted snapshots only** — they are no longer actively driven by the engine. The **dashboard and REST list** use **`Workflow::status_display()`**, which shows the live **`current_step_label`** (e.g. **`Implement ticket (cycle 2/3, run 1/1)`**).

Terminal states: `Done`, `Stopped`, `Error`.

### Engine and storage

`WorkflowEngine` (`workflow/engine.rs`) holds:

- `Arc<RwLock<Config>>`
- `Arc<RwLock<HashMap<String, Workflow>>>` — **keys are Jira ticket keys** (e.g. `PROJ-123`), not the workflow UUID
- `Arc<dyn ExternalActions>`
- `broadcast::Sender<WorkflowEvent>` for real-time updates

The authoritative store for durable workflow fields is the DB **`work_items`** table; the in-memory `HashMap<String, Workflow>` is a **read-through cache** that additionally holds non-serializable runtime handles (`CancellationToken`, broadcast wiring, live terminal-line ring buffer). At startup the cache is rebuilt from `work_items` (`list_all_for_restore` → `Workflow::from_work_item_row`), not from the snapshot. State-defining writes (Done/Error/Stopped, the initial insert) are fail-loud (retried, ERROR on persistent failure) rather than swallowed; advisory writes (ports, branch/worktree, terminal lines) stay best-effort. Multi-column logical mutations (branch+worktree at worktree creation; pr_url+state at Done) are written in a single statement so a crash can't tear them.

The **Jira poller** uses `WorkflowEngine::start_workflow` (with **`started_manually: false`**), which inserts the workflow **and** immediately discovers all dep-free (no `depends_on`) dynamic workflow definitions and calls `start_workflow_def` for each. The **dashboard manual flow** (the **+** picker) calls **`WorkflowEngine::add_to_dashboard`** instead: it inserts the workflow in **`Pending`** state with **`driver_started = false`** and does **not** spawn any drivers. For Jira tickets, `add_to_dashboard` still assigns the ticket and transitions it to **In Progress** (best-effort, spawned task). Clicking a workflow-def button on a **Pending** card calls **`POST /api/workflows/{id}/run-workflow/{def}`** → `start_workflow_def` → `drive_workflow_def` — which runs `bootstrap_new_workflow` first (Jira assign/retrieve, worktree creation, mise/install/pre-workflow) before the def's agent steps. **`max_concurrent_workflows`** caps concurrent **mise** / **install** / **agent** sessions via a shared semaphore (permits are taken **after** `wait_if_paused`, so paused workflows do not hold a heavy-work slot). The Jira poller uses **`max_active_workflows`** (or **`max_concurrent_workflows`** when **`max_active_workflows`** is **`0`**) against **`dashboard_workflow_count`**: every workflow still in the map counts as **active** — **Done**, **Paused**, **Stopped**, **Error**, and in-progress rows all count until **Mark as Done** or **Delete** drops the row from the dashboard. **`[general] max_concurrent_manual_workflows`** ( **`0`** = unlimited) caps how many **manual** starts may **occupy a slot** at once (not **Done** / **Stopped** / **Error**), enforced server-side on **`POST /api/workflows/start-manual`** and reflected by disabling the dashboard **+** tile when full.

`Workflow` carries a **`driver_started: bool`** field — `false` when added to dashboard via the **+** picker (no driver running), `true` once `start_workflow_def` spawns a def driver (set in the write lock before `tokio::spawn`). **`WorkflowSummary`** includes **`can_start: bool`** (`Pending` + `!driver_started` — informational; no separate `/start` endpoint exists). Deleting an unstarted Pending workflow is allowed (the `is_active()` guard is bypassed when `!driver_started`); for Jira, delete also unassigns and reverts to **To Do**. Snapshot persistence: `driver_started` is stored in **`PersistedWorkflowRecord`** (`#[serde(default)]` = `true` for backward compatibility); on restore, `Pending` + `!driver_started` workflows are inserted without a driver (like terminal states).

**Per-user workflow isolation:** Each `Workflow` carries an optional `user_id: Option<String>` — the ID of the user who created it. Every user — including admins — can only see and act on their own workflows. The `user_id` is persisted in snapshots (`PersistedWorkflowRecord`, `#[serde(default)]` for backward compatibility). Old workflows with `user_id = None` are not visible to any user (orphaned until reassigned — see **`migrate_orphan_workflows`** below for an opt-in one-shot reassignment at startup). The auth middleware inserts an `AuthenticatedUser` (user_id + role) into request extensions; workflow handlers extract it and enforce ownership via `require_workflow_access()`. Admin privileges are scoped to **user management only** (create/edit/suspend/delete users), not cross-user workflow access.

**Admin gates on shared state (AC-3):** Six mutating endpoints — `PUT /api/config`, `POST /api/config/reload`, `POST /api/polling/pause`, `POST /api/polling/resume`, `POST /api/workspaces/switch`, `POST /api/repos/clone` — require **`role = admin`** and respond with `403 Forbidden` for non-admin callers. The check uses `routes/admin.rs::require_admin_for(&state, &auth)` against the `AuthenticatedUser` already attached by the auth middleware (no second cookie lookup). Read-side counterparts (`GET /api/config`, `GET /api/polling`) stay open to any authenticated user.

**Poller workflow ownership (AC-4):** At startup `takuto-cli/src/main.rs::resolve_poller_owner` resolves a user-id for poller-created workflows in this order: (a) `[general] poller_owner_username` if set and the named user is non-suspended; (b) lexicographically-first non-suspended admin via `db::users::list_admins`; (c) `None`. Both pollers carry the resolved id as `JiraPoller.resolved_owner_id` / `GitHubPoller.resolved_owner_id`; when it is `None` they log a warning and skip `start_workflow` so no orphan workflows are created (the web server still serves login/setup so an admin can be registered to enable polling later). The new trailing `user_id: Option<String>` parameter on `WorkflowEngine::start_workflow` flows through `WorkflowLifecycle::start_workflow` and is stored on `Workflow.user_id`; `retry_workflow` preserves the original owner. When `[general] migrate_orphan_workflows = true`, restored workflows with `user_id == None` are reassigned to the resolved owner via `WorkflowEngine::migrate_orphan_workflows_to_owner` immediately after `restore_persisted_workflows`, then snapshot-synced so the migration survives a crash. Migration is **off by default**.

`Workflow` carries ticket metadata, `steps_log`, branch/worktree paths, `pr_url`, **`ticket_url`** (`Option<String>` — the direct URL to the issue in the ticketing system, stored at creation time; `Some(html_url)` for GitHub-sourced workflows, `None` for Jira and manual workflows; the REST API exposes this as `issue_url` in `WorkflowSummary`, falling back to the computed Jira browse URL when `ticket_url` is `None`), **`pr_merged`** (whether the associated PR has been merged on GitHub), `CancellationToken`, `last_session_id` (Claude/Cursor session ID for `--resume` across restarts), **`workspace_name`** (derived from the last path component of `config.git.repo_path` via `workspace_name_from_repo_path()`), and up to **100** recent `terminal_lines` for UI persistence.

**Per-workspace snapshot persistence:** **`workflow/snapshot.rs`** defines the JSON snapshot format for graceful shutdown / restart. The per-workspace `workflow_snapshot.json` is a **crash-recovery fallback**, not the restore source. On restore the **DB wins every conflict**; the snapshot is consulted only for `(ticket_key)` rows the DB lacks (backfilled into the DB), and to **supplement** the few runtime fields the DB does not store (`auth_pin`, `terminal_lines`, `description_session_id`, `worktree_bootstrapped`, and `ticketing_system` for github-mode rows). It is still written on graceful shutdown / periodic sync for defence-in-depth. Snapshots are **per-workspace** at **`{data_dir}/workspaces/{workspace_name}/workflow_snapshot.json`** (see **How the binary starts** step 6). On shutdown / periodic sync, **`write_all_workspace_snapshots`** groups all in-memory workflows by `workspace_name` and writes each group to its own snapshot file; workspace directories with zero records are cleaned up. On restore, **`read_all_workspace_snapshots`** scans `{data_dir}/workspaces/*/workflow_snapshot.json` and merges all records into the single in-memory map. **All** workflows in the map are snapshotted (including **Stopped**, **Error**, **Paused**) — they remain on the dashboard until explicitly deleted or marked done. On restore, **terminal** states (**Done**, **Stopped**, **Error**) and **unstarted Pending** (`Pending` + `driver_started == false`) are inserted with **no** driver; other in-progress / **Paused** rows find any `Running` workflow def entries and re-spawn `drive_workflow_def(Some(wt))` for each; if the worktree no longer exists, those defs are marked `Error`. `Running` entries in `steps_log` are stripped on restore (they represent interrupted steps that will re-execute). The persisted **`last_session_id`** lets the first re-executed agent step resume the prior Claude/Cursor conversation via `--resume`.

**Multi-workspace model:** All workflows from all workspaces coexist in a single in-memory `HashMap`. Switching workspaces (**`POST /api/workspaces/switch`**) flushes the current snapshot, updates `config.git.repo_path`, and returns — running workflows from the previous workspace **continue executing** in the background. **`GET /api/workflows`** filters to only return workflows matching the current workspace's `workspace_name`. **`GET /api/workflows/counts`** aggregates across **all** workspaces (global counters). The dashboard `SummaryStats` shows global counts while `WorkflowGrid` shows only the current workspace's workflows.

### Per-user, per-workspace flows

The buttons a user sees on a work-item card are **flows** they own, scoped **per workspace**. A flow groups one or more **steps**, each a single agent prompt (optionally with skills); the flow is the unit clicked on the dashboard. The runtime source of truth is the database (`user_work_item_flows`), **not** the TOML files — there is no per-request TOML fallback. Types live in `crates/takuto-core/src/workflow/`: `UserFlow { name, depends_on: Vec<String>, steps: Vec<UserFlowStep> }` and `UserFlowStep { name, prompt, skills: Vec<SkillRef> }`. A user's list is capped at **20 flows** per workspace (enforced server-side on `PUT /api/me/flows`).

**TOML files are the seed source only.** The `workflows/` directory is parsed once at startup (`discover_workflows`, stored on `AppState` as `Arc<Vec<UserFlow>>`); one TOML file maps to exactly one `UserFlow` (the file's `[[steps]]` become the flow's ordered steps). Command-only TOML files are skipped, and the legacy `when` / `repeat` fields are dropped at parse time with a warning (a seeded flow always runs unconditionally with `repeat = 1` per step). Seeding is idempotent (`seed_if_absent` = INSERT ON CONFLICT DO NOTHING) and happens on three paths: at user creation (`routes/users.rs::create_user`, `routes/auth.rs::register`), on first visit to a workspace (`POST /api/workspaces/switch`), and via a one-time backfill at process start for every active user against the active workspace. Editing a TOML file **never** retroactively changes an already-seeded user's flows; the only way to pull new defaults is the explicit **Re-seed from defaults** action (`POST /api/me/flows/reseed`), the user-creation hook, a never-before-seen workspace, or the startup backfill on a still-unseeded row. A user who deletes every flow keeps an **empty** row (not auto-restored); the dashboard renders an empty-state banner linking to Configuration → **Flows**.

**Resolution at runtime:** `resolve_user_flows(db, user_id, workspace, defaults)` (`workflow/definitions.rs`) returns the user's flows as `DiscoveredWorkflow`s. The TOML `defaults` are passed in by the caller because `takuto-core` cannot reach `AppState` (where the parsed `Arc<Vec<UserFlow>>` lives). Each flow becomes a multi-step pipeline (`when = Always`, `repeat = 1` per step). `filename` is a stable slug of the flow `name` (kebab-cased, length-capped) so the `workflow_def_runs` key survives across edits as long as the slug is unchanged. `depends_on` gating, cycle detection, and the `valid` / `error` lock-icon treatment reuse the existing TOML-path logic. `None` from the DAO triggers a `seed_if_absent` + one retry; a persistent `None` or an empty list both resolve to an empty button set.

**Rename = delete + re-create.** Renaming a flow changes its slug and therefore its `workflow_def_runs` key. An interrupted run is resolved on restart from the workflow's persisted `user_id` + `workspace_name`; if the flow was since renamed or deleted, the run is marked `Error` with the typed reason **`flow_no_longer_exists`** and the user triggers the renamed flow as a fresh run. An in-flight run completes against the flow snapshot resolved when it started.

### Dynamic workflow definitions — bootstrap and step sequence

All workflows are driven entirely by **dynamic TOML workflow definitions** (`*.toml` files in the `workflows/` directory). There is no static step list driven from `config.toml`. When a def button is clicked on the dashboard, **`POST /api/workflows/{id}/run-workflow/{def}`** calls `start_workflow_def` → `drive_workflow_def`:

**Bootstrap** (`bootstrap_new_workflow` in `workflow/engine.rs`) — called automatically when the workflow has no worktree yet (Pending card, first run):

1. **Assign** ticket to the **currently authenticated Jira user** (`acli` **`@me`**) and move to **In Progress** (failures may be logged/`[DRY/SKIP]` but workflow can continue). **When `workflow.jira_available` is `false`**: skipped entirely (no step log, no state transition).
2. **Jira details** via `get_ticket_details`; populate description, summary, type on `Workflow`. The owner's **per-user Jira credential** (REST) is used when one is stored, otherwise the global `acli` `JiraClient` (see **Per-user Jira credential**). **When `workflow.jira_available` is `false`**: skipped; a `JiraTicket` is constructed from the workflow’s in-memory `ticket_summary` / `ticket_description` (user-pasted via the paste-description modal).
3. **Worktree** from `git::worktree::branch_name_for_ticket` and configured `base_branch`. **`create_worktree`** removes any leftover directory at the target path first so a prior failed teardown cannot block a new run.
4. **Worktree init commands** — a single ordered list of shell commands run sequentially in the worktree directory (streaming output, cancellation-aware). Source resolution is **DB-only**: `resolve_worktree_init_commands(user_id, workspace_name, db)` (`workflow/engine/resolve.rs`) reads the workflow owner's `user_worktree_commands` row for the `(user, workspace)` pair, edited from the admin "Worktree Settings" tab. Each command appears as a step labeled **`Worktree init (i/n): {snippet}`** (first 40 chars). **There is no `config.toml` source and no global-default fallback** — when the owner has no DB row (or there is no `user_id` / no `db`), the bootstrap runs zero init commands. A stale **`[commands]`** / **`[[run_commands]]`** table in `config.toml` is ignored entirely (a startup warning is logged via `detect_legacy_command_keys`; nothing is parsed or concatenated).

Bootstrap returns `(worktree_path, ticket_detail)`. When the workflow **already has a worktree** (e.g. running a second def after the first completed), bootstrap is skipped and the driver goes directly to the agent steps.

**Agent steps** (`run_workflow_def_steps`): run the definition’s `[[steps]]` via `run_agent_step_sequence`.

   **Step types:** Each step is either an **agent step** (has `prompt` and/or `skills`) or a **command step** (has `commands`). The two modes are **mutually exclusive** — validation rejects a step that specifies both `prompt` and `commands`, or neither `prompt` nor `skills` nor `commands`.

   **Agent steps** invoke Claude Code or Cursor Agent. Each step supports an optional **`skills`** array of **`{ name, args }`** objects. For **Claude** (`--bare` mode): each referenced skill’s **`<name>/SKILL.md`** is read from skill search paths (worktree **`.claude/skills/`** first, then **`~/.claude/skills/`**), frontmatter is stripped, `$ARGUMENTS` / `$1` / `$2` … are substituted with positional args, and the concatenated result is passed via **`--system-prompt`**. Claude Code’s `--print` mode does not support native `/skill` invocations, so Takuto handles skill content injection. For **Cursor**: skills work natively, so Takuto builds `/skill-name arg1 arg2` invocation lines and prepends them to the prompt.

   **Command steps** run shell commands directly without invoking an AI agent. The `commands` field is an array of strings; each is executed sequentially via `bash -c` (or `mise exec -- sh -c` when mise is configured) in the worktree directory. If any command exits non-zero, the step fails and the workflow follows the existing error/retry behaviour. Command stdout/stderr is streamed to the dashboard terminal in real-time. The `skills` and `resume_previous` fields are ignored on command steps (they are agent-specific). Command steps respect `[agent] step_timeout_secs` for each individual command and support container isolation when DinD is available.

   All steps interpolate **`{ticket_key}`**, **`{ticket_summary}`**, **`{ticket_description}`** (alias **`{description}`**), **`{ticket_type}`**, **`{acceptance_criteria}`**, **`{ticket_context}`**, **`{base_branch}`**, **`{pr_url}`** — placeholders work in both `prompt` (agent steps) and `commands` (command steps). **`{ticket_context}`** includes a trusted policy preamble and **UNTRUSTED_JIRA** sections (see `build_ticket_context` in `workflow/engine.rs`); linked-issue inclusion follows **`[jira] linked_items_in_prompt`**. For agent steps, headless instructions from **`agent_prompt.rs`** are appended (the OpenCode/Codex suffix explicitly forbids retrying a failing command more than twice and tells the agent to record the PR URL). There is **no** built-in PM / plan-validation pass — add a **custom step** whose **prompt** asks the agent to validate plans or requirements if you want that. Step failure **aborts** the definition run (Error state). For deterministic tasks like linting or testing, use **command steps** instead of agent prompts to avoid consuming AI tokens. **No-progress guardrail** (`[agent] max_repeated_output_lines`, default **8**, `0` = off): the agent-step output relay (`spawn_output_relay`) counts consecutive repeats of the same substantive humanized line — session-lifecycle lines (`… initialized`/`completed`, via `stream_humanize::is_session_lifecycle_line`) are excluded so healthy multi-turn runs don't trip it. On reaching the threshold it sets a shared `LoopSignal` and cancels the session's `step_cancel` child token; `run_agent_step_sequence` then fails the step with **`AgentError::NoProgressLoop`** (→ Error state, **not** a pause/re-run), so a weak model looping on a failing action (e.g. an unfixable `git push`) aborts in seconds instead of churning until `step_timeout_secs`.

After the worktree is created, `bootstrap_new_workflow` calls **`configure_git_author_from_github`**: when **`[github]`** App credentials are configured, this uses the bot identity (`takuto-bot[bot]` / `{app_id}+takuto-bot[bot]@users.noreply.github.com`) and configures `gh` auth with an installation access token; otherwise it falls back to **`gh api user`** using GitHub’s **`{id}+{login}@users.noreply.github.com`** form so commits match the personal **`gh`** account.

### Dynamic workflow definitions (TOML)

`crates/takuto-core/src/workflow/definitions.rs` implements **user-customizable secondary workflows** loaded from TOML files at runtime. These are dynamic: discovered by scanning a `workflows/` directory for `*.toml` files, with no restart needed.

**TOML schema** — each `*.toml` file:
```toml
name = "Display Name"              # required, shown on UI buttons
depends_on = ["other_workflow"]    # optional, list of filenames (without .toml)

[[steps]]
name = "Step name"
prompt = """
Multi-line prompt...
"""
repeat = 1
when = "always"                    # always | ticketing | no_ticketing

[[steps]]
name = "Run tests"
commands = ["npm test"]            # command step (no AI)
```

Steps support both **agent steps** (has `prompt` and/or `skills`) and **command steps** (has `commands`). Steps are converted to the existing `AgentStepConfig` type for execution.

**Discovery** (`discover_workflows`): scans the configured directory, skips `*.example.toml` files, parses each TOML file, validates dependencies (detects cycles via DFS, marks workflows with missing deps as invalid). Invalid TOML is skipped with a warning (the `DiscoveredWorkflow` struct carries `valid: bool` and `error: Option<String>`).

**Dependency resolution**: `depends_on` uses filenames (without `.toml`). `are_dependencies_met()` checks that all upstream workflows completed successfully. `topological_order()` returns a valid execution order (Kahn's algorithm). Circular dependencies are detected at load time and the involved workflows are marked invalid.

**Runtime state**: Each workflow tracks per-definition run state in `workflow_def_runs: HashMap<String, WorkflowDefRunState>` (states: `Idle`, `Running`, `Completed`, `Error`). The field is persisted in workflow snapshots (`PersistedWorkflowRecord`).

**Engine integration** (`workflow/engine.rs`):
- `start_workflow_def(ticket_key, def_name)`: discovers definitions, validates dependencies are met, sets `Running` state (and `driver_started = true`), spawns async driver. When the workflow has no worktree (Pending, first run), passes `None` as the worktree path so the driver bootstraps first.
- `retry_workflow_def(ticket_key, def_name)`: resets `Error` → `Idle`, then calls `start_workflow_def`
- `drive_workflow_def(worktree_path: Option<PathBuf>)`: async driver. When `worktree_path` is `None`, calls `bootstrap_new_workflow` first; otherwise uses the existing worktree. Then runs the definition's steps via `run_workflow_def_steps()` / `run_agent_step_sequence()`, updates state to `Completed` or `Error`. Cancelled drivers in Stopped or Paused state exit without overwriting state.
- `bootstrap_new_workflow()`: performs Jira assign/retrieve, worktree creation, git author setup, mise install, pre-install, install, and pre-workflow commands. Returns `(worktree_path, ticket_detail)`.
- `start_definitions_watcher(cancel_token)`: background task polling the definitions directory every 5 seconds; broadcasts a `workflow_definitions_changed` WebSocket event when TOML file changes are detected

**Config**: `[general] workflow_definitions_dir` (default `"workflows"`) — path to the directory scanned for `*.toml` definitions, resolved relative to the config file's directory.

### Mark as Done

**`POST /api/workflows/{ticket_key}/mark-done`** (dashboard **Mark as Done**): transitions Jira to **`[jira] done_status`** (default **`Done`**), then **`remove_worktree`**, then best-effort **`delete_local_branch`** (**`git branch -D`**) when **`branch_name`** is set, then **`git worktree prune`**. Partial failure is returned in JSON (**`MarkDoneOutcome`**); the workflow is **removed from the map** only if **both** Jira and worktree removal succeed (branch/prune failures are logged only). A WebSocket **`workflow_removed`** event is sent when the row is dropped.

**`POST /api/workflows/{ticket_key}/delete`** (dashboard **Delete**): allowed when **`WorkflowState::is_active()`** is false (not **running** — includes **Paused**, **Done**, **Stopped**, **Error**). Cancels the workflow token, **`ContainerRunner::cleanup_for_ticket`** (force-remove worker containers), best-effort **`remove_worktree`**, best-effort **`delete_local_branch`**, **`git worktree prune`**, then removes the row from the map — **no Jira transitions**. Syncs **`workflow_snapshot.json`** best-effort; emits **`workflow_removed`**.

Pause/resume/stop apply while a workflow definition is actively running (**`is_active()`**).

Pause/resume: workflow state wraps prior state in `Paused`; `wait_if_paused` blocks the driver until resumed.

Stop (**API** / **`stop_all_workflows`**): cancels the workflow token (child processes stop via `ProcessHandle` / process groups on Unix), then **`ContainerRunner::cleanup_for_ticket`** immediately so isolated worker containers are not left running until the driver exits; Jira cleanup (unassign / To Do) runs in a spawned task as before. Drivers that exit with **`Cancelled`** while the row is **`Stopped`** or already removed do not overwrite **`Stopped`** with **`Error`**. **`acquire_agent_slot`** aborts on cancel so workflows do not block forever on the install/agent semaphore after stop. **Graceful process shutdown** (SIGINT/SIGTERM path in **`takuto-cli`**) uses snapshot persist + cancel so tickets stay **In Progress** for resume.

The **index** dashboard uses a custom **confirm** modal (**`#dashboardConfirmModal`**) for **Stop**, **Delete**, and **Mark as Done** (not `window.confirm`). **Delete** and **Mark as Done** show a dark semi-transparent overlay with a spinner on the card while the request is in flight. **Error** / **Stopped** workflows show two retry buttons: **Retry from last failure** (primary/blue, calls **`resume-from-error`** — re-drives from the failed step, skipping succeeded steps, reusing worktree and session ID; only shown when the worktree exists on disk) and **Retry from 0** (primary/blue, calls **`retry`** — removes the workflow and starts fresh).

File logging: `WorkflowLogWriter` writes under `{repo_path}/logs/<TICKET>.log`.

## AI agent integration (four v1 providers)

### Provider selection

`[agent] provider` in config: **`claude`** (default), **`cursor`**, **`codex`**, or **`opencode`**. Each provider has its own sub-table at `[agent.providers.<name>]` (`model`, `base_url`, `extra_args`, `allow_shared_default`; Cursor swaps `base_url` for `cli`; Codex adds `provider_name`; OpenCode adds optional `context_limit` / `output_limit` token windows). The model / CLI / endpoint for a provider live **only** in its sub-table — the legacy flat keys (`cursor_cli`, `cursor_model`, `model`) have been removed (no migration window).

**`[agent] share_conversation_across_steps`** (default **`false`**): controls whether the steps of a single flow run as one shared agent conversation or independent sessions. When `false`, `run_agent_step_sequence` passes `resume_id = None` for each first-run step (fresh session, no memory of earlier steps) — the historical behavior. When `true`, a step after the first resumes the prior step's session id (`--resume` / `-s`), so the agent carries full context forward. A per-step `resume_previous = true` forces resume regardless. Editable via **`PUT /api/config/agent`** (the `share_conversation_across_steps` field) and the Configuration → AI Settings toggle.

### Claude Code

`crates/takuto-core/src/claude/session.rs` spawns **`claude`** with `--dangerously-skip-permissions`, `--print`, `--verbose`, `-p`, `--output-format stream-json`, optional `--resume`, optional `--model`.

### Cursor Agent (headless)

`crates/takuto-core/src/cursor/session.rs` spawns **`cursor_cli`** (e.g. `agent`) with **print** mode, **`--output-format stream-json`**, **`--stream-partial-output`**, **`--trust`**, **`--force`**, **`--approve-mcps`**, **`--sandbox disabled`**, **`--workspace <worktree>`**, optional **`--resume`**, optional **`--model`**. This matches Cursor’s non-interactive / trusted automation flags from their CLI docs.

### Codex (Phase 4)

`crates/takuto-core/src/codex/session.rs` spawns **`codex`** with `exec --json --skip-git-repo-check --dangerously-bypass-approvals-and-sandbox --ignore-user-config --cd <worktree>`, optional `-m <model>`, prompt as the final positional. Resume mode inserts `resume <session_id>` after `exec`. JSONL event shape: `thread.started{thread_id}` (session id), `item.completed{agent_message,text}` (assistant text), `turn.completed`, `turn.failed{error.message}`. Codex sometimes leaks tracing lines to stdout — the parser skips any line that doesn't begin with `{`. Authentication reads **`OPENAI_API_KEY`** from the worker env (bridged from `/run/takuto-secrets/codex` by `BUNDLE_SOURCING_SH`).

### OpenCode (Phase 4)

`crates/takuto-core/src/opencode/session.rs` spawns **`opencode`** with `run --format json --dangerously-skip-permissions`, optional `-m <provider>/<model>`, optional `-s <session_id>` for resume, prompt as the final positional. JSONL event shape: `step_start{sessionID}` (session id), `text{part.text}` (assistant text), `step_finish{reason:"stop"}` (terminal), `error{message}`. **Self-hosted only in v1** (spec **`lore/audits/2026-05-27-opencode-self-hosted-spec.md`**): OpenCode is the LM Studio / Ollama / vLLM / private OpenAI-compatible-gateway adapter. Pointing it at api.anthropic.com or api.openai.com is not blocked but is the wrong tool — Claude / Codex have native adapters that are strictly better for the vendor case. **Authentication**: the bundle assembler materialises **`opencode.json`** per workflow under `${data_dir}/runtime/secrets/<id>/opencode-config/` (mode 0700 dir, 0400 file) and bind-mounts it read-only into the worker at **`/home/takuto/.config/opencode`**. The file embeds the admin `[agent.providers.opencode].base_url` as `provider.self_hosted.options.baseURL`, the admin `model` as `provider.self_hosted.models.<id>{}`, and the user's optional bearer (from `user_provider_credentials kind=api_key`) as `provider.self_hosted.options.apiKey`. `context_limit` / `output_limit` render as `provider.self_hosted.models.<id>.limit.{context,output}` (positive ints only — `None`/`0` omits the key) so OpenCode knows the window of a model it can't look up on models.dev. They **default to a 7B-class coder window** (`32768` / `8192`) when the TOML omits them, so a fresh OpenCode config gets a sensible window without operator input; the dashboard can override or clear (`null` → omit). When the user saved no bearer the `apiKey` defaults to the literal string `"lm-studio"` (LM Studio's OpenAI-compat server requires a non-empty placeholder). **No `/run/takuto-secrets/opencode` file is created** and **no `ANTHROPIC_API_KEY` / `OPENCODE_PROVIDER_BASE_URL` env var is exported** — both were footguns the 2026-05-27 spec deleted (use the Claude provider for Anthropic). **Validator invariant** (`config/load.rs`): when `[agent].provider == "opencode"`, both `[agent.providers.opencode].base_url` and `model` MUST be non-empty — empty surfaces a typed 400 (`opencode_base_url_required` / `opencode_model_required`) at `PUT /api/config/agent`. **LM Studio recipe**: when LM Studio runs as a sidecar in the same docker-compose, `base_url = "http://lm-studio:1234/v1"`. When LM Studio runs on the **host Mac** behind Docker Desktop, `base_url = "http://host.docker.internal:1234/v1"`. **The model prefix is always `self_hosted/`** — it's the providerId in the synthesised `opencode.json`, not `lmstudio/`. So `model = "self_hosted/<model-id-as-reported-by-LM-Studio>"` (e.g. `self_hosted/qwen/qwen2.5-vl-7b`); bearer left blank. Also set `allow_shared_default = true` unless every user has saved a per-user bearer — otherwise the worker-secrets bundle build skips and opencode launches without its `opencode.json`, surfacing as `OpenCode error: unknown error` on the dashboard. Full failure-mode catalogue (model-prefix, `allow_shared_default`, LM Studio bind interface, Docker Desktop gVisor networking) in **`docs/troubleshooting-self-hosted-models.md`**.

**`LM_BRIDGE` — host-local model servers only.** The `LM_BRIDGE=1` make flag (layering **`docker-compose.lm-bridge.yml`**) is required **only** when the model server runs on the **host machine** (LM Studio / Ollama / vLLM) **and** Docker Desktop uses its gVisor network stack (4.34+ default) — under gVisor the DinD-nested workers cannot reach `host.docker.internal`. It adds a `socat` sidecar (`takuto-lm-bridge`, pinned `172.20.0.250:1234`) on the Takuto compose network that forwards to `host.docker.internal:${LM_HOST_PORT:-1234}`; `make start … LM_BRIDGE=1` also attaches it to the legacy `bridge` network (where `host.docker.internal` resolves under gVisor). With the bridge up, set OpenCode `base_url = "http://172.20.0.250:1234/v1"`. It is **not** needed for cloud providers (Claude/Codex, or any reachable API URL) or for a model server running **as a container inside the compose network** (reach it by service name, e.g. `http://lm-studio:1234/v1`). The intended replacement — a `TAKUTO_DNS=1` path that patches workers to resolve `lm-bridge` by name — is **not yet implemented**, so the pinned-IP sidecar is the working path.

The **Docker image** installs **Node.js 23+** from **nodejs.org** (not Node 20): the Cursor **`cursor-agent`** launcher runs its bundled **`node`** with **`--use-system-ca`**, which requires **Node ≥ 23.9** on Linux; older Node fails with `bad option: --use-system-ca`. The image installs the Cursor Agent via a **pinned tarball** (audit §3.5) from `downloads.cursor.com/lab/<version>/linux/<arch>/agent-cli-package.tar.gz` (sha256-verified per arch) extracted to **`/usr/local/share/cursor-agent/versions/<VERSION>/`**, with **`/usr/local/bin/agent`** and **`/usr/local/bin/cursor-agent`** symlinked to that launcher (a bare-script copy to **`/usr/local/bin`** breaks **`index.js`** resolution).

The image also installs **mise** (apt). **`process::worktree_has_mise_config`** detects **`.mise.toml`**, **`mise.toml`**, **`.tool-versions`**, or **`.config/mise/config.toml`**; the workflow runs **`mise install`** in the worktree, then **`run_shell_command` / `run_shell_command_streaming`** use **`mise exec -- sh -c …`** when that detection matches. **`run_command`** (argv, e.g. **`acli`**) is unchanged.

### Dashboard streaming

Raw stdout lines are turned into short lines for the web UI via **`workflow/stream_humanize.rs`**: **`humanize_agent_stream_line`** dispatches on provider. Cursor stream-json events include **assistant** text (trimmed; empty chunks omitted), **`tool_call` started** (e.g. read/write paths, shell as `$ …`, generic tools with a short **arguments** snippet when present), **`tool_call` completed** for shell with **Done/Failed (exit N)** plus an optional **truncated output tail**, and **result** — so operators still see **live progress** similar to Claude. (Cursor’s docs note **thinking** events are suppressed in print mode; internal chain-of-thought is not shown, but tool use and assistant text are.)

The **index** dashboard (`ui/src/pages/Dashboard.tsx`, React + Vite SPA) renders each workflow card (`ui/src/components/IssueCard.tsx`). After the last card, a **+** tile opens a modal that loads configured-project **To Do** issues via **`GET /api/jira/todo-tickets-manual`**: **Epics excluded**, backlog **`ORDER BY rank ASC`**, ignoring **`[jira] item_types`**. When **`[jira] jql_filter`** is non-empty, it is **`AND`**-combined with that query so operators can mirror a board filter. Choosing a row opens a **second modal** on top: **`GET /api/jira/tickets/{key}/preview`** returns **`description_markdown`** (Jira **ADF** descriptions are converted server-side to Markdown); the UI renders Markdown to HTML via the shared **`MarkdownPreview`** component (`ui/src/components/MarkdownPreview.tsx`) — **marked** + **DOMPurify** with **Mermaid diagram support** (fenced ` ```mermaid ` blocks are rendered as interactive SVG diagrams; invalid syntax shows mermaid’s built-in error message). If the operator switches to another ticket before a preview request completes, stale responses are ignored so the modal does not show the wrong issue’s description. **Close** dismisses only the detail modal; **Add to Dashboard** closes both modals and **`POST`s `/api/workflows/start-manual`** (which now adds the workflow in **Pending** state without starting the driver — the user clicks **Start** on the card to begin). The **+** is hidden when **`[jira] project_keys`** is empty and **disabled** when **`[general] max_concurrent_manual_workflows`** is exceeded (non-zero). The grid keeps a **stable ticket order** across refetches and full re-renders (pause/stop/complete): keys are **not** re-sorted by state; **new** workflows **append** at the end when they first appear in **`GET /api/workflows`**. When **`pr_url`** is non-empty (from **`GET /api/workflows`** / **`GET /api/workflows/{ticket_key}`**), the card header shows a **Show PR** link (new tab) on the right. Each card’s actions include **Go to ticket** (**`jira_browse_url`**, new tab) and **Show description** (reuses the manual-start ticket modal with **Close** only — **Start** is hidden; loads **`GET /api/jira/tickets/{key}/preview`** like the **+** picker). **Description editor** (`TicketDetailModal` edit mode): a **Write / Preview** tab bar appears above the textarea; **Preview** renders the current text via `MarkdownPreview` (including Mermaid). A **"Side by side"** checkbox (top-right of the tab bar) widens the content area into two equal panes — editable textarea on the left, live preview on the right; preview updates are **debounced 400 ms** to avoid re-renders while typing. Disabling side-by-side returns to the tabbed layout on the **Write** tab. `MarkdownPreview` lazy-loads Mermaid via dynamic `import()` and initialises it once (dark theme, `startOnLoad: false`); the mermaid bundle is code-split by Vite and only downloaded when a diagram is first encountered.

**Workflow definition buttons:** Workflow cards display a **"Workflows"** section with one button per discovered `*.toml` definition (shown for all states including **Pending**). Buttons are **topologically sorted** by `depends_on` and show five visual states: **enabled** (blue, clickable), **disabled** (gray with lock icon + tooltip listing unmet deps), **running** (blue with spinner, non-interactive), **completed** (green with check), **error** (red with X, click to retry). Clicking an enabled button on a **Pending** card calls `POST /api/workflows/{id}/run-workflow/{def}` — bootstrap runs automatically; no separate Start button needed. The component (`ui/src/components/WorkflowDefButtons.tsx`) fetches definitions from `GET /api/workflow-definitions` and re-fetches on `workflow_definitions_changed`, `workflow_updated`, and `step_completed` WebSocket events (debounced 500 ms). Definitions are shared across all workflow cards (fetched once in `Dashboard.tsx` and passed through `WorkflowGrid` → `IssueCard` → `WorkflowDefButtons`).

**Port mapping display:** Workflow cards show clickable port-mapping buttons below the action buttons. Port data comes from two sources: **API** (`editor_port_mappings` from the list/detail endpoints — covers static Docker mappings and already-tracked dynamic socat forwards) and **WebSocket** (`port_forwarded`/`port_unforwarded` events — real-time updates). The `PortMappings` component (`ui/src/components/IssueCard.tsx`) merges both, deduplicating by container port (dynamic wins). Each button opens the proxy URL (`/s/{path_token}/`) in a new tab. The API returns `(container_port, proxy_url)` pairs — the frontend displays `PORT → /s/{token}/`. Port state is managed by `useWorkflows` (`ui/src/hooks/useWorkflows.ts`) which maintains a `DynamicForwards` map keyed by ticket key.

**Run commands:** When `[[run_commands]]` is configured, workflow cards in terminal states (**Done**, **Error**, **Stopped**) display a separator followed by one button per command below the port mappings. Each row is a state machine: **idle** → `Run {name}` button (primary/blue); **running** → `Stop {name}` button (danger/red) plus `Copy` and `Open` buttons (secondary/gray). When the run command's port scanner detects a listening port inside the container, `Copy` copies the proxy URL (`/s/{path_token}/`) and `Open` opens it in a new tab. When no port is detected, both buttons are disabled with a tooltip. Each command runs in a dedicated container (`takuto-run-{ticket}-{index}`) with its own socat port forwarding and proxy token. The container receives `TAKUTO_PROXY_BASE=/s/{token}/` as an env var so dev servers can configure their base path accordingly. WebSocket events `run_command_port_forwarded`, `run_command_port_unforwarded`, and `run_command_stopped` keep the UI in sync. Run commands are automatically cleaned up when a workflow is deleted or marked done.

**Terminal output persistence:** The terminal panel **never clears** between steps — when a new step starts, a visual separator (`─── Step Name ───`) is appended to the existing output. Lines accumulate across all steps in a workflow, capped at `TERMINAL_MAX_LINES = 500` (oldest lines are evicted first). For **completed, error, and stopped** workflows, the terminal remains visible on the card in a **collapsed** state (header-only); a chevron toggle expands it to show the last available output (up to 100 lines from the backend `terminal_lines` field). **Running** and **Paused** workflows show the terminal expanded with live streaming as before.

**Workflow duration:** Cards in a terminal state (**Done**, **Error**, **Stopped**) display the **total elapsed wall-clock time** (`updated_at − started_at`) in the step info area, next to the step label. The Report modal also includes a **Total Duration** field in the Ticket Info grid for terminal-state workflows.

**Action button styling:** Buttons use a **four-category colour palette** instead of per-button unique colours: **primary** (blue — Resume, Pause, Retry), **secondary** (gray — Go to ticket, Show description, Report, Editor, Terminal), **success** (emerald — Mark as Done), and **danger** (red — Delete, Close editor). **Pause** and **Resume** buttons use **SVG icons** (⏸ and ▶) instead of text labels, with screen-reader-only text for accessibility. All buttons share the same `.workflow-action-btn` base sizing.

**Show Report button (conditional):** The **Show Report** button on each workflow card is **hidden** until `has_report` becomes `true` (i.e. the consolidation step has written the report file at least once). When `has_report` is `true`, the button is visible and opens the Report modal. The Report modal fetches the report content from **`GET /api/workflows/{id}/report`** and renders it with the `MarkdownPreview` component (full Markdown including Mermaid diagrams), with a Copy button and loading/error states.

**Report modal enhancements:** Each step row now shows a **status label pill** (Success / Failed / Skipped / Running) next to the step name. Step **output lines** show the last 3 lines by default with a "Show N more lines" expand toggle when more are available. The Ticket Info grid includes **Total Duration** for terminal-state workflows.

### Prompts

Templates live in **`[[steps]]`** within TOML workflow definition files (`name`, **`prompt`**, **`repeat`**, **`when`**, **`commands`**). Default steps use **generic natural-language** prompts (no slash-commands); teams with Claude skills can still put **`/address-ticket`** or **`/review-changes`** in a template. Placeholders: **`ticket_key`**, **`ticket_summary`**, **`ticket_description`**, **`description`** (alias for `ticket_description`), **`ticket_type`**, **`acceptance_criteria`**, **`ticket_context`**, **`base_branch`**, and **`pr_url`**. Placeholders are interpolated in both `prompt` (agent steps) and `commands` (command steps). Unknown **`{placeholders}`** are left unchanged.

**`when`** (`StepAvailability`): **`always`** (default when omitted), **`ticketing`** (only when acli/Jira is authenticated), **`no_ticketing`** (only when Jira is **not** available). Steps that do not match the current ticketing availability are **filtered out** before the agent loop runs and are **not counted** in `progress_steps_total`.

### Dev-only: mock agent (`[dev] mock_agent`)

`crates/takuto-core/src/dev_mock.rs` lets E2E tests and dashboard demos exercise the full workflow pipeline (worktree creation, step driver, terminal streaming, **Improve with AI**) without spawning a real `claude` or `agent` process and without burning Claude/Cursor tokens. When activated, the very first line of `ClaudeSession::run_prompt` and `CursorSession::run_prompt` short-circuits into a scripted mock that emits canned lines through `line_tx` (when present), honors the workflow's `CancellationToken` between every line, and returns a `Result<Session>` shaped exactly like the real one (`session_id = "mock-<uuid>"`, `output = joined script + stable marker `"[mock-agent] session complete (mocked, no AI invoked)"`). For the **Improve with AI** path the mock recognizes the `IMPROVE_SYSTEM_PROMPT` marker and returns the two-pane `"<title>\n---\n<body>"` shape so `improve_ticket`'s split-once parser stays correct. Activation precedence (high → low): per-process test override (`dev_mock::set_test_override` / `MockGuard`) → env var `TAKUTO_DEV_MOCK_AGENT=1` (or `true`/`yes`) → `[dev] mock_agent = true` in `config.toml` → off. **Off by default — never on in production.** The config flag is re-installed by a small background task in `takuto-cli/src/main.rs` at the same cadence as `ConfigWatcher` so editing `mock_agent` in `config.toml` (or calling `POST /api/config/reload`) takes effect without a restart. Operators see a `[mock-agent] enabled` log line at startup when the mock is live. No engine, driver, or container-runner code is modified — the mock returns before the container is touched.

## External actions boundary

`crates/takuto-core/src/actions/traits.rs` — `ExternalActions`:

- Jira: assign to current user (`@me`), transition, unassign, get ticket details — all via direct **`acli`** invocations (no engine-level allowlist). Scope is governed by the Atlassian token permissions the user configured.
- GitHub CLI (`gh`): all `gh` invocations run via **`gh_cli::run_gh()`** (no engine-level allowlist). Scope is governed by the GitHub token permissions (fine-grained PAT or GitHub App installation token) the user configured.
- Git: create/remove worktree (**`create_worktree`** clears an existing path under **`worktrees/<branch-slug>`** before **`git worktree add`** on new workflows so orphans are not silently reused; **`remove_worktree`** runs **`git worktree remove --force`** from **`git.repo_path`**, for dashboard **Mark as Done** / **Delete** teardown even when the tree has local changes; on **Unix**, if Git reports **permission denied** (e.g. root-owned files left by older worker containers), Takuto retries once after **`sudo -n /bin/bash -c "chown -R <euid>:<egid> -- <path>"`** when that succeeds — matches the image **`sudoers`** for passwordless **`bash`**); **`delete_local_branch`** (**`git branch -D`**, tolerant if the branch is already gone) used after **Delete** / **Mark as Done** teardown; **`create_pr`** (implemented on **`RealActions`/`DryRunActions`** but **not** called by the workflow engine — agents open PRs with **`gh`** or similar), **commit_changes**, **`configure_git_author_from_github`**, **`request_github_self_as_pr_reviewer`**
- Shell: **run_command**

Helpers: **`actions/gh_github.rs`** (`gh api user`, **`gh pr edit --add-reviewer`**). In **dry mode**, **`request_github_self_as_pr_reviewer`** does not call GitHub (returns skipped); **`configure_git_author_from_github`** still sets local **`git config`** when **`gh`** is available. **`github_app.rs`** provides **`GitHubAppTokenManager`** for JWT generation and installation token caching (see **GitHub App authentication** below).

`real.rs` and `dry_run.rs` implement this for production vs dry runs.

### GitHub App authentication

`crates/takuto-core/src/github_app.rs` — optional bot-identity authentication via a GitHub App. When **`[github]`** config fields are set, `RealActions` and `DryRunActions` receive an `Arc<GitHubAppTokenManager>` and **`configure_git_author_from_github`** uses the bot identity instead of the personal `gh` user.

**Authentication flow:**

1. **JWT generation** — `GitHubAppTokenManager::generate_jwt()` creates an RS256 JWT signed with the App's private key (`iss` = App ID, 10-minute expiry, 60 s backdate).
2. **Installation token** — `get_installation_token()` exchanges the JWT for a 1-hour installation access token via `curl -s -X POST https://api.github.com/app/installations/{id}/access_tokens`. Tokens are cached in a `tokio::sync::RwLock` and refreshed 5 minutes before expiry.
3. **`gh` + git setup** — `configure_git_and_gh_auth()` installs a git credential helper (shared builder `github_app::git_credential_helper_script`) that authenticates HTTPS `github.com` pushes/fetches. Precedence is **`$GH_TOKEN` first, then the shared App-token file** (`~/.config/gh/gh-app-token`) — GH_TOKEN-first so a per-user PAT (which `token_for(Push)` returns over the App when the user attributes commits) genuinely overrides the App token; the previous file-first order silently shadowed PATs. It also sets `git config user.name` / `user.email` to the bot identity. It does **not** call `gh auth login` or `gh auth setup-git`. **This `git config` is `--global` in the main container only** — agent-step workers run with `--entrypoint ""` and don't inherit it, so `wrap_command` installs the same helper in the worker via `GIT_CRED_HELPER_SNIPPET` (see below). Without it the worker's `git push` failed with `could not read Username for 'https://github.com'`.

**Bot identity:** commits are authored as `takuto-bot[bot] <{app_id}+takuto-bot[bot]@users.noreply.github.com>`. PRs created by the agent are attributed to the App's bot user in the GitHub UI.

**Startup behavior:** `try_create_token_manager()` attempts to parse the PEM key at process start. If it fails (malformed key, missing file), a warning is logged and Takuto falls back to personal `gh` auth — **non-fatal**. Runtime errors (invalid installation ID, insufficient permissions) surface as actionable messages at operation time.

**Centralized token service:** At startup, `main.rs` calls `start_token_file_writer()` which spawns a background `tokio::task` that proactively refreshes the installation token every 5 minutes and writes it atomically (write-then-rename) to **`~/.config/gh/gh-app-token`** on the shared `gh-auth` Docker volume. Worker containers read this file dynamically on every `gh` or `git` invocation instead of relying on a frozen `GH_TOKEN` env var baked into `docker run`. This ensures tokens stay valid even for workflows that run longer than the 1-hour GitHub App token lifetime. The `ContainerRunner` no longer captures or injects a static `GH_TOKEN`; the `wrap_command` shell preamble and `worker-entrypoint.sh` both source the token file. `GH_TOKEN` remains in `PASSTHROUGH_ENV` as a fallback for local development without a GitHub App.

**Required GitHub App permissions:** `contents` (write), `pull_requests` (write), `issues` (read), `metadata` (read).

### Generated TypeScript wire types

A subset of the dashboard's wire DTOs are mirrored from the Rust source of truth into **`ui/src/api/generated/*.ts`** by **ts-rs** (`#[derive(ts_rs::TS)]`), so that TypeScript cannot silently drift from the JSON the server sends. Generation runs as a side effect of co-located **`#[cfg(test)]`** `export_all_to(...)` tests (module name **`ts_bindings`**), which resolve their output dir via `crates/takuto-web/src/ts_bindings.rs::generated_dir()`. **`ui/src/api/types.ts`** re-exports the generated files (`export type { … } from "./generated/…"`) so frontend imports stay at `@/api/types`.

- **Regenerate:** `./scripts/generate-ts-types.sh` (runs the `ts_bindings` tests, which write the committed files). Run after changing any `#[derive(TS)]` DTO and commit the result.
- **CI drift gate:** the **`ts-types-drift`** job (`.github/workflows/ci.yml`) regenerates and `git diff --exit-code -- ui/src/api/generated` — a Rust DTO change not reflected in committed TypeScript fails the build.
- **ts-rs is built without `serde-compat`** (its parser warns on `skip_serializing_if`, breaking the zero-warning bar). Serde behavior the DTOs rely on is mirrored with explicit `#[ts(...)]` attributes: `#[ts(rename = "…")]` for field/type renames (e.g. `definition_runs`, `TerminalLine`/`WorkflowCounts` type names), `#[ts(optional)]` for `skip_serializing_if = "Option::is_none"` fields, `#[ts(type = "…")]` to pin a representation (`started_at` → `string`, `definition_runs` → `Record<string, string>`).
- **GENERATED set (drift-protected):** `WorkflowSummary`, `WorkflowCounts` (`WorkflowCountsResponse`), `RunCommandStatus`, `TerminalLine` (`TerminalLineDto`), `StepLog`, `StepStatus`, `GitHubRepo` (`GitHubRepoRow`), `GitHubIssue` (`GithubIssueRow`), `TodoTicket` (`TodoTicketRow`), `TicketPreview` (`TicketDescriptionPreview`).
- **HAND-WRITTEN (deliberately NOT generated):** `ConfigResponse`, every `*Patch` body, and the `AgentConfig`/provider family (`ConfigResponse` flattens the whole core `Config` tree + loose `[key: string]: unknown` forward-compat shapes ts-rs can't reproduce). `WorkflowEvent` must stay a FLAT optional-field struct (consumers branch on `event_type` as a plain string — not a ts-rs union candidate). `PollingStatus` is left hand-written by a stop-on-fallout call: generating it would add a required `reason` field that breaks the `{ paused }` optimistic-update literal in `usePolling.ts`. The remaining response types (`AuthStatus`, `SystemStatus`, `MarkDoneOutcome`, `WorkflowDefinition`, `Open*Response`, `Improve`/`PromptResponse`, `Workspace`, `User`, credentials) are still hand-synced; migrating each is mechanical where it maps cleanly (`MarkDoneOutcome` lives in `engine/` = bd1's lane, so deferred). The boundary is documented at the top of `ui/src/api/types.ts`.

**Authentication:** Takuto uses **multi-user SQLite-backed authentication** exclusively. There is no legacy/config-based auth fallback. **HttpOnly session cookies** (`takuto_session`) with `db-` prefix are the sole session mechanism. If the database is unavailable, all protected requests are rejected with 401 and the UI shows the setup page.

**Multi-user authentication (SQLite-backed):** Takuto uses the database (`{data_dir}/takuto.db` or the configured external backend) as the **sole** authentication source. There are **no** config-seeded credentials — the single-user `[web] dashboard_username` / `dashboard_password` keys have been removed (a stale pair left in `config.toml` is silently ignored). Session cookies are prefixed with `db-` (format: `db-<session-uuid>`). When no valid DB session exists, all protected routes return **401** (no legacy fallback). **First-launch setup:** when the database has **zero users**, `GET /api/auth/status` returns `setup_required: true` and the UI shows a **setup page** instead of the login form. The admin creates the first account via `POST /api/auth/register` (public endpoint, only available when zero users exist). The **first user** automatically becomes an admin (race-condition-guarded via SQLite transaction). Passwords must be at least **12 characters** and are hashed with **argon2id** (OWASP-recommended). Each user gets **8 single-use recovery codes** on registration (format `XXXX-XXXX`, confusable characters excluded). The setup page requires the admin to acknowledge and save recovery codes before proceeding. Users have roles (`admin` / `user`) and can be suspended. Additional users are created via the admin API (`POST /api/users`).

**Common:** The browser sends the cookie automatically on same-origin **`fetch`** (use **`credentials: 'same-origin'`**) and on **WebSocket** **`/ws`**. **Cookies are host-scoped:** **`http://localhost:8080`** and **`http://127.0.0.1:8080`** do **not** share the session cookie — use one hostname consistently (otherwise you will appear logged out or get **`401`** after a successful login).

**Multi-user database schema** (`{data_dir}/takuto.db`):

| Table | Purpose |
|-------|---------|
| `users` | User accounts: `id` (UUID), `username` (unique), `role` (admin/user), `suspended`, `created_at` |
| `credentials` | Hashed passwords/passkeys per user: `id`, `user_id` (FK), `kind` (password/passkey), `data` (argon2 hash) |
| `recovery_codes` | Single-use recovery codes: `id`, `user_id` (FK), `code_hash` (argon2), `used` flag |
| `sessions` | Database-backed sessions: `id` (UUID), `user_id` (FK), `data` (JSON), `expires_at`, **`last_seen_at`** (Unix seconds; bumped by the auth middleware on the sliding-extend path), **`created_at_unix`** (Unix seconds at session creation; enforced by the absolute 30-day TTL gate) |
| `login_attempts` | **Persistent per-user authentication audit** (plan-02 AC-3). Columns: `id`, `user_id` (FK, cascade), `kind` (`'password'` \| `'recovery'` CHECK), `attempted_at` (Unix seconds), `success` (0/1). Used to enforce per-user lockout (5 failures in 10 min → 429 on the 6th attempt). |
| `user_repositories` | Per-user repository access control (schema only — enforcement in future PR) |
| `container_users` | Container isolation tracking (schema only — enforcement in future PR) |
| `user_work_item_flows` | Per-user, per-workspace flow definitions. PK `(user_id, workspace_name)`; `flows_json` (TEXT, default `'[]'`) holds the full ordered flow list as a JSON array; `updated_at` (Unix ms). FK `user_id` → `users(id)` ON DELETE CASCADE. **No row** = not seeded yet for that workspace; **empty array** = seeded then emptied by the user (the two states are deliberately distinct so the seeder never re-fills a list the user intentionally cleared). |
| `user_jira_credentials` | Per-user Jira API credential (singleton, PK `user_id` FK cascade). The **token** is sealed with the envelope scheme (`ciphertext`/`nonce`/`wrapped_dek`/`wnonce`), modeled on `user_github_credentials`. Plain metadata columns: `site` (base URL), `email` (Basic-auth username), `account_id` / `account_name` (captured at validation), `last_validated_at`. DAO: `db/jira_credentials.rs`. |
| `schema_migrations` | Migration version tracking |

Schema migrations run automatically at startup via **sqlx**'s migration runner driven by `DialectAwareMigrationSource` (plan-11). Source files live at `crates/takuto-core/migrations/*.sql`; each one is hand-translated to portable SQL (single source per migration; the source rewrites `BLOB` → `BYTEA`, the `INTEGER PRIMARY KEY AUTOINCREMENT` shapes, and a closed set of MySQL quirks per backend at apply time — see `db/migrate.rs::translate_for_backend`). Applied versions are tracked in the sqlx-managed `_sqlx_migrations` table — re-running is a no-op and checksum drift is caught at startup. **Current set: 12 migrations** (V1 users/credentials/sessions/recovery_codes/user_repositories/container_users → V2 login_attempts + session columns → V3 workspace_commands → V4 user_worktree_commands → V5 repositories/user_repositories → V6 user_provider_credentials/user_github_credentials/credential_audit/onboarding_state → V7 `system_metadata` → V8 `work_items_and_logs` → V9 `work_items_repository_id` → V10 `user_work_item_flows` → V11 `work_items_soft_delete` → V12 `user_jira_credentials`).

**Migration-dialect ceiling.** The single-portable-source + per-backend regex transform is intentionally not a SQL parser; it holds only while the per-backend delta is a handful of context-free textual rewrites. Switch to **per-backend migration files** (committed under `migrations/{backend}/`, selected by `DbBackend` in `DialectAwareMigrationSource::resolve`) the first time a rule would need to understand SQL structure (balance parens, know a token's table/column, skip a match inside a string/comment), the backends diverge in DDL *shape* rather than spelling (a statement one backend needs and another forbids, differing column counts, a constraint that must be split), or a single migration accumulates more than a couple of its own backend-specific rewrites. New backend syntax must never land by editing the rules that feed an already-applied version (migrations are checksummed/immutable) — it goes in a new migration or a per-backend file. The full criteria live in the `db/migrate.rs` module doc.

**External database backend** (`[database]` block — plan-11):

By default Takuto stores auth state in SQLite at `{data_dir}/takuto.db`. Operators can point at PostgreSQL, MySQL, or MariaDB by setting `[database].connection`:

```toml
[database]
connection = "postgres://takuto:s3cret@db.example:5432/takuto"
# max_connections = 10
# acquire_timeout_secs = 30
# idle_timeout_secs = 600
# fail_fast = true
# import_from_sqlite = true
```

Supported schemes: `sqlite://`, `postgres://` / `postgresql://`, `mysql://` (covers MariaDB). Empty / omitted `connection` keeps the current SQLite default — zero-config behaviour for single-instance deployments. `GET /api/config` redacts the password component of the URL; `PUT /api/config` cannot patch the section (backend change is restart-only, same policy as `web.host` / `web.port`).

`Database::connect(data_dir, &DatabaseConfig, allow_auto_generate_secret_key)` is the entry point. Flow:

1. Build the backend-specific sqlx pool with the configured pool tuning.
2. Run `SELECT 1` with a 5 s timeout. On failure:
   - `fail_fast = true` (default) → error propagates; CLI startup aborts with a "Database backend unreachable" log line (URL redacted).
   - `fail_fast = false` → log a warning and **fall back to local SQLite** at `{data_dir}/takuto.db` for the current process.
3. Apply all sqlx migrations through `DialectAwareMigrationSource`.
4. If backend ≠ SQLite **and** `import_from_sqlite = true` (default) **and** `{data_dir}/takuto.db` exists **and** the target has no `system_metadata.import_complete` row, run the **one-shot SQLite → remote importer**:
   - Open the source SQLite file read-only.
   - Inside a single sqlx transaction, copy every user-data table in FK-dependency order (`users` → `repositories` → all children → `user_repositories`). Explicit column lists per table; autoincrement IDs are regenerated by the target on INSERT.
   - Overwrite imported `sessions.expires_at = '0'` so users are forced to re-authenticate post-cutover (plan-11 §8.6).
   - Upsert `system_metadata.{import_complete, import_source_path}` and COMMIT.
   - A mid-import crash leaves the target untouched (single BEGIN/COMMIT). On next start the marker check sees `import_complete` is still NULL and reruns from scratch.

Operators who want to **re-import** after a target wipe delete the `import_complete` row manually (one SQL statement against the target); there is no CLI to redo the import on purpose (footgun avoidance).

**Env-var override:** `TAKUTO_DATABASE_CONNECTION` overrides `[database].connection` from `config.toml` at startup. Used by `docker-compose.postgres.yml` / `docker-compose.mariadb.yml` so the operator's checked-in config stays SQLite-default while the deployment overlay points at the sidecar DB.

**Docker Compose variants:** the base `docker-compose.yml` runs Takuto against SQLite (default, unchanged). Two overlays add an external DB sidecar:
- `docker-compose.postgres.yml` — adds a `postgres:16` service and sets `TAKUTO_DATABASE_CONNECTION=postgres://...@postgres:5432/...`
- `docker-compose.mariadb.yml` — adds a `mariadb:11` service and sets `TAKUTO_DATABASE_CONNECTION=mysql://...@mariadb:3306/...`

`make start BACKEND=postgres` (or `mariadb`) layers the right overlay; `BACKEND=sqlite` (default) uses the base file only. Every Make target (`start` / `stop` / `restart` / `logs` / `ps` / `build`) respects the variable.

Every DAO and call site goes through the backend-agnostic `&DbAdapter` (wrapping sqlx pools); no production code path mentions `rusqlite::Connection` or per-backend pool types. The dialect-aware transform plus typed `DbValue` / `DbRow` accessors keep SQL portable across SQLite / PostgreSQL / MySQL.

**CORS:** The Axum router applies an **explicit origin allowlist** (not `CorsLayer::permissive()`). Allowed origins come from **`[web] cors_origins`**; when that list is empty (default), the origin is auto-computed from `host`/`port` (e.g. `http://localhost:8080`). Methods are restricted to **GET, POST, PUT, DELETE, PATCH**; allowed headers to **Content-Type**; **`Access-Control-Allow-Credentials: true`** is set so session cookies work cross-origin. Requests from unlisted origins receive no `Access-Control-Allow-Origin` header. For deployments behind a reverse proxy or TLS terminator, set `cors_origins` to the external-facing origin (e.g. `["https://takuto.example.com"]`). This setting is **startup-only** — not patchable via `PUT /api/config`.

**CSRF (Origin/Referer allowlist):** Every `POST/PUT/DELETE/PATCH` against `/api/*` (both `api_public` and `api_protected`) is gated by the **`csrf_middleware`** in `crates/takuto-web/src/middleware/csrf.rs`. It reads the `Origin` header (preferred) or falls back to the `Referer` (extracting `scheme://authority`), and rejects with **`403 Forbidden`** (body `{"error":"missing or invalid Origin/Referer"}`) if the value is missing or not in `[web] cors_origins` (or the auto-computed default). Safe methods (`GET/HEAD/OPTIONS`) short-circuit before the check. On `api_protected` the CSRF layer wraps the auth layer so a planted cross-origin POST is rejected **before** any DB session lookup — including against `/api/auth/login`, which means a bad-origin login attempt returns 403, not 401. The shared-port `/s/*` proxy is **exempt** (it authenticates via the opaque path token in the URL; the dashboard cookie is not the gate there).

**Security response headers:** The **`security_headers_middleware`** (in `crates/takuto-web/src/middleware/security_headers.rs`) is the **outermost** layer on the top-level router, so it injects defence-in-depth headers on every response — including 401 from the auth middleware, 403 from CSRF, 404 from the static fallback, and OPTIONS preflights. The dashboard policy:

| Header | Value |
|---|---|
| `Content-Security-Policy` | `default-src 'self'; img-src 'self' data: blob:; script-src 'self' 'wasm-unsafe-eval'; style-src 'self' 'unsafe-inline'; connect-src 'self' wss:; frame-ancestors 'self'; base-uri 'self'; form-action 'self'` |
| `X-Frame-Options` | `SAMEORIGIN` |
| `Referrer-Policy` | `strict-origin` |
| `X-Content-Type-Options` | `nosniff` |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` (only when HTTPS context — `X-Forwarded-Proto: https` or any `https://` cors origin) |

For `/s/*` proxy responses the policy is **loosened**: no `Content-Security-Policy` (the embedded editor uses inline scripts), no `X-Frame-Options`, and `Referrer-Policy` is **overridden to `no-referrer`** (the session path token is sensitive and must not leak via `Referer` to outbound links). `X-Content-Type-Options: nosniff` is still set.

**Login rate-limit + account lockout (plan-02 AC-3):** Authentication endpoints are gated by two complementary layers.

- **Per-IP rate limit (`tower_governor`):** wired only on `POST /api/auth/login` and `POST /api/auth/recover` (the brute-force-prone routes). Quota: **1 permit / 6 seconds with a burst of 10** — i.e. 10 requests/min/IP steady-state, with a 10-request burst before the throttle engages. Returns **`429 Too Many Requests`** with a `Retry-After` header. The custom **`LoginKeyExtractor`** (in `crates/takuto-web/src/server.rs`) prefers `X-Forwarded-For` then `X-Real-IP` then the connection peer address (so reverse-proxy deployments work out of the box), and falls back to a sentinel string only in tests where no peer addr is available.
- **Per-user persistent lockout:** every login or recovery attempt against a **known** `user_id` is recorded in `login_attempts` (kind = `'password'` or `'recovery'`). When `failed_count_in_window(user_id, kind, 600s) >= 5`, the handler returns **`429`** with body `{"error":"account temporarily locked, try again in N minutes"}` and a `Retry-After` header pointing at `oldest_failure_ts + 10min - now`. On successful login the failed counter for that `(user_id, kind)` is cleared. **Unknown usernames are never recorded** — otherwise the 429 boundary would leak account existence (G/W/T 3.9). The password and recovery counters are independent (filtered by `kind`). A `tracing::warn!` line with `event="login_lockout"` accompanies every block. Admins clear the counters via **`POST /api/users/{id}/unlock`** (clears both kinds → 204). The unlock endpoint is admin-gated.

**Session lifecycle (plan-02 AC-5):** Sessions live in the `sessions` table and use a sliding-extend model with an absolute cap.

| Constant (`crates/takuto-web/src/auth.rs`) | Value | Meaning |
|---|---|---|
| `SESSION_IDLE_TTL_SECS` | 24 h | A session is rejected once it has been idle this long (`expires_at` check). |
| `SESSION_ABSOLUTE_TTL_SECS` | 30 days | A session is rejected even when active once `now - created_at_unix` exceeds this. The row is lazily deleted on the rejection path. |
| `SESSION_EXTEND_THRESHOLD_SECS` | 5 min | The auth middleware UPDATEs `last_seen_at` + slides `expires_at` only when `now - last_seen_at` exceeds this — caps the write rate to one UPDATE per session per 5 min of activity. |

`create_db_session` writes `last_seen_at = created_at_unix = now`. `validate_db_session` runs the absolute-TTL gate first (delete + None), then the `expires_at` gate (None), then the sliding-extend UPDATE if the threshold is crossed. There is a `validate_db_session_no_extend` variant for code paths that need a check without the write (rare).

**Session rotation:** `POST /api/auth/login` consults `[web] kick_other_sessions_on_login` (default **`true`**). When true, prior sessions for the same user are deleted before the new session is created — single-session enforcement. Set to `false` for multi-client concurrent sessions. **Role changes invalidate all sessions:** `PATCH /api/users/{id}` calls `delete_user_sessions(target_user_id)` whenever the role actually changes, so the target user must re-authenticate with the new role.

**Test clock seam:** `crates/takuto-web/src/auth.rs` exposes `now_unix()`, `set_test_now_unix(t)`, and `clear_test_now_unix()`. Tokens have no expiry; they live as long as the session container and are removed from the `PathTokenRegistry` before port teardown. Implementation: `crates/takuto-web/src/routes/sessions.rs`. |

Static files: embedded from `ui/dist/` (React + Vite SPA; built via `npm run build` in the `ui/` directory). The `login.html` and `config.html` pages are React routes. **Config page** is **runtime settings** only (dashboard login + the two concurrency caps); it shows a read-only JSON snapshot of the rest. **`PUT /api/config`** from the UI sends only that patch. **Vite dev proxy:** `ui/vite.config.ts` proxies **`"/s/"`** (with trailing slash) to the backend — the trailing slash is required to avoid intercepting `/src/main.tsx` and other SPA source requests that start with `/s`.

## Configuration (`Config`)

Loaded in `crates/takuto-core/src/config/` (module split across `agent.rs`, `general.rs`, `git.rs`, `jira.rs`, `web.rs`, `runtime.rs`, `template.rs`, `mod.rs`) — sections:

- **Root (before any `[table]` in TOML)**: **no** config-level command arrays. `[[run_commands]]` (dashboard run/stop buttons) is **not** read from `config.toml` — it is per-`(user, workspace)` DB state edited from the Worktree Settings tab; a stale `[[run_commands]]` entry in the file is logged and ignored. All workflow step definitions live in **TOML workflow definition files** in the `workflows/` directory — there are no inline step arrays in `config.toml`. Shapes are in **`workflows/*.example.toml`** (copy to e.g. **`workflows/implement_ticket.toml`** — local overrides are gitignored). Each definition file has top-level **`name`**, optional **`depends_on`**, and **`[[steps]]`** entries. Each step has `name`, `repeat` (default 1), optional **`when`** (`always` / `ticketing` / `no_ticketing`, default `always` — see **Prompts** section). A step is either an **agent step** (`prompt` and/or `skills`) or a **command step** (`commands` — array of shell strings executed sequentially, no AI). The two modes are mutually exclusive. Users can refer to `config.toml.example` for documentation. Workflow step files and all other non-patchable fields: edit **`config.toml`** on disk; the **config file watcher** (5 s polling interval) will detect the change and hot-swap the in-memory config without a restart, or use **`POST /api/config/reload`** for immediate effect. Fields that require a restart even after reload: `web.host`, `web.port`, `web.cors_origins`, `docker.*`. **`{ticket_context}`** is built with explicit **UNTRUSTED_JIRA** framing and optional size limits — Jira text remains an untrusted prompt surface; use Jira permissions and human PR review.

### Config persistence (`ConfigWriter`)

`crates/takuto-core/src/config_writer.rs` — the **single write path** for `config.toml`. All dashboard and wizard writes funnel through this module; request handlers never touch the filesystem directly.

**Write protocol:** validate → acquire advisory file lock (`fs2::FileExt::lock_exclusive` on `config.toml.lock`) → serialize to TOML → write to `config.toml.tmp` → `fs::rename` over `config.toml` (atomic on POSIX) → record write timestamp → release lock. Invalid configs are rejected **before** any I/O.

**Self-authored write detection:** `ConfigWriter` records the epoch-millis timestamp of each successful write in a shared `AtomicU64`. The `ConfigWatcher` checks this timestamp and skips reload events that fall within a 2 s window after an API write — this prevents a write→detect→reload loop.

### Config file watcher (`ConfigWatcher`)

`crates/takuto-core/src/config_watcher.rs` — a background task that polls `config.toml` for mtime changes every 5 s (configurable). When a change is detected:

1. Skip if the change is self-authored (within 2 s of an API write).
2. Read, parse, and validate the new config.
3. If valid: hot-swap the `Arc<RwLock<Config>>`. Log warnings for restart-required fields (`web.host`, `web.port`, `web.cors_origins`).
4. If invalid: log a warning with the validation error; keep the current in-memory config.

Uses a polling approach (`mtime` comparison) rather than `inotify` because inotify does **not** fire for host-side edits on Docker Desktop for macOS (VirtioFS/gRPC-FUSE bind mounts). Polling works universally.

The watcher is spawned in the main `tokio::select!` loop alongside the snapshot syncer, ticketing pollers, PR merge poller, and web server.

## Jira poller

`crates/takuto-core/src/jira/poller.rs`: on an interval, if `project_keys` non-empty, lists **To Do** tickets (via the injected `TicketLister` — production is `JiraClient` / `acli`, see **read-side trait seams** below), skips keys that already exist in the workflow map, and only starts new workflows when **`dashboard_workflow_count`** is less than **`effective_max_active_workflows()`** (**`[general] max_active_workflows`**, or **`max_concurrent_workflows`** when **`max_active_workflows`** is **`0`**). **Every row still on the dashboard counts** — **Done**, **Paused**, **Stopped**, **Error**, and in-progress — until **Mark as Done** or **Delete** removes it from the map. **Restored** workflows are in the map before the first poll, so they consume slots. Uses `cancel_token` for shutdown. When **`AppState.polling_paused`** is set (dashboard **Pause polling**, **`POST /api/polling/pause`**, or **`[general] auto_polling = false`** at process start), the poller still waits on the interval but **skips** `poll_once` (including the initial poll on startup if already paused). In-memory pause/resume toggles are **not** written to **`config.toml`**; a restart applies **`auto_polling`** again if set. Workflow snapshot resume is unchanged (see **How the binary starts**). The **`[polling]`** policy (filtering, single auto-start flow, parallel-item cap) layers on top of this loop — see **Item polling policy** below.

### Read-side trait seams (testability)

So the **poll->start** path and ticket-detail retrieval are testable with **no live `acli`**, `crates/takuto-core/src/jira/source.rs` defines two segregated read traits (Interface Segregation): **`TicketLister`** (`list_todo_tickets`, used by the poller) and **`TicketReader`** (`get_ticket_details`, used by the bootstrap "Retrieve Details" step). `JiraClient` implements both by delegating to its inherent methods. **`TicketListerFactory`** builds a per-repo `Arc<dyn TicketLister>` (the poller resolves `repo_path` fresh each cycle — it changes on workspace switch); production passes **`DbBackedJiraSourceFactory`** (see **Per-user Jira credential** below), constructed in `takuto-cli/src/main.rs` and handed to `JiraPoller::new`. `#[cfg(test)]` fakes (`FakeTicketLister` / `FakeJiraSourceFactory`) let `poller.rs` tests drive `poll_once` end-to-end (fake tickets → `WorkflowEngine::start_workflow` → in-memory rows) without any external binary; with an empty `workflow_definitions_dir` no dep-free def runs, so no git/docker is touched.

**Per-user Jira credential (REST read path + endpoints).** A user can paste a per-user Jira credential (site URL + account email + API token) instead of relying solely on the host-wide `acli` auth. The token is the secret (sealed in `user_jira_credentials`; site/email/account stored as plain columns). `crates/takuto-core/src/jira/rest.rs` provides **`JiraRestClient`** (implements both `TicketLister` and `TicketReader` against the Jira Cloud REST API `/rest/api/3/*`) over a **`JiraHttp`** boundary (`RealJiraHttp` shells `curl --user email:token`; tests inject a mock). **`validate_jira_credential`** hits `GET /rest/api/3/myself` and returns typed errors (`invalid_token` 401 / 404, `unauthorized` 403, `jira_transport_error` → 502). **`resolve_rest_credential(db, user_id)`** reads the row and decrypts the token with the master key. **Fallback to `acli`:** both the poller's **`DbBackedJiraSourceFactory`** (resolves the poller owner's credential lazily inside `list_todo_tickets`) and the bootstrap "Retrieve Details" step (resolves the workflow owner's credential) use the REST client **only when** a per-user credential exists, otherwise they fall back to the global `acli` `JiraClient`. Endpoints (session-auth, per-user): **`POST /api/users/me/jira-credential`** `{ site, email, token }` → validate + seal + store, returns `{ site, email, account: { account_id, display_name } }`; **`DELETE /api/users/me/jira-credential`**. Status folded into **`GET /api/users/me/credentials`** (`jira: { site, email, account_id, account_name, last_validated_at } | null`) and surfaced as **`jira_credential_present: bool`** on **`GET /api/onboarding/status`** for authenticated callers. Every mutation writes a `credential_audit` row (`kind = jira_credential`) co-committed in the same transaction.

Container orchestration has a matching seam: `crates/takuto-core/src/container/runtime.rs` defines **`ContainerRuntime`** over the **process-spawning** docker ops (`is_available`, `discover_worker_image`, `cleanup_for_ticket`), with **`DockerRuntime`** delegating to the existing `ContainerRunner` associated functions and a `#[cfg(test)]` `FakeContainerRuntime`. The pure `docker run` argument assembly (`ContainerRunner::wrap_command` / `wrap_shell_command`) is already unit-tested directly since it only builds strings.

## Item polling policy (`[polling]`)

`crates/takuto-core/src/config/polling.rs` defines the admin-tunable `[polling]` section that shapes what the Jira / GitHub pollers auto-add, which flow they auto-start, and how many items run in parallel. It is read **live** by both pollers each cycle (no restart), stored in `config.toml`, and patched only through the admin-gated **`PUT /api/config/polling`** endpoint. There are **no secrets** in the section, so `GET /api/config` surfaces it unredacted via the flattened `ConfigResponse`.

**Schema** (`PollingConfig`):

- **`auto_start_flow`** (`String`, default `""`) — slug of the single flow the poller auto-starts for each discovered item.
- **`max_parallel_items`** (`u32`, default `0` = unlimited) — cap on items occupying a concurrency slot at once.
- **`max_parallel_per_user`** (`bool`, default `false`) — when `true`, `max_parallel_items` is counted **per workflow owner**; when `false`, **globally**.
- **`[polling.jira] summary_keywords`** (`Vec<String>`, default empty) — case-insensitive ANY-substring filter on the ticket summary.
- **`[polling.github] labels`** (`Vec<String>`, default empty) — exact label-membership, ANY match.
- **`[polling.github] title_keywords`** (`Vec<String>`, default empty) — case-insensitive ANY-substring filter on the issue title.

Jira **item types are not duplicated** here — they reuse the existing **`[jira] item_types`** field (which `PUT /api/config/polling` can also patch via its top-level `item_types`).

**Filter semantics** (`matches_any_keyword`): an empty keyword/label list means **no filter** (matches everything). A non-empty keyword list matches when the haystack contains **any** keyword as a case-insensitive substring; a non-empty label list matches when the issue carries **any** listed label (exact membership). Blank entries are rejected by `Config::validate` at write time (`config/load.rs`) — `polling.jira.summary_keywords`, `polling.github.labels`, and `polling.github.title_keywords` may not contain empty/whitespace-only strings (this avoids a degenerate match-everything filter). There is no validation on `auto_start_flow` (the slug is resolved per-workspace/user at runtime) or `max_parallel_items` (`0` is valid).

**Poller filtering** — after fetch, each poller clones its filter lists before dropping the config lock:

- **Jira** (`jira/poller.rs::poll_once`): `tickets.retain(|t| matches_any_keyword(&t.summary, &summary_keywords))`. Item types are already applied upstream via `list_todo_tickets(project_keys, item_types)`.
- **GitHub** (`github/poller.rs::poll_once`): keeps an issue when (`labels.is_empty()` **or** any of its labels is in the filter) **and** `matches_any_keyword(&title, &title_keywords)`. `github/mod.rs` parses `labels[].name` into `GitHubIssue.labels` in `fetch_open_issues`; the field is additive and the `GET /api/github/issues` handler tolerates it.

Both pollers log an `info!` with pre/post counts after filtering, so the effect is observable under `RUST_LOG=info`.

**Single auto-start flow** (`workflow/engine/lifecycle.rs::start_workflow`, poller path only): the engine builds the set of all dependency-free (no `depends_on`) flows as before, then narrows by `polling.auto_start_flow`:

- **empty slug** → start **all** dep-free flows (legacy behavior).
- **slug present and in the dep-free set** → start **just that one** flow.
- **slug present but absent** from the dep-free set → `warn!` and start **nothing** (the row is still added to the dashboard so the admin notices — no silent fallback).

This applies only to poller-created workflows. The dashboard **+** manual flow uses `add_to_dashboard`, which spawns no drivers and is unaffected.

**Parallel-item cap.** `repository.rs::count_occupying_slot_for_user(user_id: Option<&str>)` counts workflows whose state `occupies_concurrency_slot()`, optionally filtered to one owner; `engine/mod.rs::active_item_count(user_id)` exposes it. Enforcement:

- **Pollers** (`jira/poller.rs`, `github/poller.rs`): derive `item_slots` from `max_parallel_items` (scope `Some(owner)` when `max_parallel_per_user`, else global `None`), keep the legacy `effective_max_active_workflows` gate, and take **`slots = legacy_slots.min(item_slots)`**. So `max_parallel_items = 0` preserves exactly the legacy ceiling; a non-zero value tightens it. (The two ceilings count slightly different sets by design — the legacy gate counts every dashboard row, the item cap counts only slot-occupying states.) The poller skips the cycle when 0 slots remain.
- **Manual starts** (`routes/workflows/manual.rs`): after the existing `max_concurrent_manual_workflows` check, when `polling.max_parallel_items > 0` the handler counts slot-occupying items (per-user when `max_parallel_per_user`, else global) and returns **`409 CONFLICT`** with a message when at cap — reusing the existing 409 → `showToast` path in `Dashboard.tsx` (no new UI signal). The legacy `max_concurrent_manual_workflows` ceiling stays **independent** of the `[polling]` cap; manual starts must clear both.

## PR merge status poller

`crates/takuto-core/src/github/pr_merge_poller.rs`: a lightweight background poller that checks whether PRs associated with workflows have been merged on GitHub. Runs on a configurable interval (**`[general] pr_merge_poll_interval_secs`**, default **60**, **`0`** disables). On each tick, iterates over all workflows in the engine map, filters to those with a non-empty **`pr_url`** and **`pr_merged == false`**, parses the GitHub PR URL (`parse_pr_url` in `github/mod.rs` extracts `owner/repo` and PR number from `https://github.com/{owner}/{repo}/pull/{number}`), and calls **`gh api repos/{owner_repo}/pulls/{number} --jq .merged`**. When the API returns `true`, sets **`workflow.pr_merged = true`** and broadcasts a **`workflow_updated`** `WorkflowEvent` (with **`pr_merged: Some(true)`**) so the dashboard updates in real time. Errors are logged at **`warn`** level and do not abort the polling loop. The poller respects the shared **`CancellationToken`** for graceful shutdown. It is wired into the main `tokio::select!` loop in `crates/takuto-cli/src/main.rs` alongside the ticketing pollers, snapshot syncer, and web server.

The dashboard renders a **"Merged"** badge (purple pill with a checkmark icon, styled via `.workflow-merged-badge` in `styles.css`) next to the **"Show PR"** link when **`pr_merged`** is true. The badge appears within one polling cycle of the PR being merged, pushed via WebSocket without a page refresh. **`WorkflowSummary`** includes **`pr_merged: bool`** so the badge state is available from both **`GET /api/workflows`** and **`GET /api/workflows/{ticket_key}`**. The **`pr_merged`** field is persisted in **`PersistedWorkflowRecord`** (`#[serde(default)]` for backward compatibility) so merge status survives container restarts.

## Process management

`crates/takuto-core/src/process.rs`: `ProcessHandle::spawn`, streaming readers, timeouts, cancellation; Unix uses **process groups** so child trees can be killed together.

## Docker entrypoint and CLI helpers

- **`docker/entrypoint.sh`**: root section re-execs as `takuto` via **`setpriv --reuid --regid --init-groups`** (preserves the controlling TTY; `runuser` would call `setsid()` and detach it). `setup` mode (5 steps — required: `gh` + agent provider auth; Step 1 (GitHub CLI / `gh auth login`) is **skipped** when **`[github] app_id`** is non-zero in config (GitHub App provides tokens automatically at runtime — no interactive auth needed); Step 2 (Atlassian CLI / `acli`) is **skipped** when `ticketing_system` is not `jira`; optional: repo clone; Step 5: report generation toggle). Agent auth reads **`[agent] provider`** from config: `claude` → launches **`claude`** (full setup wizard; user exits with `/exit` when done; skipped when **`CLAUDE_CODE_OAUTH_TOKEN`** is set); `cursor` → `agent login` (skipped when **`CURSOR_API_KEY`** is set). `NODE_OPTIONS=--dns-result-order=ipv4first` is exported in setup mode so Claude's OAuth callback server binds to `127.0.0.1` (IPv4) not `[::1]` (IPv6). **`make auth`** runs with **`--network=host`** so the host browser's OAuth redirect reaches the callback server inside the container. Normal mode: **validates config.toml mount** (hard error with actionable message if missing — the image ships without baked-in config), **warns if takuto.env is not mounted** (optional), then **`takuto preflight`** (Phase 0 — always exits 0; **`TAKUTO_PREFLIGHT_ERROR`** capture is retained as a one-release back-compat fallback for the DB-unavailable case, but the dashboard now reads `GET /api/onboarding/status` for the structured `SystemStatus` snapshot), **`takuto docker-hooks startup`** (`[docker] compose_up_commands`, on failure: appends error to **`TAKUTO_PREFLIGHT_ERROR`**, logs warning, continues), **workspace check** (`/workspace/.git`, on failure: appends error to **`TAKUTO_PREFLIGHT_ERROR`**, logs warning, continues), then **`exec takuto`**. In all failure cases the web server starts and the dashboard shows a degraded-mode banner with structured warnings from `/api/onboarding/status` so the operator knows what to fix. **`config.toml` missing is the only hard exit** — everything else is degraded mode.
- **Distributed image**: the Docker image ships **without** `config.toml`, `takuto.env`, or user workflow files baked in. Users must volume-mount all three at runtime. Example reference files are available inside the image at **`/etc/takuto/examples/`** (`config.toml.example`, `takuto.env.example`, `workflows/*.example.toml`). To extract them: `docker run --rm --entrypoint cat takuto /etc/takuto/examples/config.toml.example > config.toml`. The `.dockerignore` excludes `config.toml`, `takuto.env`, and user workflow files from the build context to prevent secrets from leaking into image layers.
- **Compose mounts**: **`config.toml`** is mounted at **`/etc/takuto/config.toml`** as **read-write** (`:rw`) so the `ConfigWriter` can persist dashboard changes; `takuto.env` and `workflows/` remain **read-only** (`:ro`). **`docker-compose.yml`** bind-mounts **`./workflows` → `/etc/takuto/workflows`** so dynamic TOML workflow definitions are available to the engine. Without that mount, the entrypoint fails with a clear error message telling the user how to mount the config. The bind mount must include your **local** workflow definition files (e.g. **`implement_ticket.toml`**, **`address_pr_comments.toml`**, **`merge_base.toml`**) — copy from **`*.example.toml`** in the repo; tracked examples alone are not used at runtime.
- **`/etc/takuto/` ownership (task #37, Phase 2c)**: the root section of **`docker/entrypoint.sh`** does **`chown takuto:takuto /etc/takuto`** (non-recursive) before `setpriv`'ing to the takuto user. `ConfigWriter::write_atomic` writes a `.tmp` sibling then `rename(2)`s — POSIX requires write on the **parent directory** but not the target file, so this single non-recursive chown is enough to make `PUT /api/config/agent` persist across restarts. The host's `config.toml` ownership stays as the user set it; only the container-side directory entry flips. If the chown fails (e.g. the dir is on a read-only mount), the entrypoint logs a warning and continues — the boot-time `config_dir_not_writable` SystemStatus warning (emitted by **`docker_hooks::check_config_dir_writable`** at boot and on every `PUT /api/config/agent` refresh) surfaces the issue in the dashboard. The warning is platform-level (`severity = "critical"`, not user-filterable), so every authenticated caller sees it until the dir becomes writable.
- **`ConfigWriter` in-place fallback (task #38, Phase 2c continued)**: when `config.toml` is bind-mounted as a **single file** under Docker (the default `./config.toml:/etc/takuto/config.toml` mount), `rename(2)` returns `EBUSY` because Docker holds the file inode busy. The atomic rename remains the fast path; on EBUSY the writer falls back to: (a) best-effort `fs::copy(config.toml -> config.toml.bak)` so power loss mid-write preserves the previous valid content; (b) `OpenOptions::truncate(true)` + `write_all` + `sync_all` on `config.toml` in place; (c) `rm` the orphan `.tmp`. The fallback updates `last_write_epoch_ms` identically so `ConfigWatcher`'s self-write dedup keeps working. `ConfigWriter::used_inplace_fallback()` returns an `Arc<AtomicBool>` that latches `true` on first fallback (sticky for the process lifetime); the `PUT /api/config/agent` refresh path reads it and pushes an **info**-severity `config_file_bind_mounted` warning so admins know which write protocol is active. There are two distinct warning codes: **`config_dir_not_writable`** (critical, dir is broken, saves will fail) and **`config_file_bind_mounted`** (info, dir is fine but the file is bind-mounted, saves succeed via fallback). Neither is in `apply_user_warning_filter`'s allow-list, so both pass through to every authenticated caller unchanged.
- **Container engine access**: the default Compose stack does **not** provide a Docker/Podman daemon. Image installs Debian’s **`docker.io`** package for the **`docker`** CLI (no in-container daemon). To give the CLI a daemon, merge **`docker-compose.dind.yml`** — a **Docker-in-Docker sidecar** that adds a **`docker:27-dind`** container running a real Docker daemon. Takuto connects via **`DOCKER_HOST=tcp://dind:2375`** over the internal compose network. The sidecar runs **`--privileged`** (required by DinD) but is isolated to its own container; Takuto itself gains no extra privileges. The **`workspace`** named volume is shared so paths like **`/workspace/my-project/…`** resolve identically from both containers. A **`dind-storage`** volume persists Docker image layers across restarts. The entrypoint waits for the DinD daemon to become ready (retries `docker info` against **`DOCKER_HOST`**) before starting Takuto. Works on all platforms (macOS Podman, macOS Docker Desktop, Linux). **`podman-compose`** (Python) does not read `COMPOSE_FILE` from `.env` — use explicit `-f` flags: `podman compose -f docker-compose.yml -f docker-compose.dind.yml up -d`.
- **GH-45 breaking changes**: (1) The DinD port range mapping (`ports: "9200-9300:9100-9200"`) was removed from **`docker-compose.dind.yml`** — editor and terminal sessions are now routed through the shared-port proxy at `/s/<path-token>/...` on the dashboard listener port (no direct host-port access). Existing deployments that relied on direct `localhost:9100–9200` access to editor/terminal sessions must use the proxy URLs returned by the API instead. (2) **`container_name`** was removed from both the `takuto` and `dind` services in Compose so that multiple worktrees/Compose stacks can coexist without name collisions. Scripts or monitoring referencing `takuto` or `takuto-dind` by container name must be updated to use `docker compose exec takuto` / `docker compose exec dind` or the dynamic container names assigned by Compose.
- **Workflow isolation**: when DinD is available, each workflow’s **install**, **pre_install**, **mise install**, and **agent steps** (Claude/Cursor sessions) run in **ephemeral Docker containers** (`docker run --rm`) instead of directly on the Takuto host. Each worker container gets its own network namespace (no port conflicts between concurrent workflows) and a `--cap-add=NET_ADMIN` for egress rules. The worker image is set via **`[general] worker_image`**, auto-detected from the running Takuto container (`docker inspect $HOSTNAME` — the container ID), or from the **`TAKUTO_REGISTRY_IMAGE`** env var (baked into the Docker image at build time as `ghcr.io/takuto-team/takuto-core:<version>`); final fallback is **`takuto:latest`**. The entrypoint **auto-pulls** the worker image into DinD when it is not found locally (using `TAKUTO_REGISTRY_IMAGE`), then tags it as `takuto:latest`. For local development, **`make load-worker`** (`podman save | docker load`) loads the locally-built image instead. Auth volumes are mounted into DinD at **`/shared-auth/*`** and bind-mounted into workers at the standard home paths. A **`playwright-cache`** volume maps to **`/shared-auth/playwright-cache`** (DinD) and **`~/.cache/ms-playwright`** in workers so Playwright uses the **project’s** browser revision from **`npm`/`playwright install`**, not a mismatched Chromium baked into the image (which caused visual snapshot drift). Workers default **`TZ=UTC`**, **`LANG`/`LC_ALL=C.UTF-8`**; the host may pass through **`CI`**, **`TZ`**, **`LANG`**, **`LC_ALL`**, **`PLAYWRIGHT_BROWSERS_PATH`** when set. The worker entrypoint (**`worker-entrypoint.sh`**) applies egress rules, sources **`/etc/takuto/env`**, then `runuser`s as **`takuto`**. Direct argv commands (**`claude`**, **`mise`**, Cursor **`agent`**) bypass that entrypoint (`--entrypoint ""`) but use **`--user takuto:takuto`** so bind-mounted worktree files stay owned by the same uid/gid as the Takuto process (avoids **`git worktree remove`** **permission denied** after isolated runs). Shell **`worktree_init_commands`** still use **`worker-entrypoint.sh`** (root applies egress, then **`runuser`**). Worker containers are force-removed on **cancel**, **stop**, **mark-done**, **delete**, and **stop_all_workflows** via **`ContainerRunner::cleanup_for_ticket`**. When DinD is unavailable (no `DOCKER_HOST` or `docker info` fails), the engine logs an error and **any workflow that reaches the agent/install phase will immediately fail** — DinD is required; there is no local-execution fallback.
- **Per-issue filesystem isolation**: worker containers, editor containers, and run-command containers receive **targeted bind mounts** instead of the broad `/workspace:/workspace` volume. Each container sees only three paths: (1) its own worktree directory (`/workspace/worktrees/{slug}` — read-write), (2) the shared `.git` directory (`/workspace/.git` — needed for git operations like commit, push, status), and (3) the shared `.takuto` directory (`/workspace/.takuto` — read-only; contains `.npmrc` for npm registry auth). This prevents a compromised agent in one issue’s container from reading or modifying files belonging to another issue’s worktree. The isolation is implemented via `build_volume_args()` in `container/runner.rs` which conditionally replaces the `/workspace:/workspace` mount. `ContainerRunner.with_isolate_workspace()` enables isolation for worker containers; `start_editor()` (in `container/editor.rs`) and `start_run_command()` (in `container/run_command.rs`) accept an `isolate_workspace: bool` parameter. Auth volumes (`/shared-auth/*`) and config (`/etc/takuto:ro`) remain shared across all containers. The `bootstrap_new_workflow` (in `workflow/engine/bootstrap.rs`) and `run_workflow_def_steps` (in `workflow/engine/step_runner.rs`) functions enable isolation by default for all new workflows.
- **Figma CLI (`fcli`)**: installed at image build time to **`/usr/local/bin/fcli`** (amd64 uses the GitHub prebuilt tarball; arm64 has no prebuilt, so `cargo install --git` from source via a temporary rustup toolchain that is removed after build). Available to every container using the takuto image (main, workers, editor, terminal). Auth: set **`FIGMA_ACCESS_TOKEN`** in **`takuto.env`** (it's in **`PASSTHROUGH_ENV`**), or run **`fcli auth login`** once from the dashboard terminal — credentials are stored in **`~/.config/fcli/auth.json`** which is mounted from the shared **`fcli-auth`** volume, so they persist across restarts and are visible to all containers.
- **Lokalise CLI v2 (`lokalise2`)**: installed at image build time to **`/usr/local/bin/lokalise2`** from the GitHub prebuilt tarball for the current architecture (both amd64 and arm64 have Linux builds). Available to every container using the takuto image. Auth: pass **`--token`** on the CLI, or put the token in **`config.yml`** (`--config` flag), or export **`LOKALISE_API_TOKEN`** in **`takuto.env`** and wrap invocations with `--token "$LOKALISE_API_TOKEN"` (the var is in **`PASSTHROUGH_ENV`**).
- **`takuto github-app-token`**: generates a GitHub App installation token from the `[github]` config and prints it to stdout. Used by the setup script's repository clone step (Step 4) when a GitHub App is configured and `gh` is not authenticated interactively. Exits non-zero on error.
- **`takuto preflight`**: Phase 0 (04_architecture.md §1) — the subcommand now **always exits 0** under normal mode. Every former hard-error (missing `gh` auth, missing provider auth) becomes a structured `SystemStatus` warning served from **`GET /api/onboarding/status`**. The legacy **`TAKUTO_PREFLIGHT_ERROR`** env var is still populated by `docker/entrypoint.sh` as a one-release back-compat fallback (DB-unavailable case only). Pass **`--strict`** to flip back to a hard-fail exit code for CI pipelines that want to gate on a clean boot. GitHub auth check is **skipped entirely** when **`[github] app_id`** is non-zero (GitHub App is configured — tokens are generated at runtime, no `gh auth` needed); Atlassian CLI (acli) auth is a **soft-fail** (always was) and now surfaces as a `severity = "warning"` entry in `system_status.warnings`. The deprecated `docker_hooks::preflight()` Rust function is retained for one release but internal callers must use `collect_system_status()`. Cursor: skips **`agent status`** when **`CURSOR_API_KEY`** is set or when **`cli-config.json`** under **`CURSOR_CONFIG_DIR`** looks authenticated; otherwise **`agent status`** with timeout and process-group kill. Compose sets **`CURSOR_CONFIG_DIR=/home/takuto/.cursor`** to align with the **`cursor-auth`** volume.
- **`GET /api/onboarding/status`** (public, no auth): returns the boot-time `SystemStatus` snapshot — `{ config_toml_ok, github: { mode, app_configured, app_id, app_name }, provider: { selected, deployment_default_credential_present, headless_capable, custom_base_url }, ticketing: { system, acli_ok }, per_user_required, warnings: [{ code, severity, message }] }`. Captured once at server startup from `docker_hooks::collect_system_status(&config)` and stored in `AppState::system_status`. **`GET /api/auth/status`** mirrors three derived fields from this snapshot: `provider_selected`, `github_mode`, `degraded` (= "any critical warning present"). See `tmp/multi-agents/04_architecture.md §1.2`–`§1.3`.
- **`PUT /api/config/agent`** (admin-only): patch endpoint for the `[agent]` section that the `PUT /api/config` allowlist intentionally does not cover. Body shape is `{ provider?: string, available_providers?: string[], providers?: { claude?, cursor?, codex?, opencode? }, share_conversation_across_steps?: bool, step_timeout_secs?: u64 (>= 1), improve_timeout_secs?: u64, max_repeated_output_lines?: u32 (0 = off) }` with `deny_unknown_fields` at every level (`config.validate()` enforces the timeout floors → **400**). Each provider sub-table accepts `model`, `base_url`, `extra_args`, `allow_shared_default` (Cursor swaps `base_url` for `cli` per amendment A1; Codex adds `provider_name`; OpenCode adds `context_limit` / `output_limit`, sent as a JSON number to set, `null` to clear, or omitted to leave unchanged — validated `> 0`). `extra_args` is validated against the `DENIED_EXTRA_ARG_FLAGS` deny-list (`--dangerously-skip-permissions`, `--output-format`, `--resume`, `--print`, `--verbose`, `-p`) — denied → 400 with stable code `extra_args_denied`. On success the patched config is persisted via `ConfigWriter` (same atomic temp+rename path as `PUT /api/config`) and the response shape mirrors `UpdateConfigResponse`. When the active `provider` changes, the engine broadcasts a `provider_changed` WebSocket event carrying `{ event_type, provider_from, provider_to, affected_users }` (the latter is empty in Phase 1; Phase 2 fills it from per-user credentials).
- **`AiAgentProvider` enum (four v1 variants)**: `claude`, `cursor`, `codex`, `opencode`. **All four have wired runtime adapters as of Phase 4.** The workflow driver dispatches to `ClaudeSession`, `CursorSession`, `CodexSession`, or `OpenCodeSession` via a `match` on the active provider; `is_runtime_implemented(self)` returns true for any of them. Codex spawns `codex exec --json …` (JSONL events: `thread.started`, `item.completed{agent_message,text}`, `turn.completed`, `turn.failed`) and reads `OPENAI_API_KEY` from the worker env. OpenCode spawns `opencode run --format json …` (JSONL events: `step_start`, `text{part.text}`, `step_finish{reason:"stop"}`, `error`) — note OpenCode does NOT auto-read env vars; provider auth needs a project `opencode.json` or `~/.local/share/opencode/auth.json`. Per-provider settings live exclusively in the `[agent.providers.<name>]` sub-tables; the legacy flat `cursor_cli` / `cursor_model` / `model` keys (and their `Config::load` migration shim) have been **removed**.

- **LM Studio recipe (arch §7.4, amendment A2)**: LM Studio is anchored under OpenCode in v1. The documented admin setup is `[agent] provider = "opencode"` plus `[agent.providers.opencode] base_url` set to `http://lm-studio:1234/v1` (sidecar) or `http://host.docker.internal:1234/v1` (LM Studio on the host Mac), and `model = "self_hosted/<model-id-as-reported-by-LM-Studio>"` (the `self_hosted/` prefix is the providerId in the synthesised `opencode.json`; passing `lmstudio/<model>` surfaces as `ProviderModelNotFoundError`). Also set `allow_shared_default = true` when users aren't expected to paste a per-user bearer — otherwise the worker-secrets bundle build fails and opencode starts without its `opencode.json`. The "API key" per-user field accepts any non-empty dummy string (LM Studio's OpenAI-compat server doesn't validate it). Admins who want LM Studio behind the Codex CLI can still use `[agent.providers.codex] provider_name = "lmstudio"` + `base_url` as a power-user setup, but OpenCode is the documented path. Full troubleshooting checklist (incl. Docker Desktop gVisor networking edge case) in **`docs/troubleshooting-self-hosted-models.md`**.
- **Model resolution — `effective_claude_model()`**: every caller that needs the Claude `--model` value MUST go through `AgentConfig::effective_claude_model() -> Option<&str>` (in `crates/takuto-core/src/config/agent.rs`), which reads `[agent.providers.claude].model` and returns `None` when unset. Returns `Option` (not a hardcoded fallback) because Claude is happy to run without `--model`, picking its own current default — which is exactly what proxies that don't recognise older models (pantheon, etc.) need. Mirrors the `effective_cursor_cli` / `effective_cursor_model` accessors. Wired at the two known consumers: `workflow/engine/driver.rs::drive_workflow_def` and `routes/tickets.rs::run_description_session`.
- **SQLite migration v6 (Phase 2a foundation)** — 04_architecture.md §3.1: four new tables (`user_provider_credentials`, `user_github_credentials`, `credential_audit`, `onboarding_state`). All credential tables use **envelope encryption**: every row stores `ciphertext`, `nonce`, `wrapped_dek`, `wnonce` — the per-row DEK is itself sealed with the deployment master key (XChaCha20-Poly1305, 24-byte nonces, 16-byte AEAD tag). Schema is additive (no destructive changes to v1–v5). FKs on `user_id` cascade so `DELETE FROM users` wipes every credential row. Row-shape structs live in `crates/takuto-core/src/db/{provider_credentials,github_credentials,credential_audit,onboarding}.rs`; Phase 2b adds the CRUD helpers + REST endpoints.
- **Deployment master key (Phase 2a)** — 04_architecture.md §3.2: resolved by `auth::master_key::load_or_init_master_key(data_dir, allow_auto_generate)` in resolution order (env var `TAKUTO_SECRET_KEY` → `${data_dir}/secret.key` → auto-generate). Auto-generate is gated by `[general] allow_auto_generate_secret_key` (default `true`; set `false` in hardened deployments to require operator-provisioned keys). When resolution fails or the keyfile permissions are not `0600`, `SystemStatus` surfaces a critical warning (`master_key_unavailable` or `secret_key_world_readable`) and the dashboard renders degraded copy. `MasterKey` is zeroized on drop and its `Debug` impl redacts the bytes.
- **`takuto keys reset` CLI (Phase 2a, A5)** — destructive recovery hatch. Requires `--yes-i-am-sure`. Refuses to run while any workflow is in flight (non-terminal, non-paused) per `${data_dir}/workspaces/*/workflow_snapshot.json` — the scan helper `takuto_core::workflow::snapshot::scan_in_flight_workflow_keys` is unit-tested. On success: clears every row in the four Phase 2a tables and regenerates `${data_dir}/secret.key`. When `TAKUTO_SECRET_KEY` is set, only the DB rows are wiped (the operator rotates the env var manually). v1 ships reset-only — there is no rotation CLI (rotation is forward-compatible because `wrapped_dek` is per-row).
- **Per-user credential endpoints (Phase 2b.1, 04_architecture.md §3 + §4)** — `GET /api/users/me/credentials`, `POST/DELETE /api/users/me/credentials/{provider}`, `POST/DELETE /api/users/me/github-pat`, `PATCH /api/users/me/github`, and admin-only `GET /api/admin/users/{id}/github-status`. AI-provider endpoints seal the pasted `api_key` via `auth::seal::seal(master_key, …)` and upsert into `user_provider_credentials`. The GitHub PAT endpoint validates the token against `auth::pat_validation::validate_pat` first (classic `repo` OR fine-grained `contents:write + pull_requests:write + issues:read`, plus per-org SSO check), then seals + upserts into `user_github_credentials`. Every write is co-committed with a `credential_audit` row inside the same SQLite transaction; failures produce a `validation_failed`/`error` audit row before the 400 response. **Wire-vs-column rename**: the JSON body uses `attribute_commits` per arch doc A3 ("attribute commits to me" — git author/committer env vars, NOT GPG/SSH signing) but the SQLite column is `sign_commits`; the boundary is bridged exactly once via `#[serde(rename = "attribute_commits")]` in `routes/credentials.rs`. Write endpoints return `503 master_key_unavailable` in degraded mode (no master key resolved at boot). `GET /api/users/me/credentials` never includes ciphertext, nonce, wrapped_dek, wnonce, or raw tokens — a snapshot test (`get_my_credentials_never_returns_sealed_bytes_or_tokens`) enforces this.
- **Injectable `gh` shim (Phase 2b.1)** — `takuto_core::auth::GhClient` trait + `RealGhClient` (shells out via `tokio::task::spawn_blocking` to `gh api -i user` / `gh api orgs/<org>` with `-H "Authorization: token <pat>"`) + `SharedGhClient` alias. AppState carries an `Arc<dyn GhClient>` so tests inject a `MockGhClient` and never hit github.com. `pat_validation::validate_pat(&dyn GhClient, &str, &[org])` is pure logic with 11 unit tests covering classic / fine-grained scope sets, 401/403 → `invalid_pat`, missing-scope → `insufficient_scopes`, SSO-required 403 + `X-GitHub-SSO` header → `sso_authorization_required` (response body carries `org` + `org_sso_url`), and transport-error distinct from invalid-pat.
- **Auth-status + onboarding-status Phase 2b.1 extensions** — `GET /api/auth/status` gains `provider_credential_present: bool` (mirrored from a per-user `find_active` query against the deployment-wide active provider; `false` for unauthenticated callers). `GET /api/onboarding/status` flattens the boot-time `SystemStatus` plus an optional `user_onboarding` summary (the user's `onboarding_state` row); `step_4_credentials` auto-flips to `"completed"` when the user has at least one active provider credential, saving the wizard click after a paste.
- **`GitAuthResolver` (Phase 2b.2, 04_architecture.md §4)** — picks an App installation token vs the user's PAT per `GitAction` per the §4.2 decision matrix (7 actions × 4 modes = 28 cells; every cell unit-tested). Lives in `crates/takuto-core/src/github/auth_resolver.rs`. The `GitToken` it returns wraps the bearer in a `SecretToken` (redacted `Debug`) plus `TokenSource`, `author_name`, `author_email`, and a `credential_row_id` slot for Phase 2b.3's `auth_pin`. `mode_for_user` resolves to `AppOnly | AppPlusPat | PatOnly | Missing` and `Display`'s to the wire string already in `GET /api/auth/status::github_mode`. SSO re-validation lives on the resolver too (`revalidate_sso(user, gh, orgs)`); it short-circuits to `Ok` when the user has no PAT (App-only flows aren't SSO-gated). Audit log emits a `kind = github_pat, event = used` row on first use within a ~60-second debounce window; the row is co-written with `touch_last_validated`. AppState carries `Option<Arc<GitAuthResolver>>` — `Some` whenever `db` is `Some`, `None` only in test fixtures that skip the DB. `do_clone` (the repository-clone path) prefers the resolver when both `user_id` and the resolver are present, falling back to the legacy `actions.get_gh_installation_token` chain otherwise.
- **Auth-pin + worker secrets bundle (Phase 2b.3, 04_architecture.md §6 + §7.2)** — closes Phase 2. The workflow's `PersistedWorkflowRecord.auth_pin: Option<AuthPin>` is **`#[serde(default)]`** so pre-v6 snapshots without the field deserialize as `None` (back-compat per blind spot B.7). `AuthPin` carries `provider`, `provider_credential_row_id`, `github_mode`, `github_credential_row_id`, `started_at`. The pin is captured once at the first agent step (in `driver::bootstrap_new_workflow` via `ensure_workflow_auth_pin`) and survives admin provider switches mid-flight — the running workflow keeps the credential row it was pinned to even after the row's `inactive` flag flips.
- **`WorkerSecretsBundle` (Phase 2b.3)** — `crates/takuto-core/src/auth/bundle/assembler.rs::build(config, db, resolver, pin, user_id)`. Unseals the user's pinned provider credential via `auth::seal::open` + the master key, writes each secret into a `tempfile::TempDir` (mode `0700` parent, `0400` files via `OpenOptions::mode`). Same flow for the GitHub token via `GitAuthResolver::token_for(GitAction::Push, …)`. Non-secret env vars (`TAKUTO_AUTH_BUNDLE=1`, `ANTHROPIC_BASE_URL`, `OPENAI_BASE_URL`, `GIT_AUTHOR_*`, `GIT_COMMITTER_*`) flow as `-e` strings; **token bytes are NEVER `-e`-passed** — `docker inspect <container>` cannot leak them. RAII: the bundle's `TempDir` drop `rm -rf`s the secrets directory. **OpenCode self-hosted (spec 2026-05-27)**: for `provider = "opencode"`, the bundle skips the `/run/takuto-secrets/opencode` secret-file path entirely and emits a separate `opencode_config_dir` (an `opencode-config/` subdir under the same `TempDir`) holding `opencode.json` with the user's bearer embedded in `options.apiKey`. The container runner bind-mounts this dir RO at `/home/takuto/.config/opencode`. There is no `OPENCODE_PROVIDER_BASE_URL` env var; the OpenCode CLI ignores env-var base URLs and reads `opencode.json` instead.
- **`ContainerRunner::with_secrets_bundle` (Phase 2b.3)** — opt-in builder that bind-mounts `bundle.host_dir():/run/takuto-secrets:ro`, sets `TAKUTO_AUTH_BUNDLE=1`, and **suppresses the legacy `PASSTHROUGH_ENV` forwarding for `CLAUDE_CODE_OAUTH_TOKEN` / `ANTHROPIC_BASE_URL` / `CURSOR_API_KEY` / `GH_TOKEN`** so the ambient host env doesn't leak into the container via `docker inspect`. Non-secret passthrough (`CI`, `TZ`, `LANG`, `LC_ALL`, `FIGMA_*`, `LOKALISE_*`, `PLAYWRIGHT_BROWSERS_PATH`) stays. When the bundle is absent (workflow `user_id = None` or `git_auth_resolver = None`), the runner falls back to the legacy `PASSTHROUGH_ENV` path — a `tracing::debug!` records that the bundle path was skipped and why.
- **`worker-entrypoint.sh` (Phase 2b.3)** — when `TAKUTO_AUTH_BUNDLE=1` and `/run/takuto-secrets/` exists, the entrypoint sources each present file into the matching env var (`claude` → `CLAUDE_CODE_OAUTH_TOKEN`, `cursor` → `CURSOR_API_KEY`, `codex` → `OPENAI_API_KEY`, `opencode` → `ANTHROPIC_API_KEY`, `gh` → `GH_TOKEN`), then `rm -f`'s the on-disk copy to shrink the blast radius if the container is later compromised. When the discriminator is unset, the entrypoint uses the legacy `gh-app-token` file path (single-tenant / poller flows). **Only invocations that go through the entrypoint** (`wrap_shell_command`, `start_editor`, `start_run_command`) take this path — direct agent invocations (`claude`, `cursor`, `codex`, `opencode`) bypass the entrypoint via `--entrypoint ""` and rely on the [`BUNDLE_SOURCING_SH`] splice below.
- **`BUNDLE_SOURCING_SH` / `wrap_command` (Phase 2b.3, task #36)** — `crates/takuto-core/src/container/runner.rs::BUNDLE_SOURCING_SH` is a `pub(crate)` shell-string constant mirroring `worker-entrypoint.sh`'s `/run/takuto-secrets/*` → env-var block exactly (same five mappings, same `rm -f` step, same `TAKUTO_AUTH_BUNDLE=1` self-gate). `ContainerRunner::wrap_command` (the path agent invocations take — claude/cursor/codex/opencode, which run with `--entrypoint ""`) splices this snippet into its `sh -c` payload BEFORE the `exec` whenever `self.has_secrets_bundle()`. Without this splice the bundle's tmpfs files mount correctly but never get sourced, so the agent CLI sees no token and emits "Not logged in" — that was the actual Phase 2b.3 outage symptom. A drift-detection test (`bundle_sourcing_matches_worker_entrypoint_shell_script`) reads `docker/worker-entrypoint.sh` at test time and fails if the entrypoint adds a new provider mapping without updating `BUNDLE_SOURCING_SH` in lockstep. The same `build_sh_payload` also splices **`GIT_CRED_HELPER_SNIPPET`** before `exec` (both bundle and no-bundle branches): a `git config --global credential.https://github.com.helper` that mirrors `github_app::git_credential_helper_script` (GH_TOKEN-first, App-token-file fallback). Workers don't inherit the main container's `~/.gitconfig`, so without this the agent's `git push` failed with `could not read Username for 'https://github.com'` and weak models looped on it.
- **`.claude.json` shadowing (Phase 2b.3)** — `wrap_command` still restores `~/.claude.json` from `~/.claude/backups/` (the pre-Phase-2b.3 OAuth-state-restore path for `user_id = None` workflows). When `TAKUTO_AUTH_BUNDLE=1` is set, Claude Code uses `CLAUDE_CODE_OAUTH_TOKEN` from the bundle's tmpfs file; the restored `.claude.json` is still on disk but the env var takes precedence, so the on-disk state is effectively shadowed. Single-tenant / poller flows (no `user_id`) keep working unchanged.
- **`WorkerSecretsBundle` host-path translation for DinD (task #43, Phase 2c)**: takuto and the DinD sidecar share VOLUMES, not `/tmp`. Earlier phases created the bundle's `TempDir` under the process tempdir (`tempfile::TempDir::new()` → `/tmp/...`); when takuto told DinD `docker run -v /tmp/.tmpXYZ:/run/takuto-secrets:ro`, DinD resolved the source in **its own filesystem** and bind-mounted an empty (or non-existent) directory into the editor / worker. The bundle now lives under `<data_dir>/runtime/secrets/<random>` (the shared `takuto-data` volume) and `container/runner.rs::translate_path_for_dind()` swaps the takuto-side prefix (`/home/takuto/.takuto`) for the DinD-side prefix (`/shared-auth/takuto-data`, configurable via `TAKUTO_DIND_DATA_PREFIX`) when constructing the `-v` flag. Translation is gated on `DOCKER_HOST=tcp://...` (a remote daemon); in local-Docker dev (`DOCKER_HOST` unset / unix socket) takuto and the daemon share the filesystem and the helper is a no-op. The constant `auth::bundle::SECRETS_DIR_REL = "runtime/secrets"` is the single source of truth for the bundle location relative to `data_dir`; if it changes, the path-translation logic must update in lockstep. A startup sweep `auth::bundle::cleanup_orphan_secrets(data_dir)` runs in the CLI's `run_server` early in boot to remove any bundle dirs left by a crashed prior run (RAII can't fire when takuto dies mid-flight).
- **`WorkerSecretsBundle` lifetime in detached containers (task #42, Phase 2c)**: `start_editor` and `start_run_command` spawn DETACHED containers (`docker run -d …`) and return before the agent finishes. The bundle's `TempDir` is RAII-owned by the bundle struct; without a long-lived owner the route-handler stack scope drops the `Arc` immediately after returning, the TempDir `rm -rf`s itself, and the still-running container is left bind-mounted onto an empty host directory. To prevent this, `AppState` carries two maps — `editor_bundles: HashMap<TicketKey, Arc<WorkerSecretsBundle>>` and `run_command_bundles: HashMap<(TicketKey, usize), Arc<WorkerSecretsBundle>>` — that take a clone of the bundle's `Arc` before the route returns. `close_editor` / `stop_run_command` remove the entry, and the workflow teardown paths (`stop_workflow`, `delete_workflow`/`cleanup_run_commands`, `mark_done`) also wipe every entry for the ticket. The worker path (`drive_workflow_def` → `wrap_command`) is **not** affected because the agent step's `ContainerRunner` holds the bundle Arc for the full foreground duration of `docker run` — no detach, no lifetime gap. Observability: `BUNDLE_SOURCING_SH` (and the mirrored `worker-entrypoint.sh`) emit a `[takuto-bundle]` stderr breadcrumb when `TAKUTO_AUTH_BUNDLE=1` is set but `/run/takuto-secrets/` is empty, so future regressions of the lifetime bug surface in the workflow / editor terminal instead of going silent.
- **Claude `kind=cli_state` (task #39 + #41, Phase 2c)**: a second credential `kind` for Claude only that stores the user's `~/.claude.json` blob (the `oauthAccount` block Claude Code requires for "logged in" on team / Pro plans). `UNIQUE(user_id, provider, kind)` lets a user have BOTH `api_key` and `cli_state` rows simultaneously. The bundle writes them to two independent tmpfs files (`/run/takuto-secrets/claude` for the bearer, `/run/takuto-secrets/claude_session.json` for the session JSON, mode 0400). `BUNDLE_SOURCING_SH` and `worker-entrypoint.sh` **shallow-merge** the session blob onto `$HOME/.claude.json` via `jq -s '.[0] * .[1]'` (existing keys preserved, bundle keys win per-top-level-key) — so the user can paste **only** the `oauthAccount` subtree without wiping `hasCompletedOnboarding`, `userID`, `tipsHistory`, etc. that the legacy backups-restore step had placed there. Plain `cp` is kept as a fallback for the no-jq / no-existing-file case. jq is in the takuto image (Dockerfile line 62). Validator (`POST /api/users/me/credentials/claude` with `{"kind":"cli_state","claude_session_json":"…"}`) requires the JSON to parse AND contain `oauthAccount.{accountUuid, emailAddress, organizationUuid}` — rejected with structured `claude_session_invalid` otherwise. Size cap is **1 MiB** (task #41 raised from 64 KiB to accommodate accumulated `tipsHistory` / `cachedGrowthBookFeatures` / `numStartups` state in real `.claude.json` files); error code `claude_session_json_too_long` carries a `message` + `max_bytes` hint pointing at the most common cause (file path or full dump). Non-Claude providers reject `cli_state` with `kind_not_supported_for_provider`. `DELETE /api/users/me/credentials/{provider}` accepts an optional `?kind=api_key|cli_state` query so the UI can wipe one slot without touching the other; omitting the query wipes both (back-compat). `GET /api/users/me/credentials` returns `provider: { provider, api_key?, cli_state? }` — old single-kind clients can still read `provider.provider` for the active-provider name and switch to `provider.api_key.kind` for the per-slot fields. `WorkerSecretsBundle::claude_session_file: Option<PathBuf>` carries the host-side path; the drift-detection test in `container::tests` asserts the `claude_session.json` file path AND the `jq -s '.[0] * .[1]'` invocation are present in both the Rust constant and the shell script (so a regression to plain-cp gets caught at test time).
- **Engine wiring (Phase 2b.3)** — `WorkflowEngine::with_git_auth_resolver(resolver)` is a builder that stores the resolver on the engine **and** propagates it to the three service structs that spawn driver tasks (`WorkflowDefinitionManager`, `WorkflowPersistence`, `WorkflowTransitions`). Each `tokio::spawn(drive_workflow_def(...))` site clones the resolver into the spawned task, so a restored workflow (snapshot reload), a resumed workflow (after pause), and a freshly-started workflow all take the same Phase 2b.3 path. `WorkflowLifecycle` does not spawn drivers and so does not carry the resolver.
- **`auth::bundle::build_for_endpoint` (Phase 2b.3.x)** — endpoint-side variant of `build` used by improve/prompt/editor/run-command. No caller-supplied pin: it synthesizes an ephemeral one from the live config and returns the same `WorkerSecretsBundle`. Same failure modes (`master_key_unavailable`, `provider_credential_missing`). Wired at four sites: `POST /api/tickets/{key}/improve` and `POST /api/tickets/{key}/prompt` (via `run_description_session`), `POST /api/workflows/{id}/editor` (`open_editor`), and `POST /api/workflows/{id}/run-command/start` (`start_run_command`). All four sites attach via `ContainerRunner::with_secrets_bundle` for the description-session path, or via `apply_secrets_bundle_to_args` for the editor/run-command paths (which build docker argv directly).
- **`GitAuthResolver::revalidate_pat_for_workflow(user_id, gh, orgs)` (Phase 2b.3.x)** — best-effort PAT revalidation invoked at workflow restore (`persistence.rs`) **and** Paused→Resumed transition (`transitions.rs`) via `tokio::spawn`, so it never blocks the driver respawn. Skips silently when the user has no PAT row (App-only flows). On failure, writes a `credential_audit` row (event=`validation_failed`, mapped `error_code`) and the engine broadcasts a new `WorkflowEvent { event_type: "auth_warning", ticket_key, user_id, auth_warning_code, auth_warning_message }` on the existing event channel — clients pick this up over the same `/ws` subscription as state changes. The `auth_warning_code` is a stable identifier (`sso_authorization_required`, `master_key_unavailable`, `resolver_internal`, …) produced by `auth_resolver::auth_warning_payload`. Wired through `WorkflowEngine::with_gh_client(Arc<dyn GhClient>)` (CLI installs `RealGhClient`; tests inject a mock). Without a `gh_client` set, revalidation is silently skipped so legacy tests keep passing.
- **`takuto docker-hooks build|startup`**: runs `build_commands` or `compose_up_commands` from config as **`bash -c`** in `git.repo_path` (used by Dockerfile `RUN` and entrypoint; **`sh`** on Debian is often dash and lacks `pipefail`). Hook children get **`HOME`**, **`TAKUTO_HOME`** (compose: `/home/takuto`), and **`CURSOR_CONFIG_DIR`** so writes to **`~/.claude`** / **`~/.cursor`** hit named volumes, not **`/.claude`** when **`HOME`** is unset.

## Tool layout and extensibility

**Principle:** *Baked = required for advertised features. Provisioning = admin preferences. Removed = specialized use cases.*

Takuto ships a three-tier extension model for adding tools to the workflow environment without forking the upstream image. When you (or an AI agent) consider adding or removing a tool, apply this rule of thumb:

| Question | Tier |
|---|---|
| Is this needed for an advertised Takuto feature? | **Bake** in the Dockerfile |
| Is it an opt-in convenience that some admins want? | **Provisioning default** in `config.toml.example` |
| Is it specialized (one user in a hundred)? | **Don't add** — document the custom-image pattern |

### Mechanism 1 — Baked (Dockerfile)

The `Dockerfile` builds **two image targets** since the 2026-05-21 audit (§3.5 image hardening):

- **`runtime-base`** = `takuto:slim` — the foundation layer. Includes the takuto binary, all CLI bakes (claude / codex / opencode / cursor / gh / acli / mise / openvscode-server / ttyd / docker.io), Node.js, Playwright Chromium libs, the takuto user, sudoers, and the entrypoint. Excludes the Rust toolchain and `build-essential` / `autoconf` / `libssl-dev` / `libyaml-dev`. Build with `docker build --target runtime-base -t takuto:slim .`.
- **`runtime-build-tools`** = `takuto:full` — `FROM runtime-base` + the C toolchain (autoconf, bison, build-essential, lib*-dev) for `mise install` source builds + the Rust toolchain (`rustup`, `cargo`, `rustfmt`, `clippy`, stable). This is the **default** target — plain `docker build .` and `docker compose build` produce `takuto:full`.

**Supply-chain hardening** (audit §3.5 Phase 3, commit `2dc36ed` and predecessors):

- Every `FROM` is pinned by `@sha256:` digest (Renovate-managed, refreshed weekly). Bases: `rust:1.88-bookworm`, `node:23-bookworm-slim`, `debian:bookworm-slim`.
- Every direct download verifies a per-arch sha256 against an ARG-pinned digest: ttyd (`TTYD_SHA256_X86_64` / `_AARCH64`), Node tarball (`NODE_SHA256_X64` / `_ARM64`), openvscode-server (`OPENVSCODE_SHA256_X64` / `_ARM64`), Cursor agent (`CURSOR_AGENT_SHA256_X64` / `_ARM64`).
- The Cursor upstream `curl https://cursor.com/install | bash` is replaced with a pinned `downloads.cursor.com/lab/<version>/linux/<arch>/agent-cli-package.tar.gz` tarball + sha256 verify — bumping the version is a one-line ARG change (`CURSOR_AGENT_VERSION` + the two per-arch SHA ARGs).
- Every npm global pins to a version via ARG: `CLAUDE_CODE_VERSION`, `CODEX_VERSION`, `OPENCODE_AI_VERSION`. No `@latest` anywhere in image build.

Bake set as of task #48 (versions are ARG-pinned in `Dockerfile`; values below are the captured baseline at audit time, not necessarily current):

- **Foundation system layer** — `ca-certificates`, `curl`, `wget`, `gnupg2`, `docker.io`, `git`, `jq`, `iptables`, `iproute2`, `openssh-client`, `python3`, `socat`, Playwright Chromium system deps.
- **Runtimes** — Node.js 23.11.0 (`node`, `npm`); Rust toolchain (`rustup` + `cargo` + `rustfmt` + `clippy`, stable) **only in `takuto:full`**.
- **VCS / ticketing** — `gh` (GitHub CLI, apt), `acli` (Atlassian CLI, apt; amd64 only — falls back to a build-time WARN on arm64), `mise`.
- **AI provider CLIs (all four advertised providers)** — `claude` (npm `@anthropic-ai/claude-code` v2.1.146), `cursor-agent` (pinned tarball from `downloads.cursor.com`, version 2026.05.20-2b5dd59, installed under `/usr/local/share/cursor-agent/versions/<VERSION>/`), `codex` (npm `@openai/codex` v0.132.0), `opencode` (npm `opencode-ai` v1.15.6). Codex + opencode are baked even though their Phase 4 runtime adapters aren't wired yet — the binaries exist so the image is ready the moment the adapter lands.
- **Editor** — `openvscode-server` v1.109.5.
- **Web terminal** — `ttyd` v1.7.7.
- **Process tools** — `sudo` (with the narrow `/usr/bin/bash` sudoers rule for the docker-hook sudo path).

The runtime stage(s) total **11 `RUN` layers** (down from 26 pre-audit) — apt installs are collapsed into one mega RUN, mkdirs + chowns are consolidated, profile.d writes are combined, and executable bits on shell scripts are set via `COPY --chmod=0755` (no separate chmod RUN). The Dockerfile declares `USER takuto` immediately before `ENTRYPOINT` for explicit audit-visible identity, then `USER root` to preserve current behaviour because the entrypoint still needs root for iptables / volume chowns / `[provisioning]` writes (the script setpriv's back to takuto for the server process).

Removed from the bake (task #48), now `[provisioning]` defaults: `fcli`, `lokalise2`, `figma-cli`. Removed entirely: AWS CLI v2 (admin opt-in via custom image OR a commented `[provisioning]` example in `config.toml.example`).

### Mechanism 2 — Provisioning (`[provisioning].install_commands`)

Admin-controlled shell commands that run at takuto startup as **root** in the takuto container, with `TAKUTO_TOOLS_BIN=/opt/takuto-tools/bin` exported. The directory is a Docker NAMED volume (`takuto-tools`) mounted RW into the takuto service and DinD, and RO into every spawned worker / editor / run-command via `container/runner.rs::build_volume_args`. `ENV PATH="/opt/takuto-tools/bin:..."` in the Dockerfile prepends the volume, so a binary dropped there SHADOWS a baked tool of the same name — the admin's pin-a-version lever.

**SHA-gating.** The canonical sha256 of the JSON-encoded (order-preserving) `install_commands` list is recorded at `/opt/takuto-tools/.provisioning-sha`. Subsequent boots compare the live config's SHA against the file; matching SHA → fast-path skip. Edit the list (add, remove, reorder, tweak a command) → SHA changes → install pass runs. `Config::provisioning_sha()` in `config.rs` is the single source of truth.

**Entrypoint integration.** `docker/entrypoint.sh::bootstrap_provisioning` is called in the ROOT section BEFORE `setpriv`, after the chowns and before the egress rules. It shells out to `takuto provisioning sha` and `takuto provisioning commands` (NUL-separated) — the CLI subcommand exists because the entrypoint can't parse TOML directly. Per-command failures log a WARN and continue; the SHA file is ONLY written on full success, so partial-failure boots retry on the next restart.

### Mechanism 3 — Custom Dockerfile (`FROM takuto:latest`)

For system packages (apt-managed libs, daemons), multi-path installs, or anything the `/opt/takuto-tools/bin` single-directory volume can't hold. Documented in `README.md` and `docs/extending-takuto.md`. Admins maintain their own `my-takuto.Dockerfile` + `docker-compose.yml` `build:` block.

### Rules for AI agents modifying the codebase

1. **Don't add new baked tools without a clear advertised-feature link.** If you find yourself wanting to bake a tool, first check whether `[provisioning]` covers the case — most often it does. The bake set increases image size for every deployment forever; the bar is "Takuto literally fails without it".
2. **Don't move a baked agent CLI to provisioning.** `claude`, `cursor-agent`, `codex`, `opencode` are advertised providers — they MUST be baked because workflows fail at spawn if they're missing.
3. **PATH order is part of the design contract.** `/opt/takuto-tools/bin` is FIRST so provisioning shadows bake; `/home/takuto/.npm-global/bin` second so npm globals work; `/home/takuto/.local/share/mise/shims` next so mise wins over baked Node/etc. Reordering these breaks documented admin escape hatches.
4. **Worker spawn paths MUST mount the tools volume.** `build_volume_args` in `container/runner.rs` appends `takuto-tools:/opt/takuto-tools/bin:ro` to every spawn. If you add a new spawn path (a new container kind), make sure it also calls `build_volume_args` OR adds the mount explicitly; otherwise the workflow's `kubectl` / `terraform` / pinned `claude` won't be there.
