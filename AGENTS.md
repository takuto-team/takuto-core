# Maestro — context for AI coding agents

**New agent session:** this file must be the **first** project file you read (before `README.md`, `ARCHITECTURE.md`, or the crate tree). Cursor and Claude project rules enforce that.

This file is the **canonical high-level reference** for what the repository is, how it is structured, and how the main runtime paths work. Human-oriented detail lives in `README.md`, `ARCHITECTURE.md`, and `docs/workflow.md`; prefer those for setup, troubleshooting, and deep diagrams. **Keep this file aligned with the code** when you change behavior or layout (see `.cursor/rules/` and `CLAUDE.md`).

## What this project does

**Maestro** is a Rust application that **polls Jira** for work in **To Do**, then for each ticket runs an **automated pipeline**: assign / transition status, **clone work via a git worktree**, optional **pre_install** / **install**, then configurable **workflow definition steps** (Claude Code or Cursor Agent in **headless** mode — implement, review, lint/tests, **open a PR with `gh`**, or any custom sequence defined in TOML workflow files). The engine **does not** run a separate PR step: when the last agent session exits successfully, the workflow **finalizes** to **`Done`**. An optional PR URL for the dashboard comes from **`.maestro/outcome.toml`** (`pr_url = "…"`) or a stdout line **`MAESTRO_PR_URL: …`** (see **`agent_prompt`** headless suffix). A small **web dashboard** lists workflows, streams terminal output over **WebSocket**, and exposes REST endpoints for control and config.

**Dry mode** (`[general] dry_mode = true` or CLI `--dry-run`) skips real Jira/GitHub side effects while still running local work (worktrees, commands, Claude) through the `ExternalActions` trait (`RealActions` vs `DryRunActions`).

**Ticketing modes** (`[general] ticketing_system` in `config.toml`, default `none`):

- **`jira`** — current behavior: polls Jira for To Do tickets via `acli`, assigns/transitions tickets, full Jira integration. Requires Atlassian CLI auth (`make setup` Step 2). If acli auth fails at startup, falls back to no-ticketing mode with a warning.
- **`github`** — polls GitHub Issues from the repo's git remote via `gh api`; no Atlassian auth required. Issues appear in the dashboard `+` picker. No Jira transitions — `jira_available` is `false` for all workflows. `when: ticketing` steps run; `when: no_ticketing` steps do not.
- **`none`** (default) — no ticketing system. The Jira poller and GitHub poller are both idle. Workflows started manually only (dashboard `+` opens a paste-description modal). `when: no_ticketing` steps run; `when: ticketing` steps do not. Dashboard shows an **amber warning banner** and a one-time **alert dialog** on page load.

The dashboard hides polling controls when `ticketing_system = none`. In Jira and GitHub modes polling controls are visible. When `ticketing_system = none`: the **+** button opens a **1000px-wide paste-description modal** (workflow name + description textarea); the workflow name is slugified into the **`ticket_key`** (which drives the branch and worktree names, e.g. `feat/add-user-login`) while the original name becomes **`ticket_summary`**. When the name field is left empty, the server falls back to **`MANUAL-{timestamp_millis}`**. The user-provided description is stored as **`ticket_description`** and available in prompts via **`{ticket_description}`** or the **`{description}`** alias. When `ticketing_system = github`, the `+` button opens a GitHub Issues picker (calls `GET /api/github/issues`) instead. `GET /api/config` returns **`jira_available: bool`** and **`ticketing_system: string`** (`"jira"`, `"github"`, or `"none"`); `WorkflowSummary` includes **`jira_available`** and **`ticket_description`** per workflow. The `Workflow` struct carries **`jira_available: bool`** (persisted in snapshots with `#[serde(default)]` = `true` for backward compatibility).

## How the binary starts

`crates/maestro-cli/src/main.rs`:

1. Parses CLI (`--config` / `MAESTRO_CONFIG`, default `config.toml`; `--dry-run`).
2. Loads `Config` from file or defaults.
3. Initializes **JSON** logging via `tracing_subscriber` and `EnvFilter` (including `general.log_level`).
4. Builds `Arc<RwLock<Config>>` and `Arc<dyn ExternalActions>` from dry mode.
5. Probes acli authentication (`docker_hooks::check_acli_auth()`) → `jira_available: Arc<AtomicBool>`. When `false`: logs a warning, workflows skip Jira steps, poller is idle.
6. Constructs `WorkflowEngine` with **`[general] max_concurrent_workflows`** (drives an internal semaphore for concurrent **mise** / **install** / **agent** sessions) **and** `jira_available`. Calls **`restore_persisted_workflows`** so workflows saved on the last graceful shutdown are loaded before the poller’s first poll. **All workspaces** are restored at once — workflows from every workspace coexist in the single in-memory `HashMap<String, Workflow>`, tagged by `Workflow.workspace_name`. Snapshots are **per-workspace** at **`{data_dir}/workspaces/{workspace_name}/workflow_snapshot.json`**, where the **persistent data directory** (`data_dir`) is resolved as: **`$MAESTRO_DATA_DIR`** → **`$MAESTRO_HOME/.maestro`** → **`$HOME/.maestro`** → **`{git.repo_path}/.maestro`** (legacy fallback). In Docker, the **`maestro-data`** named volume is mounted at **`/home/maestro/.maestro`** and **`MAESTRO_DATA_DIR`** points there. On startup, `read_all_workspace_snapshots(data_dir)` scans `{data_dir}/workspaces/*/workflow_snapshot.json` and merges all records. A 3-tier migration handles legacy layouts: workspace-specific → in-repo legacy (`{git.repo_path}/.maestro/`) → old global (`{data_dir}/workflow_snapshot.json`). The Jira poller separately enforces **`[general] max_active_workflows`** (see **Jira poller**).
7. Constructs `JiraPoller` with a shared `CancellationToken` and **`Arc<AtomicBool>`** polling pause flag (shared with `AppState`). The flag is **`false`** by default (polling enabled); set **`[general] auto_polling = false`** in **`config.toml`** to start in the same paused state as the dashboard **Pause polling** action, then `build_router`. When `!jira_available`, the poller future is replaced with `std::future::pending()` (never runs).
8. Creates a `ConfigWriter` from the config file path and a `ConfigWatcher` that polls for external edits (5 s interval). Both share a `last_write_epoch_ms` atomic to prevent reload loops from API writes.
9. Runs poller, Axum server, config watcher, snapshot syncer, PR merge poller with graceful shutdown on cancel, and signal handler (`SIGINT` / `SIGTERM` on Unix). On **graceful** shutdown the process cancels the token, **`persist_interrupt_for_restart`** (calls **`write_all_workspace_snapshots`** — groups all in-memory workflows by `workspace_name`, writes each group to `{data_dir}/workspaces/{name}/workflow_snapshot.json` — and cancels their driver tokens **without** Jira unassign / **`Stopped`**), then waits briefly for cleanup. On restore, **terminal** workflows (**Done**, **Stopped**, **Error**) are loaded with **no driver** (idle on the dashboard); **Paused** and in-progress workflows get a driver spawned. The last Claude/Cursor **session ID** (`Workflow.last_session_id`) is persisted in the snapshot; on resume the first re-executed agent step uses **`--resume`** with the prompt **"Resume what you were doing before the session was closed."** so the agent continues its prior conversation. **`POST …/stop`** and **`stop_all_workflows`** still unassign and move tickets back to **To Do** (explicit stop, not container resume) — **only when `jira_available` is true**.

## Workflow model

### State machine

Defined in `crates/maestro-core/src/workflow/state.rs` as `WorkflowState`: `Pending` → `Assigning` → `RetrievingDetails` → `CreatingWorktree` → **`AddressingTicket { pass }`** (`pass` is the outer cycle index, always `1` — the built-in step list runs once; use per-step **`repeat`** for multiple runs) → **`Done`**, `Error { .. }`, `Paused { .. }`, `Stopped`. The **`Reviewing`**, **`CreatingPR`**, **`AddressingPrComments`**, and **`MergingBaseBranch`** variants remain for **deserialization of older persisted snapshots only** — they are no longer actively driven by the engine. The **dashboard and REST list** use **`Workflow::status_display()`**, which shows the live **`current_step_label`** (e.g. **`Implement ticket (cycle 2/3, run 1/1)`**).

Terminal states: `Done`, `Stopped`, `Error`.

### Engine and storage

`WorkflowEngine` (`workflow/engine.rs`) holds:

- `Arc<RwLock<Config>>`
- `Arc<RwLock<HashMap<String, Workflow>>>` — **keys are Jira ticket keys** (e.g. `PROJ-123`), not the workflow UUID
- `Arc<dyn ExternalActions>`
- `broadcast::Sender<WorkflowEvent>` for real-time updates

The **Jira poller** uses `WorkflowEngine::start_workflow` (with **`started_manually: false`**), which inserts the workflow **and** immediately discovers all dep-free (no `depends_on`) dynamic workflow definitions and calls `start_workflow_def` for each. The **dashboard manual flow** (the **+** picker) calls **`WorkflowEngine::add_to_dashboard`** instead: it inserts the workflow in **`Pending`** state with **`driver_started = false`** and does **not** spawn any drivers. For Jira tickets, `add_to_dashboard` still assigns the ticket and transitions it to **In Progress** (best-effort, spawned task). Clicking a workflow-def button on a **Pending** card calls **`POST /api/workflows/{id}/run-workflow/{def}`** → `start_workflow_def` → `drive_workflow_def` — which runs `bootstrap_new_workflow` first (Jira assign/retrieve, worktree creation, mise/install/pre-workflow) before the def's agent steps. **`max_concurrent_workflows`** caps concurrent **mise** / **install** / **agent** sessions via a shared semaphore (permits are taken **after** `wait_if_paused`, so paused workflows do not hold a heavy-work slot). The Jira poller uses **`max_active_workflows`** (or **`max_concurrent_workflows`** when **`max_active_workflows`** is **`0`**) against **`dashboard_workflow_count`**: every workflow still in the map counts as **active** — **Done**, **Paused**, **Stopped**, **Error**, and in-progress rows all count until **Mark as Done** or **Delete** drops the row from the dashboard. **`[general] max_concurrent_manual_workflows`** ( **`0`** = unlimited) caps how many **manual** starts may **occupy a slot** at once (not **Done** / **Stopped** / **Error**), enforced server-side on **`POST /api/workflows/start-manual`** and reflected by disabling the dashboard **+** tile when full.

`Workflow` carries a **`driver_started: bool`** field — `false` when added to dashboard via the **+** picker (no driver running), `true` once `start_workflow_def` spawns a def driver (set in the write lock before `tokio::spawn`). **`WorkflowSummary`** includes **`can_start: bool`** (`Pending` + `!driver_started` — informational; no separate `/start` endpoint exists). Deleting an unstarted Pending workflow is allowed (the `is_active()` guard is bypassed when `!driver_started`); for Jira, delete also unassigns and reverts to **To Do**. Snapshot persistence: `driver_started` is stored in **`PersistedWorkflowRecord`** (`#[serde(default)]` = `true` for backward compatibility); on restore, `Pending` + `!driver_started` workflows are inserted without a driver (like terminal states).

`Workflow` carries ticket metadata, `steps_log`, branch/worktree paths, `pr_url`, **`ticket_url`** (`Option<String>` — the direct URL to the issue in the ticketing system, stored at creation time; `Some(html_url)` for GitHub-sourced workflows, `None` for Jira and manual workflows; the REST API exposes this as `issue_url` in `WorkflowSummary`, falling back to the computed Jira browse URL when `ticket_url` is `None`), **`pr_merged`** (whether the associated PR has been merged on GitHub), `CancellationToken`, `last_session_id` (Claude/Cursor session ID for `--resume` across restarts), **`workspace_name`** (derived from the last path component of `config.git.repo_path` via `workspace_name_from_repo_path()`), and up to **100** recent `terminal_lines` for UI persistence.

**Per-workspace snapshot persistence:** **`workflow/snapshot.rs`** defines the JSON snapshot format for graceful shutdown / restart. Snapshots are **per-workspace** at **`{data_dir}/workspaces/{workspace_name}/workflow_snapshot.json`** (see **How the binary starts** step 6). On shutdown / periodic sync, **`write_all_workspace_snapshots`** groups all in-memory workflows by `workspace_name` and writes each group to its own snapshot file; workspace directories with zero records are cleaned up. On restore, **`read_all_workspace_snapshots`** scans `{data_dir}/workspaces/*/workflow_snapshot.json` and merges all records into the single in-memory map. **All** workflows in the map are snapshotted (including **Stopped**, **Error**, **Paused**) — they remain on the dashboard until explicitly deleted or marked done. On restore, **terminal** states (**Done**, **Stopped**, **Error**) and **unstarted Pending** (`Pending` + `driver_started == false`) are inserted with **no** driver; other in-progress / **Paused** rows find any `Running` workflow def entries and re-spawn `drive_workflow_def(Some(wt))` for each; if the worktree no longer exists, those defs are marked `Error`. `Running` entries in `steps_log` are stripped on restore (they represent interrupted steps that will re-execute). The persisted **`last_session_id`** lets the first re-executed agent step resume the prior Claude/Cursor conversation via `--resume`.

**Multi-workspace model:** All workflows from all workspaces coexist in a single in-memory `HashMap`. Switching workspaces (**`POST /api/workspaces/switch`**) flushes the current snapshot, updates `config.git.repo_path`, and returns — running workflows from the previous workspace **continue executing** in the background. **`GET /api/workflows`** filters to only return workflows matching the current workspace's `workspace_name`. **`GET /api/workflows/counts`** aggregates across **all** workspaces (global counters). The dashboard `SummaryStats` shows global counts while `WorkflowGrid` shows only the current workspace's workflows.

### Dynamic workflow definitions — bootstrap and step sequence

All workflows are driven entirely by **dynamic TOML workflow definitions** (`*.toml` files in the `workflows/` directory). There is no static step list driven from `config.toml`. When a def button is clicked on the dashboard, **`POST /api/workflows/{id}/run-workflow/{def}`** calls `start_workflow_def` → `drive_workflow_def`:

**Bootstrap** (`bootstrap_new_workflow` in `workflow/engine.rs`) — called automatically when the workflow has no worktree yet (Pending card, first run):

1. **Assign** ticket to the **currently authenticated Jira user** (`acli` **`@me`**) and move to **In Progress** (failures may be logged/`[DRY/SKIP]` but workflow can continue). **When `workflow.jira_available` is `false`**: skipped entirely (no step log, no state transition).
2. **Jira details** via `JiraClient` / `get_ticket_details`; populate description, summary, type on `Workflow`. **When `workflow.jira_available` is `false`**: skipped; a `JiraTicket` is constructed from the workflow’s in-memory `ticket_summary` / `ticket_description` (user-pasted via the paste-description modal).
3. **Worktree** from `git::worktree::branch_name_for_ticket` and configured `base_branch`. **`create_worktree`** removes any leftover directory at the target path first so a prior failed teardown cannot block a new run.
4. Optional **`pre_install`** (array of shell commands) then optional **`install`** in the worktree (streaming output), then optional **`pre_workflow`** commands.

Bootstrap returns `(worktree_path, ticket_detail)`. When the workflow **already has a worktree** (e.g. running a second def after the first completed), bootstrap is skipped and the driver goes directly to the agent steps.

**Agent steps** (`run_workflow_def_steps`): run the definition’s `[[steps]]` via `run_agent_step_sequence`.

   **Step types:** Each step is either an **agent step** (has `prompt` and/or `skills`) or a **command step** (has `commands`). The two modes are **mutually exclusive** — validation rejects a step that specifies both `prompt` and `commands`, or neither `prompt` nor `skills` nor `commands`.

   **Agent steps** invoke Claude Code or Cursor Agent. Each step supports an optional **`skills`** array of **`{ name, args }`** objects. For **Claude** (`--bare` mode): each referenced skill’s **`<name>/SKILL.md`** is read from skill search paths (worktree **`.claude/skills/`** first, then **`~/.claude/skills/`**), frontmatter is stripped, `$ARGUMENTS` / `$1` / `$2` … are substituted with positional args, and the concatenated result is passed via **`--system-prompt`**. Claude Code’s `--print` mode does not support native `/skill` invocations, so Maestro handles skill content injection. For **Cursor**: skills work natively, so Maestro builds `/skill-name arg1 arg2` invocation lines and prepends them to the prompt.

   **Command steps** run shell commands directly without invoking an AI agent. The `commands` field is an array of strings; each is executed sequentially via `bash -c` (or `mise exec -- sh -c` when mise is configured) in the worktree directory. If any command exits non-zero, the step fails and the workflow follows the existing error/retry behaviour. Command stdout/stderr is streamed to the dashboard terminal in real-time. The `skills` and `resume_previous` fields are ignored on command steps (they are agent-specific). Command steps respect `[agent] step_timeout_secs` for each individual command and support container isolation when DinD is available.

   All steps interpolate **`{ticket_key}`**, **`{ticket_summary}`**, **`{ticket_description}`** (alias **`{description}`**), **`{ticket_type}`**, **`{acceptance_criteria}`**, **`{ticket_context}`**, **`{base_branch}`**, **`{pr_url}`** — placeholders work in both `prompt` (agent steps) and `commands` (command steps). **`{ticket_context}`** includes a trusted policy preamble and **UNTRUSTED_JIRA** sections (see `build_ticket_context` in `workflow/engine.rs`); linked-issue inclusion follows **`[jira] linked_items_in_prompt`**. For agent steps, headless instructions from **`agent_prompt.rs`** are appended. There is **no** built-in PM / plan-validation pass — add a **custom step** whose **prompt** asks the agent to validate plans or requirements if you want that. Step failure **aborts** the definition run (Error state). For deterministic tasks like linting or testing, use **command steps** instead of agent prompts to avoid consuming AI tokens.

After the worktree is created, `bootstrap_new_workflow` calls **`configure_git_author_from_github`**: when **`[github]`** App credentials are configured, this uses the bot identity (`maestro-bot[bot]` / `{app_id}+maestro-bot[bot]@users.noreply.github.com`) and configures `gh` auth with an installation access token; otherwise it falls back to **`gh api user`** using GitHub’s **`{id}+{login}@users.noreply.github.com`** form so commits match the personal **`gh`** account.

### Dynamic workflow definitions (TOML)

`crates/maestro-core/src/workflow/definitions.rs` implements **user-customizable secondary workflows** loaded from TOML files at runtime. These are dynamic: discovered by scanning a `workflows/` directory for `*.toml` files, with no restart needed.

**TOML schema** — each `*.toml` file:
```toml
name = "Display Name"              # required, shown on UI buttons
depends_on = ["other_workflow"]    # optional, list of filenames (without .toml)

[[steps]]
name = "Step name"
prompt = """
Multi-line prompt...
"""
repeat = 1
when = "always"                    # always | ticketing | no_ticketing

[[steps]]
name = "Run tests"
commands = ["npm test"]            # command step (no AI)
```

Steps support both **agent steps** (has `prompt` and/or `skills`) and **command steps** (has `commands`). Steps are converted to the existing `AgentStepConfig` type for execution.

**Discovery** (`discover_workflows`): scans the configured directory, skips `*.example.toml` files, parses each TOML file, validates dependencies (detects cycles via DFS, marks workflows with missing deps as invalid). Invalid TOML is skipped with a warning (the `DiscoveredWorkflow` struct carries `valid: bool` and `error: Option<String>`).

**Dependency resolution**: `depends_on` uses filenames (without `.toml`). `are_dependencies_met()` checks that all upstream workflows completed successfully. `topological_order()` returns a valid execution order (Kahn's algorithm). Circular dependencies are detected at load time and the involved workflows are marked invalid.

**Runtime state**: Each workflow tracks per-definition run state in `workflow_def_runs: HashMap<String, WorkflowDefRunState>` (states: `Idle`, `Running`, `Completed`, `Error`). The field is persisted in workflow snapshots (`PersistedWorkflowRecord`).

**Engine integration** (`workflow/engine.rs`):
- `start_workflow_def(ticket_key, def_name)`: discovers definitions, validates dependencies are met, sets `Running` state (and `driver_started = true`), spawns async driver. When the workflow has no worktree (Pending, first run), passes `None` as the worktree path so the driver bootstraps first.
- `retry_workflow_def(ticket_key, def_name)`: resets `Error` → `Idle`, then calls `start_workflow_def`
- `drive_workflow_def(worktree_path: Option<PathBuf>)`: async driver. When `worktree_path` is `None`, calls `bootstrap_new_workflow` first; otherwise uses the existing worktree. Then runs the definition's steps via `run_workflow_def_steps()` / `run_agent_step_sequence()`, updates state to `Completed` or `Error`. Cancelled drivers in Stopped or Paused state exit without overwriting state.
- `bootstrap_new_workflow()`: performs Jira assign/retrieve, worktree creation, git author setup, mise install, pre-install, install, and pre-workflow commands. Returns `(worktree_path, ticket_detail)`.
- `start_definitions_watcher(cancel_token)`: background task polling the definitions directory every 5 seconds; broadcasts a `workflow_definitions_changed` WebSocket event when TOML file changes are detected

**Config**: `[general] workflow_definitions_dir` (default `"workflows"`) — path to the directory scanned for `*.toml` definitions, resolved relative to the config file's directory.

### Mark as Done

**`POST /api/workflows/{ticket_key}/mark-done`** (dashboard **Mark as Done**): transitions Jira to **`[jira] done_status`** (default **`Done`**), then **`remove_worktree`**, then best-effort **`delete_local_branch`** (**`git branch -D`**) when **`branch_name`** is set, then **`git worktree prune`**. Partial failure is returned in JSON (**`MarkDoneOutcome`**); the workflow is **removed from the map** only if **both** Jira and worktree removal succeed (branch/prune failures are logged only). A WebSocket **`workflow_removed`** event is sent when the row is dropped.

**`POST /api/workflows/{ticket_key}/delete`** (dashboard **Delete**): allowed when **`WorkflowState::is_active()`** is false (not **running** — includes **Paused**, **Done**, **Stopped**, **Error**). Cancels the workflow token, **`ContainerRunner::cleanup_for_ticket`** (force-remove worker containers), best-effort **`remove_worktree`**, best-effort **`delete_local_branch`**, **`git worktree prune`**, then removes the row from the map — **no Jira transitions**. Syncs **`workflow_snapshot.json`** best-effort; emits **`workflow_removed`**.

Pause/resume/stop apply while a workflow definition is actively running (**`is_active()`**).

Pause/resume: workflow state wraps prior state in `Paused`; `wait_if_paused` blocks the driver until resumed.

Stop (**API** / **`stop_all_workflows`**): cancels the workflow token (child processes stop via `ProcessHandle` / process groups on Unix), then **`ContainerRunner::cleanup_for_ticket`** immediately so isolated worker containers are not left running until the driver exits; Jira cleanup (unassign / To Do) runs in a spawned task as before. Drivers that exit with **`Cancelled`** while the row is **`Stopped`** or already removed do not overwrite **`Stopped`** with **`Error`**. **`acquire_agent_slot`** aborts on cancel so workflows do not block forever on the install/agent semaphore after stop. **Graceful process shutdown** (SIGINT/SIGTERM path in **`maestro-cli`**) uses snapshot persist + cancel so tickets stay **In Progress** for resume.

The **index** dashboard uses a custom **confirm** modal (**`#dashboardConfirmModal`**) for **Stop**, **Delete**, and **Mark as Done** (not `window.confirm`). **Delete** and **Mark as Done** show a dark semi-transparent overlay with a spinner on the card while the request is in flight. **Error** / **Stopped** workflows show two retry buttons: **Retry from last failure** (primary/blue, calls **`resume-from-error`** — re-drives from the failed step, skipping succeeded steps, reusing worktree and session ID; only shown when the worktree exists on disk) and **Retry from 0** (primary/blue, calls **`retry`** — removes the workflow and starts fresh).

File logging: `WorkflowLogWriter` writes under `{repo_path}/logs/<TICKET>.log`.

## AI agent integration (Claude Code or Cursor Agent)

### Provider selection

`[agent] provider` in config: **`claude`** (default) or **`cursor`**. **`[agent] cursor_cli`** sets the Cursor Agent executable (default **`agent`**). **`[claude] model`** is passed to Claude Code when non-empty. **`[agent] cursor_model`** (default **`Auto`**) sets Cursor Agent `--model`; **`Auto`** (any ASCII case) or empty means automatic model selection (passed as `--model Auto`).

### Claude Code

`crates/maestro-core/src/claude/session.rs` spawns **`claude`** with `--dangerously-skip-permissions`, `--print`, `--verbose`, `-p`, `--output-format stream-json`, optional `--resume`, optional `--model`.

### Cursor Agent (headless)

`crates/maestro-core/src/cursor/session.rs` spawns **`cursor_cli`** (e.g. `agent`) with **print** mode, **`--output-format stream-json`**, **`--stream-partial-output`**, **`--trust`**, **`--force`**, **`--approve-mcps`**, **`--sandbox disabled`**, **`--workspace <worktree>`**, optional **`--resume`**, optional **`--model`**. This matches Cursor’s non-interactive / trusted automation flags from their CLI docs.

The **Docker image** installs **Node.js 23+** from **nodejs.org** (not Node 20): the Cursor **`cursor-agent`** launcher runs its bundled **`node`** with **`--use-system-ca`**, which requires **Node ≥ 23.9** on Linux; older Node fails with `bad option: --use-system-ca`. The image copies the **full** Cursor Agent package under **`/usr/local/share/cursor-agent`** and symlinks **`/usr/local/bin/agent`** to that launcher (copying only the script to **`/usr/local/bin`** breaks **`index.js`** resolution).

The image also installs **mise** (apt). **`process::worktree_has_mise_config`** detects **`.mise.toml`**, **`mise.toml`**, **`.tool-versions`**, or **`.config/mise/config.toml`**; the workflow runs **`mise install`** in the worktree, then **`run_shell_command` / `run_shell_command_streaming`** use **`mise exec -- sh -c …`** when that detection matches. **`run_command`** (argv, e.g. **`acli`**) is unchanged.

### Dashboard streaming

Raw stdout lines are turned into short lines for the web UI via **`workflow/stream_humanize.rs`**: **`humanize_agent_stream_line`** dispatches on provider. Cursor stream-json events include **assistant** text (trimmed; empty chunks omitted), **`tool_call` started** (e.g. read/write paths, shell as `$ …`, generic tools with a short **arguments** snippet when present), **`tool_call` completed** for shell with **Done/Failed (exit N)** plus an optional **truncated output tail**, and **result** — so operators still see **live progress** similar to Claude. (Cursor’s docs note **thinking** events are suppressed in print mode; internal chain-of-thought is not shown, but tool use and assistant text are.)

The **index** dashboard (`ui/src/pages/Dashboard.tsx`, React + Vite SPA) renders each workflow card (`ui/src/components/IssueCard.tsx`). After the last card, a **+** tile opens a modal that loads configured-project **To Do** issues via **`GET /api/jira/todo-tickets-manual`**: **Epics excluded**, backlog **`ORDER BY rank ASC`**, ignoring **`[jira] item_types`**. When **`[jira] jql_filter`** is non-empty, it is **`AND`**-combined with that query so operators can mirror a board filter. Choosing a row opens a **second modal** on top: **`GET /api/jira/tickets/{key}/preview`** returns **`description_markdown`** (Jira **ADF** descriptions are converted server-side to Markdown); the UI renders Markdown to HTML via the shared **`MarkdownPreview`** component (`ui/src/components/MarkdownPreview.tsx`) — **marked** + **DOMPurify** with **Mermaid diagram support** (fenced ` ```mermaid ` blocks are rendered as interactive SVG diagrams; invalid syntax shows mermaid’s built-in error message). If the operator switches to another ticket before a preview request completes, stale responses are ignored so the modal does not show the wrong issue’s description. **Close** dismisses only the detail modal; **Add to Dashboard** closes both modals and **`POST`s `/api/workflows/start-manual`** (which now adds the workflow in **Pending** state without starting the driver — the user clicks **Start** on the card to begin). The **+** is hidden when **`[jira] project_keys`** is empty and **disabled** when **`[general] max_concurrent_manual_workflows`** is exceeded (non-zero). The grid keeps a **stable ticket order** across refetches and full re-renders (pause/stop/complete): keys are **not** re-sorted by state; **new** workflows **append** at the end when they first appear in **`GET /api/workflows`**. When **`pr_url`** is non-empty (from **`GET /api/workflows`** / **`GET /api/workflows/{ticket_key}`**), the card header shows a **Show PR** link (new tab) on the right. Each card’s actions include **Go to ticket** (**`jira_browse_url`**, new tab) and **Show description** (reuses the manual-start ticket modal with **Close** only — **Start** is hidden; loads **`GET /api/jira/tickets/{key}/preview`** like the **+** picker). **Description editor** (`TicketDetailModal` edit mode): a **Write / Preview** tab bar appears above the textarea; **Preview** renders the current text via `MarkdownPreview` (including Mermaid). A **"Side by side"** checkbox (top-right of the tab bar) widens the content area into two equal panes — editable textarea on the left, live preview on the right; preview updates are **debounced 400 ms** to avoid re-renders while typing. Disabling side-by-side returns to the tabbed layout on the **Write** tab. `MarkdownPreview` lazy-loads Mermaid via dynamic `import()` and initialises it once (dark theme, `startOnLoad: false`); the mermaid bundle is code-split by Vite and only downloaded when a diagram is first encountered.

**Workflow definition buttons:** Workflow cards display a **"Workflows"** section with one button per discovered `*.toml` definition (shown for all states including **Pending**). Buttons are **topologically sorted** by `depends_on` and show five visual states: **enabled** (blue, clickable), **disabled** (gray with lock icon + tooltip listing unmet deps), **running** (blue with spinner, non-interactive), **completed** (green with check), **error** (red with X, click to retry). Clicking an enabled button on a **Pending** card calls `POST /api/workflows/{id}/run-workflow/{def}` — bootstrap runs automatically; no separate Start button needed. The component (`ui/src/components/WorkflowDefButtons.tsx`) fetches definitions from `GET /api/workflow-definitions` and re-fetches on `workflow_definitions_changed`, `workflow_updated`, and `step_completed` WebSocket events (debounced 500 ms). Definitions are shared across all workflow cards (fetched once in `Dashboard.tsx` and passed through `WorkflowGrid` → `IssueCard` → `WorkflowDefButtons`).

**Port mapping display:** Workflow cards show clickable port-mapping buttons below the action buttons. Port data comes from two sources: **API** (`editor_port_mappings` from the list/detail endpoints — covers static Docker mappings and already-tracked dynamic socat forwards) and **WebSocket** (`port_forwarded`/`port_unforwarded` events — real-time updates). The `PortMappings` component (`ui/src/components/IssueCard.tsx`) merges both, deduplicating by container port (dynamic wins). Each button opens `http://localhost:<host_port>` in a new tab. Port state is managed by `useWorkflows` (`ui/src/hooks/useWorkflows.ts`) which maintains a `DynamicForwards` map keyed by ticket key.

**Run commands:** When `[[run_commands]]` is configured, workflow cards in terminal states (**Done**, **Error**, **Stopped**) display a separator followed by one button per command below the port mappings. Each row is a state machine: **idle** → `Run {name}` button (primary/blue); **running** → `Stop {name}` button (danger/red) plus `Copy` and `Open` buttons (secondary/gray). When the run command's port scanner detects a listening port inside the container, `Copy` copies `http://localhost:<mapped_port>` and `Open` opens it in a new tab. When no port is detected, both buttons are disabled with a tooltip. Each command runs in a dedicated container (`maestro-run-{ticket}-{index}`) with its own socat port forwarding. WebSocket events `run_command_port_forwarded`, `run_command_port_unforwarded`, and `run_command_stopped` keep the UI in sync. Run commands are automatically cleaned up when a workflow is deleted or marked done.

**Terminal output persistence:** The terminal panel **never clears** between steps — when a new step starts, a visual separator (`─── Step Name ───`) is appended to the existing output. Lines accumulate across all steps in a workflow, capped at `TERMINAL_MAX_LINES = 500` (oldest lines are evicted first). For **completed, error, and stopped** workflows, the terminal remains visible on the card in a **collapsed** state (header-only); a chevron toggle expands it to show the last available output (up to 100 lines from the backend `terminal_lines` field). **Running** and **Paused** workflows show the terminal expanded with live streaming as before.

**Workflow duration:** Cards in a terminal state (**Done**, **Error**, **Stopped**) display the **total elapsed wall-clock time** (`updated_at − started_at`) in the step info area, next to the step label. The Report modal also includes a **Total Duration** field in the Ticket Info grid for terminal-state workflows.

**Action button styling:** Buttons use a **four-category colour palette** instead of per-button unique colours: **primary** (blue — Resume, Pause, Retry), **secondary** (gray — Go to ticket, Show description, Report, Editor, Terminal), **success** (emerald — Mark as Done), and **danger** (red — Delete, Close editor). **Pause** and **Resume** buttons use **SVG icons** (⏸ and ▶) instead of text labels, with screen-reader-only text for accessibility. All buttons share the same `.workflow-action-btn` base sizing.

**Show Report button (conditional):** The **Show Report** button on each workflow card is **hidden** until `has_report` becomes `true` (i.e. the consolidation step has written the report file at least once). When `has_report` is `true`, the button is visible and opens the Report modal. The Report modal fetches the report content from **`GET /api/workflows/{id}/report`** and renders it with the `MarkdownPreview` component (full Markdown including Mermaid diagrams), with a Copy button and loading/error states.

**Report modal enhancements:** Each step row now shows a **status label pill** (Success / Failed / Skipped / Running) next to the step name. Step **output lines** show the last 3 lines by default with a "Show N more lines" expand toggle when more are available. The Ticket Info grid includes **Total Duration** for terminal-state workflows.

### Prompts

Templates live in **`[[steps]]`** within TOML workflow definition files (`name`, **`prompt`**, **`repeat`**, **`when`**, **`commands`**). Default steps use **generic natural-language** prompts (no slash-commands); teams with Claude skills can still put **`/address-ticket`** or **`/review-changes`** in a template. Placeholders: **`ticket_key`**, **`ticket_summary`**, **`ticket_description`**, **`description`** (alias for `ticket_description`), **`ticket_type`**, **`acceptance_criteria`**, **`ticket_context`**, **`base_branch`**, and **`pr_url`**. Placeholders are interpolated in both `prompt` (agent steps) and `commands` (command steps). Unknown **`{placeholders}`** are left unchanged.

**`when`** (`StepAvailability`): **`always`** (default when omitted), **`ticketing`** (only when acli/Jira is authenticated), **`no_ticketing`** (only when Jira is **not** available). Steps that do not match the current ticketing availability are **filtered out** before the agent loop runs and are **not counted** in `progress_steps_total`.

## External actions boundary

`crates/maestro-core/src/actions/traits.rs` — `ExternalActions`:

- Jira: assign to current user (`@me`), transition, unassign, get ticket details — all via direct **`acli`** invocations (no engine-level allowlist). Scope is governed by the Atlassian token permissions the user configured.
- GitHub CLI (`gh`): all `gh` invocations run via **`gh_cli::run_gh()`** (no engine-level allowlist). Scope is governed by the GitHub token permissions (fine-grained PAT or GitHub App installation token) the user configured.
- Git: create/remove worktree (**`create_worktree`** clears an existing path under **`worktrees/<branch-slug>`** before **`git worktree add`** on new workflows so orphans are not silently reused; **`remove_worktree`** runs **`git worktree remove --force`** from **`git.repo_path`**, for dashboard **Mark as Done** / **Delete** teardown even when the tree has local changes; on **Unix**, if Git reports **permission denied** (e.g. root-owned files left by older worker containers), Maestro retries once after **`sudo -n /bin/bash -c "chown -R <euid>:<egid> -- <path>"`** when that succeeds — matches the image **`sudoers`** for passwordless **`bash`**); **`delete_local_branch`** (**`git branch -D`**, tolerant if the branch is already gone) used after **Delete** / **Mark as Done** teardown; **`create_pr`** (implemented on **`RealActions`/`DryRunActions`** but **not** called by the workflow engine — agents open PRs with **`gh`** or similar), **commit_changes**, **`configure_git_author_from_github`**, **`request_github_self_as_pr_reviewer`**
- Shell: **run_command**

Helpers: **`actions/gh_github.rs`** (`gh api user`, **`gh pr edit --add-reviewer`**). In **dry mode**, **`request_github_self_as_pr_reviewer`** does not call GitHub (returns skipped); **`configure_git_author_from_github`** still sets local **`git config`** when **`gh`** is available. **`github_app.rs`** provides **`GitHubAppTokenManager`** for JWT generation and installation token caching (see **GitHub App authentication** below).

`real.rs` and `dry_run.rs` implement this for production vs dry runs.

### GitHub App authentication

`crates/maestro-core/src/github_app.rs` — optional bot-identity authentication via a GitHub App. When **`[github]`** config fields are set, `RealActions` and `DryRunActions` receive an `Arc<GitHubAppTokenManager>` and **`configure_git_author_from_github`** uses the bot identity instead of the personal `gh` user.

**Authentication flow:**

1. **JWT generation** — `GitHubAppTokenManager::generate_jwt()` creates an RS256 JWT signed with the App's private key (`iss` = App ID, 10-minute expiry, 60 s backdate).
2. **Installation token** — `get_installation_token()` exchanges the JWT for a 1-hour installation access token via `curl -s -X POST https://api.github.com/app/installations/{id}/access_tokens`. Tokens are cached in a `tokio::sync::RwLock` and refreshed 5 minutes before expiry.
3. **`gh` + git setup** — `configure_git_and_gh_auth()` installs a git credential helper that reads the token from the shared token file (`~/.config/gh/gh-app-token`), falling back to the `$GH_TOKEN` env var. It also sets `git config user.name` / `user.email` to the bot identity. It does **not** call `gh auth login` or `gh auth setup-git`.

**Bot identity:** commits are authored as `maestro-bot[bot] <{app_id}+maestro-bot[bot]@users.noreply.github.com>`. PRs created by the agent are attributed to the App's bot user in the GitHub UI.

**Startup behavior:** `try_create_token_manager()` attempts to parse the PEM key at process start. If it fails (malformed key, missing file), a warning is logged and Maestro falls back to personal `gh` auth — **non-fatal**. Runtime errors (invalid installation ID, insufficient permissions) surface as actionable messages at operation time.

**Centralized token service:** At startup, `main.rs` calls `start_token_file_writer()` which spawns a background `tokio::task` that proactively refreshes the installation token every 5 minutes and writes it atomically (write-then-rename) to **`~/.config/gh/gh-app-token`** on the shared `gh-auth` Docker volume. Worker containers read this file dynamically on every `gh` or `git` invocation instead of relying on a frozen `GH_TOKEN` env var baked into `docker run`. This ensures tokens stay valid even for workflows that run longer than the 1-hour GitHub App token lifetime. The `ContainerRunner` no longer captures or injects a static `GH_TOKEN`; the `wrap_command` shell preamble and `worker-entrypoint.sh` both source the token file. `GH_TOKEN` remains in `PASSTHROUGH_ENV` as a fallback for local development without a GitHub App.

**Required GitHub App permissions:** `contents` (write), `pull_requests` (write), `issues` (read), `metadata` (read).

## Web API and UI

**Authentication:** When **`[web] dashboard_username`** and **`dashboard_password`** are both set, every route below except **`GET /api/health`**, **`GET /api/auth/status`**, **`POST /api/auth/login`**, and **`POST /api/auth/logout`** requires a valid **HttpOnly session cookie** issued by **`POST /api/auth/login`** (JSON **`{ "username", "password" }`**). The browser sends the cookie automatically on same-origin **`fetch`** (use **`credentials: 'same-origin'`**) and on **WebSocket** **`/ws`**. Sign in at **`/login.html`**. **`GET /api/config`** never returns the password; runtime **`PUT /api/config`** may **omit** **`dashboard_password`** or send an **empty string** while **`dashboard_username`** is still non-empty to **preserve** the existing password (the config page omits the key when the password field is left blank). **Cookies are host-scoped:** **`http://localhost:8080`** and **`http://127.0.0.1:8080`** do **not** share the session cookie — use one hostname consistently (otherwise you will appear logged out or get **`401`** after a successful login).

**CORS:** The Axum router applies an **explicit origin allowlist** (not `CorsLayer::permissive()`). Allowed origins come from **`[web] cors_origins`**; when that list is empty (default), the origin is auto-computed from `host`/`port` (e.g. `http://localhost:8080`). Methods are restricted to **GET, POST, PUT, DELETE**; allowed headers to **Content-Type**; **`Access-Control-Allow-Credentials: true`** is set so session cookies work cross-origin. Requests from unlisted origins receive no `Access-Control-Allow-Origin` header. For deployments behind a reverse proxy or TLS terminator, set `cors_origins` to the external-facing origin (e.g. `["https://maestro.example.com"]`). This setting is **startup-only** — not patchable via `PUT /api/config`.

Loaded in `crates/maestro-core/src/config.rs` — sections:

### Config persistence (`ConfigWriter`)

`crates/maestro-core/src/config_writer.rs` — the **single write path** for `config.toml`. All dashboard and wizard writes funnel through this module; request handlers never touch the filesystem directly.

**Write protocol:** validate → acquire advisory file lock (`fs2::FileExt::lock_exclusive` on `config.toml.lock`) → serialize to TOML → write to `config.toml.tmp` → `fs::rename` over `config.toml` (atomic on POSIX) → record write timestamp → release lock. Invalid configs are rejected **before** any I/O.

**Self-authored write detection:** `ConfigWriter` records the epoch-millis timestamp of each successful write in a shared `AtomicU64`. The `ConfigWatcher` checks this timestamp and skips reload events that fall within a 2 s window after an API write — this prevents a write→detect→reload loop.

### Config file watcher (`ConfigWatcher`)

`crates/maestro-core/src/config_watcher.rs` — a background task that polls `config.toml` for mtime changes every 5 s (configurable). When a change is detected:

1. Skip if the change is self-authored (within 2 s of an API write).
2. Read, parse, and validate the new config.
3. If valid: hot-swap the `Arc<RwLock<Config>>`. Log warnings for restart-required fields (`web.host`, `web.port`, `web.cors_origins`).
4. If invalid: log a warning with the validation error; keep the current in-memory config.

Uses a polling approach (`mtime` comparison) rather than `inotify` because inotify does **not** fire for host-side edits on Docker Desktop for macOS (VirtioFS/gRPC-FUSE bind mounts). Polling works universally.

The watcher is spawned in the main `tokio::select!` loop alongside the snapshot syncer, ticketing pollers, PR merge poller, and web server.

## Jira poller

`crates/maestro-core/src/jira/poller.rs`: on an interval, if `project_keys` non-empty, lists **To Do** tickets (via `JiraClient` / `acli`), skips keys that already exist in the workflow map, and only starts new workflows when **`dashboard_workflow_count`** is less than **`effective_max_active_workflows()`** (**`[general] max_active_workflows`**, or **`max_concurrent_workflows`** when **`max_active_workflows`** is **`0`**). **Every row still on the dashboard counts** — **Done**, **Paused**, **Stopped**, **Error**, and in-progress — until **Mark as Done** or **Delete** removes it from the map. **Restored** workflows are in the map before the first poll, so they consume slots. Uses `cancel_token` for shutdown. When **`AppState.polling_paused`** is set (dashboard **Pause polling**, **`POST /api/polling/pause`**, or **`[general] auto_polling = false`** at process start), the poller still waits on the interval but **skips** `poll_once` (including the initial poll on startup if already paused). In-memory pause/resume toggles are **not** written to **`config.toml`**; a restart applies **`auto_polling`** again if set. Workflow snapshot resume is unchanged (see **How the binary starts**).

## PR merge status poller

`crates/maestro-core/src/github/pr_merge_poller.rs`: a lightweight background poller that checks whether PRs associated with workflows have been merged on GitHub. Runs on a configurable interval (**`[general] pr_merge_poll_interval_secs`**, default **60**, **`0`** disables). On each tick, iterates over all workflows in the engine map, filters to those with a non-empty **`pr_url`** and **`pr_merged == false`**, parses the GitHub PR URL (`parse_pr_url` in `github/mod.rs` extracts `owner/repo` and PR number from `https://github.com/{owner}/{repo}/pull/{number}`), and calls **`gh api repos/{owner_repo}/pulls/{number} --jq .merged`**. When the API returns `true`, sets **`workflow.pr_merged = true`** and broadcasts a **`workflow_updated`** `WorkflowEvent` (with **`pr_merged: Some(true)`**) so the dashboard updates in real time. Errors are logged at **`warn`** level and do not abort the polling loop. The poller respects the shared **`CancellationToken`** for graceful shutdown. It is wired into the main `tokio::select!` loop in `crates/maestro-cli/src/main.rs` alongside the ticketing pollers, snapshot syncer, and web server.

The dashboard renders a **"Merged"** badge (purple pill with a checkmark icon, styled via `.workflow-merged-badge` in `styles.css`) next to the **"Show PR"** link when **`pr_merged`** is true. The badge appears within one polling cycle of the PR being merged, pushed via WebSocket without a page refresh. **`WorkflowSummary`** includes **`pr_merged: bool`** so the badge state is available from both **`GET /api/workflows`** and **`GET /api/workflows/{ticket_key}`**. The **`pr_merged`** field is persisted in **`PersistedWorkflowRecord`** (`#[serde(default)]` for backward compatibility) so merge status survives container restarts.

## Docker entrypoint and CLI helpers

- **`docker/entrypoint.sh`**: root section re-execs as `maestro` via **`setpriv --reuid --regid --init-groups`** (preserves the controlling TTY; `runuser` would call `setsid()` and detach it). `setup` mode (5 steps — required: `gh` + agent provider auth; Step 1 (GitHub CLI / `gh auth login`) is **skipped** when **`[github] app_id`** is non-zero in config (GitHub App provides tokens automatically at runtime — no interactive auth needed); Step 2 (Atlassian CLI / `acli`) is **skipped** when `ticketing_system` is not `jira`; optional: repo clone; Step 5: report generation toggle). Agent auth reads **`[agent] provider`** from config: `claude` → launches **`claude`** (full setup wizard; user exits with `/exit` when done; skipped when **`CLAUDE_CODE_OAUTH_TOKEN`** is set); `cursor` → `agent login` (skipped when **`CURSOR_API_KEY`** is set). `NODE_OPTIONS=--dns-result-order=ipv4first` is exported in setup mode so Claude's OAuth callback server binds to `127.0.0.1` (IPv4) not `[::1]` (IPv6). **`make auth`** runs with **`--network=host`** so the host browser's OAuth redirect reaches the callback server inside the container. Normal mode: **validates config.toml mount** (hard error with actionable message if missing — the image ships without baked-in config), **warns if maestro.env is not mounted** (optional), then **`maestro preflight`** (on failure: captures error into **`MAESTRO_PREFLIGHT_ERROR`**, logs warning, continues in degraded mode — **does not exit**), **`maestro docker-hooks startup`** (`[docker] compose_up_commands`, on failure: appends error to **`MAESTRO_PREFLIGHT_ERROR`**, logs warning, continues), **workspace check** (`/workspace/.git`, on failure: appends error to **`MAESTRO_PREFLIGHT_ERROR`**, logs warning, continues), then **`exec maestro`**. In all failure cases the web server starts and the dashboard shows a red banner with the accumulated error messages so the operator knows what to fix. **`config.toml` missing is the only hard exit** — everything else is degraded mode.
- **Distributed image**: the Docker image ships **without** `config.toml`, `maestro.env`, or user workflow files baked in. Users must volume-mount all three at runtime. Example reference files are available inside the image at **`/etc/maestro/examples/`** (`config.toml.example`, `maestro.env.example`, `workflows/*.example.toml`). To extract them: `docker run --rm --entrypoint cat maestro /etc/maestro/examples/config.toml.example > config.toml`. The `.dockerignore` excludes `config.toml`, `maestro.env`, and user workflow files from the build context to prevent secrets from leaking into image layers.
- **Compose mounts**: **`config.toml`** is mounted at **`/etc/maestro/config.toml`** as **read-write** (`:rw`) so the `ConfigWriter` can persist dashboard changes; `maestro.env` and `workflows/` remain **read-only** (`:ro`). **`docker-compose.yml`** bind-mounts **`./workflows` → `/etc/maestro/workflows`** so dynamic TOML workflow definitions are available to the engine. Without that mount, the entrypoint fails with a clear error message telling the user how to mount the config. The bind mount must include your **local** workflow definition files (e.g. **`implement_ticket.toml`**, **`address_pr_comments.toml`**, **`merge_base.toml`**) — copy from **`*.example.toml`** in the repo; tracked examples alone are not used at runtime.
- **Container engine access**: the default Compose stack does **not** provide a Docker/Podman daemon. Image installs Debian’s **`docker.io`** package for the **`docker`** CLI (no in-container daemon). To give the CLI a daemon, merge **`docker-compose.dind.yml`** — a **Docker-in-Docker sidecar** that adds a **`docker:27-dind`** container running a real Docker daemon. Maestro connects via **`DOCKER_HOST=tcp://dind:2375`** over the internal compose network. The sidecar runs **`--privileged`** (required by DinD) but is isolated to its own container; Maestro itself gains no extra privileges. The **`workspace`** named volume is shared so paths like **`/workspace/my-project/…`** resolve identically from both containers. A **`dind-storage`** volume persists Docker image layers across restarts. The entrypoint waits for the DinD daemon to become ready (retries `docker info` against **`DOCKER_HOST`**) before starting Maestro. Works on all platforms (macOS Podman, macOS Docker Desktop, Linux). **`podman-compose`** (Python) does not read `COMPOSE_FILE` from `.env` — use explicit `-f` flags: `podman compose -f docker-compose.yml -f docker-compose.dind.yml up -d`.
- **GH-45 breaking changes**: (1) The DinD port range mapping (`ports: "9200-9300:9100-9200"`) was removed from **`docker-compose.dind.yml`** — editor and terminal sessions are now routed through the shared-port proxy at `/s/<path-token>/...` on the dashboard listener port (no direct host-port access). Existing deployments that relied on direct `localhost:9100–9200` access to editor/terminal sessions must use the proxy URLs returned by the API instead. (2) **`container_name`** was removed from both the `maestro` and `dind` services in Compose so that multiple worktrees/Compose stacks can coexist without name collisions. Scripts or monitoring referencing `maestro` or `maestro-dind` by container name must be updated to use `docker compose exec maestro` / `docker compose exec dind` or the dynamic container names assigned by Compose.
- **Workflow isolation**: when DinD is available, each workflow’s **install**, **pre_install**, **mise install**, and **agent steps** (Claude/Cursor sessions) run in **ephemeral Docker containers** (`docker run --rm`) instead of directly on the Maestro host. Each worker container gets its own network namespace (no port conflicts between concurrent workflows) and a `--cap-add=NET_ADMIN` for egress rules. The worker image is set via **`[general] worker_image`**, auto-detected from the running Maestro container (`docker inspect $HOSTNAME` — the container ID), or from the **`MAESTRO_REGISTRY_IMAGE`** env var (baked into the Docker image at build time as `ghcr.io/morphet81/maestro:<version>`); final fallback is **`maestro:latest`**. The entrypoint **auto-pulls** the worker image into DinD when it is not found locally (using `MAESTRO_REGISTRY_IMAGE`), then tags it as `maestro:latest`. For local development, **`make load-worker`** (`podman save | docker load`) loads the locally-built image instead. Auth volumes are mounted into DinD at **`/shared-auth/*`** and bind-mounted into workers at the standard home paths. A **`playwright-cache`** volume maps to **`/shared-auth/playwright-cache`** (DinD) and **`~/.cache/ms-playwright`** in workers so Playwright uses the **project’s** browser revision from **`npm`/`playwright install`**, not a mismatched Chromium baked into the image (which caused visual snapshot drift). Workers default **`TZ=UTC`**, **`LANG`/`LC_ALL=C.UTF-8`**; the host may pass through **`CI`**, **`TZ`**, **`LANG`**, **`LC_ALL`**, **`PLAYWRIGHT_BROWSERS_PATH`** when set. The worker entrypoint (**`worker-entrypoint.sh`**) applies egress rules, sources **`/etc/maestro/env`**, then `runuser`s as **`maestro`**. Direct argv commands (**`claude`**, **`mise`**, Cursor **`agent`**) bypass that entrypoint (`--entrypoint ""`) but use **`--user maestro:maestro`** so bind-mounted worktree files stay owned by the same uid/gid as the Maestro process (avoids **`git worktree remove`** **permission denied** after isolated runs). Shell **`pre_install`** / **`install`** still use **`worker-entrypoint.sh`** (root applies egress, then **`runuser`**). Worker containers are force-removed on **cancel**, **stop**, **mark-done**, **delete**, and **stop_all_workflows** via **`ContainerRunner::cleanup_for_ticket`**. When DinD is unavailable (no `DOCKER_HOST` or `docker info` fails), the engine logs an error and **any workflow that reaches the agent/install phase will immediately fail** — DinD is required; there is no local-execution fallback.
- **Per-issue filesystem isolation**: worker containers, editor containers, and run-command containers receive **targeted bind mounts** instead of the broad `/workspace:/workspace` volume. Each container sees only three paths: (1) its own worktree directory (`/workspace/worktrees/{slug}` — read-write), (2) the shared `.git` directory (`/workspace/.git` — needed for git operations like commit, push, status), and (3) the shared `.maestro` directory (`/workspace/.maestro` — read-only; contains `.npmrc` for npm registry auth). This prevents a compromised agent in one issue’s container from reading or modifying files belonging to another issue’s worktree. The isolation is implemented via `build_volume_args()` in `container.rs` which conditionally replaces the `/workspace:/workspace` mount. `ContainerRunner.with_isolate_workspace()` enables isolation for worker containers; `start_editor()` and `start_run_command()` accept an `isolate_workspace: bool` parameter. Auth volumes (`/shared-auth/*`) and config (`/etc/maestro:ro`) remain shared across all containers. The `bootstrap_new_workflow` and `run_workflow_def_steps` functions in `driver.rs` enable isolation by default for all new workflows.
- **Figma CLI (`fcli`)**: installed at image build time to **`/usr/local/bin/fcli`** (amd64 uses the GitHub prebuilt tarball; arm64 has no prebuilt, so `cargo install --git` from source via a temporary rustup toolchain that is removed after build). Available to every container using the maestro image (main, workers, editor, terminal). Auth: set **`FIGMA_ACCESS_TOKEN`** in **`maestro.env`** (it's in **`PASSTHROUGH_ENV`**), or run **`fcli auth login`** once from the dashboard terminal — credentials are stored in **`~/.config/fcli/auth.json`** which is mounted from the shared **`fcli-auth`** volume, so they persist across restarts and are visible to all containers.
- **Lokalise CLI v2 (`lokalise2`)**: installed at image build time to **`/usr/local/bin/lokalise2`** from the GitHub prebuilt tarball for the current architecture (both amd64 and arm64 have Linux builds). Available to every container using the maestro image. Auth: pass **`--token`** on the CLI, or put the token in **`config.yml`** (`--config` flag), or export **`LOKALISE_API_TOKEN`** in **`maestro.env`** and wrap invocations with `--token "$LOKALISE_API_TOKEN"` (the var is in **`PASSTHROUGH_ENV`**).
- **`maestro github-app-token`**: generates a GitHub App installation token from the `[github]` config and prints it to stdout. Used by the setup script's repository clone step (Step 4) when a GitHub App is configured and `gh` is not authenticated interactively. Exits non-zero on error.
- **`maestro preflight`**: validates GitHub and provider-specific auth (hard errors); GitHub auth check is **skipped entirely** when **`[github] app_id`** is non-zero (GitHub App is configured — tokens are generated at runtime, no `gh auth` needed); Atlassian CLI (acli) auth is a **soft-fail** — returns `PreflightResult { acli_ok }` instead of `Err`. When `acli_ok` is `false` the app still starts in **no-Jira mode** (see top of this file). Cursor: skips **`agent status`** when **`CURSOR_API_KEY`** is set or when **`cli-config.json`** under **`CURSOR_CONFIG_DIR`** looks authenticated; otherwise **`agent status`** with timeout and process-group kill. Compose sets **`CURSOR_CONFIG_DIR=/home/maestro/.cursor`** to align with the **`cursor-auth`** volume.
- **`maestro docker-hooks build|startup`**: runs `build_commands` or `compose_up_commands` from config as **`bash -c`** in `git.repo_path` (used by Dockerfile `RUN` and entrypoint; **`sh`** on Debian is often dash and lacks `pipefail`). Hook children get **`HOME`**, **`MAESTRO_HOME`** (compose: `/home/maestro`), and **`CURSOR_CONFIG_DIR`** so writes to **`~/.claude`** / **`~/.cursor`** hit named volumes, not **`/.claude`** when **`HOME`** is unset.
